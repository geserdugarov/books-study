# Layer 7 · Topic 20 — Language Evolution & Future

> How each language evolves — Rust's edition system with six-week release trains and the RFC process, Java's six-month cadence with preview features and major OpenJDK projects (Amber, Panama, Loom, Valhalla, Leyden), and Python's annual releases governed by PEPs with the ongoing transformation from dynamic scripting language to gradually-typed systems language (typing evolution, free-threaded mode, JIT compiler) — and the governance structures, deprecation strategies, and migration tooling that enable languages to evolve without breaking existing code. Capstone topic synthesizing all prior layers to understand how language design decisions compound over time.

### Coverage Gaps

The owned books **do not** cover:

- **Rust 2024 edition** — Klabnik & Nichols' Appendix E covers editions through 2021, but the 2024 edition (stabilized in Rust 1.85, February 2025) introduces significant changes including `gen` blocks, `unsafe_op_in_unsafe_fn` lint, lifetime capture rules changes (`use<>` syntax), and the RPIT (Return Position Impl Trait) lifetime capture defaults; the Rust Edition Guide and Rust blog announcements are needed
- **Rust RFC process in depth** — Gjengset Ch.13 discusses staying up to date with Rust's evolution, but no owned book explains the RFC (Request for Comments) process in detail: how RFCs are proposed, discussed, accepted, implemented behind feature gates, stabilized through nightly→beta→stable, and how the Rust governance structure (teams, working groups, the Leadership Council replacing the Core Team) operates; the Rust RFC repository and Rust governance documentation are needed
- **Java JEP process and LTS release strategy in detail** — Evans Ch.1 introduces JEPs and the six-month cycle, but no owned book provides a detailed explanation of the JEP lifecycle (draft → submitted → candidate → targeted → integrated → completed/withdrawn), the LTS release strategy (Java 8, 11, 17, 21 as LTS, with different vendor support timelines from Oracle, Amazon Corretto, Eclipse Temurin), or how organizations decide which Java version to adopt; the OpenJDK JEP process documentation and vendor support matrices are needed
- **Python PEP process in depth** — Martelli Ch.1 introduces PEPs and governance, but no owned book explains the full PEP lifecycle (draft → accepted → final → superseded/rejected/withdrawn), the different PEP types (Standards Track, Informational, Process), the role of the Steering Council (post-BDFL governance after Guido van Rossum's retirement in 2018), or how the annual release schedule (PEP 602) works; the PEP 0 index and Python developer's guide are needed
- **Python typing evolution trajectory (PEP 484 through PEP 695)** — Martelli Ch.5 and Ramalho Ch.8/Ch.15 cover typing as of Python 3.10–3.11, but no owned book covers the complete typing evolution: PEP 484 (2014, type hints), PEP 526 (variable annotations), PEP 544 (Protocol), PEP 612 (ParamSpec), PEP 646 (variadic generics), PEP 673 (Self type), PEP 681 (dataclass_transform), PEP 695 (Python 3.12, `type` statement and simplified generic syntax `def foo[T](x: T) -> T`), and the ongoing evolution toward a more ergonomic and powerful type system; the PEP documents and mypy/pyright changelogs are needed
- **Python free-threaded mode (PEP 703)** — Shaw Ch.8 covers the GIL in depth, but the book predates PEP 703 (accepted 2023) which introduces experimental free-threaded CPython (no-GIL build), available as an experimental option in Python 3.13+; no owned book covers the implementation approach (biased reference counting, deferred reference counting, per-object locking), the migration path for C extensions, or the performance implications; PEP 703, the CPython free-threading documentation, and benchmark analyses are needed
- **Python JIT compiler (PEP 744)** — Shaw's CPython Internals predates the experimental copy-and-patch JIT compiler introduced in CPython 3.13; no owned book covers the JIT architecture, how it differs from PyPy's tracing JIT, or the Faster CPython project's broader performance roadmap; PEP 744 and the Faster CPython project documentation are needed
- **Cross-language governance comparison** — no single source compares the three languages' governance models side-by-side: Rust's Leadership Council with domain-specific teams vs Java's OpenJDK stewardship under Oracle with community input vs Python's Steering Council with BDFL-delegate system; understanding these differences illuminates why each language evolves at different rates and in different directions
- **Deprecation tooling and migration automation** — Klabnik Appendix D mentions rustfix, and Slatkin Item 123 covers Python's warnings module, but no owned book provides a comprehensive comparison of deprecation and migration tooling: Rust's `#[deprecated]` attribute + cargo fix + edition migration, Java's `@Deprecated(since, forRemoval)` + jdeprscan + JDK migration guides, and Python's `warnings.warn(DeprecationWarning)` + pyupgrade + 2to3 (historical); automated migration tools and their effectiveness across languages are not covered
- **Semantic versioning across ecosystems** — Gjengset Ch.5 covers Rust's semver approach, but no owned book compares how Rust (strict semver enforced by cargo, crater for ecosystem-wide testing), Java (no enforced versioning standard, varying conventions across Maven/Gradle artifacts), and Python (PEP 440 versioning, varying adherence across PyPI packages) approach API compatibility and breaking changes
- **Language specification formality** — no owned book compares the formality of each language's specification: Rust's reference (semi-formal, evolving toward a formal specification via the Ferrocene project), Java's JLS (Java Language Specification, highly formal, legally significant), and Python's reference (informal, CPython-is-the-spec philosophy); this difference affects how each language handles edge cases and portability
- **Feature preview/experimental mechanisms compared** — Evans Ch.1 mentions Java's preview features, Matthews Ch.3 discusses Rust's nightly, but no owned book provides a systematic comparison: Rust's feature gates (`#![feature(...)]` on nightly only) → stabilization, Java's preview features (`--enable-preview`) → incubating → standard, and Python's `__future__` imports → feature flags → default behavior; understanding these mechanisms is essential for tracking each language's evolution pipeline

### Session 1 — Governance & Release Models (Cross-Language Comparison)

**Goal:** understand how each language is governed, who makes decisions, how changes are proposed and accepted, and how the release cadence shapes the evolution velocity — establishing the meta-framework for understanding why each language evolves the way it does.

| Step | Time | Activity |
|------|------|----------|
| 1.1 | 30 min | **Java: the six-month cadence and JEP process.** Read Evans et al Ch.1 pp. 3–25 (the new Java release model, JSRs, JEPs, incubating features). Read Evans & Gough Ch.3 p. 73 (the Java Release Cycle). Key insight: Java's evolution is governed by a layered process. The Java Community Process (JCP) produces Java Specification Requests (JSRs) for major platform changes, while the JDK Enhancement Proposal (JEP) process handles individual features within the OpenJDK. Since Java 10 (2018), Java follows a strict six-month release cadence: a new version ships every March and September, regardless of which features are ready. This time-based model replaced the old feature-based model where releases (Java 7, Java 8) would slip for years waiting for flagship features. The result: features land when they're ready, not all at once. Preview features (`--enable-preview`) allow developers to try features before they're finalized — pattern matching for instanceof (preview in Java 14, finalized in Java 16) took three versions to stabilize. Long-Term Support (LTS) releases (Java 8, 11, 17, 21, 25) receive extended security patches and are the versions most organizations deploy to production. Oracle stewards the OpenJDK specification, but the ecosystem includes multiple vendors (Amazon Corretto, Eclipse Temurin, Azul Zulu, Red Hat) providing their own builds with different support timelines. This vendor ecosystem means that Java's evolution is both centralized (Oracle drives the specification) and decentralized (multiple vendors compete on support and features). |
| 1.2 | 25 min | **Rust: six-week trains and the RFC process.** Read Klabnik & Nichols Appendix E pp. 515–516 (Rust editions and their role in language evolution). Read Matthews Ch.3 pp. 43–62 (stable vs nightly toolchains, Clippy, cargo-expand, cargo-fuzz). Read the Rust Governance documentation and RFC repository online. Key insight: Rust has the fastest release cadence of the three languages: a new stable version every six weeks, following a "release train" model (nightly → beta → stable). Experimental features are gated behind `#![feature(...)]` attributes available only on nightly, allowing rapid experimentation without destabilizing the stable channel. The RFC process is the primary mechanism for language evolution: any community member can propose an RFC (Request for Comments) via a pull request to the rfcs repository. RFCs are discussed publicly, revised, and accepted or rejected by the relevant team (Language, Compiler, Library, etc.). Once accepted, an RFC is implemented behind a feature gate on nightly, tested extensively, and eventually stabilized into stable Rust — a process that can take months to years. Rust's governance underwent a major transition in 2023: the Core Team was replaced by a Leadership Council (RFC 3392) with representatives from each team, distributing authority across domain-specific teams (Language, Compiler, Library, Cargo, Infrastructure, etc.). This team-based governance means that different areas of the language evolve semi-independently. The edition system (covered in depth in Session 2) is the mechanism that enables backwards-incompatible evolution: editions are opt-in migration points that can change language semantics while maintaining interoperability between crates using different editions. |
| 1.3 | 25 min | **Python: annual releases and PEP governance.** Read Martelli et al Ch.1 pp. 1–19 (Python's history, development process, versioning model, PEP governance). Read Slatkin Item 1 p. 1 (Know Which Version of Python You're Using) and Item 2 pp. 3–5 (PEP 8). Read PEP 13 (Python Language Governance) and PEP 602 (Annual Release Cycle) online. Key insight: Python's governance underwent a dramatic transformation in 2018 when Guido van Rossum retired as BDFL (Benevolent Dictator For Life) after the contentious PEP 572 (walrus operator) debate. PEP 13 established the Steering Council: five elected members who make final decisions on PEPs, with elections held at each major release. BDFL-delegates can be appointed for specific PEPs (domain experts who make the accept/reject decision for their area). The PEP process is Python's equivalent of Rust's RFCs and Java's JEPs: Standards Track PEPs propose language or library changes, Informational PEPs describe design issues, and Process PEPs describe procedural changes. Since PEP 602 (adopted in 2019), Python follows an annual release cycle: a new minor version (3.x) ships every October, with a 17-month development cycle (alpha → beta → release candidate → final). Each version receives security fixes for 5 years. The Python Software Foundation (PSF) oversees the ecosystem but does not control CPython development directly — CPython's core developers operate semi-autonomously. Compared to Rust and Java: Python's release cadence is the slowest (annual vs six-weekly/six-monthly), its governance is the most recently reformed, and its evolution has historically been the most conservative (the Python 2→3 transition traumatized the community and led to extreme caution about breaking changes). |
| 1.4 | 20 min | **Cross-language comparison: governance spectrum.** Build a comparison table across five dimensions: (1) **Decision authority** — Rust: distributed across teams with Leadership Council coordination; Java: Oracle-steered with community input via JCP/OpenJDK; Python: Steering Council with BDFL-delegate system. (2) **Proposal mechanism** — Rust: RFCs (public pull requests, anyone can propose); Java: JEPs (typically from OpenJDK developers, more formal); Python: PEPs (anyone can propose, but need a sponsor/champion). (3) **Release cadence** — Rust: 6 weeks (stable), editions every 3 years; Java: 6 months, LTS every 2 years; Python: annual, support for 5 years. (4) **Experimentation mechanism** — Rust: nightly feature gates; Java: preview features (--enable-preview); Python: `__future__` imports + experimental build flags. (5) **Breaking change strategy** — Rust: editions (opt-in, interoperable); Java: deprecation → removal over multiple LTS cycles; Python: DeprecationWarning for 2+ versions before removal. Key insight: the governance model directly shapes each language's evolution velocity and character. Rust evolves fastest at the syntax/library level (six-week releases) but gates major changes behind editions. Java evolves fastest at the JVM/platform level (six-month releases bring platform improvements that benefit all Java versions). Python evolves most cautiously but is currently undergoing its most ambitious transformation period (free-threaded mode + JIT compiler simultaneously). The three languages represent three models of open-source governance: Rust's decentralized meritocracy, Java's corporate stewardship with open process, and Python's elected representative democracy. |

### Session 2 — Rust Editions Deep Dive

**Goal:** understand Rust's edition system in depth — how editions enable backwards-incompatible language changes without breaking the ecosystem, the concrete changes introduced in each edition (2015, 2018, 2021, 2024), and the migration tooling (cargo fix, rustfix) that automates the transition.

| Step | Time | Activity |
|------|------|----------|
| 2.1 | 30 min | **The edition mechanism and philosophy.** Read Klabnik & Nichols Appendix E pp. 515–516 (editions and their role). Read the Rust Edition Guide online (Introduction, What are Editions, Creating a New Edition). Key insight: Rust's edition system is the language's most innovative governance mechanism and a direct response to the "Python 2→3 problem." An edition is a coherent package of language changes that is opt-in per crate (specified in `Cargo.toml` as `edition = "2021"`). The critical design constraint: crates using different editions can interoperate seamlessly — a crate compiled with edition 2015 can depend on a crate using edition 2024 and vice versa. This is possible because editions only change surface syntax and semantics that the compiler can translate to a common intermediate representation. Editions cannot change the standard library API, the ABI, or the trait system in incompatible ways. The result: the Rust ecosystem never fragments along edition boundaries. Migration is automated: `cargo fix --edition` rewrites code to be compatible with the new edition, handling most changes automatically. Editions are released approximately every three years (2015, 2018, 2021, 2024), giving the community time to adopt each one. Compare with Java: Java has no equivalent mechanism — breaking changes require deprecation cycles spanning multiple LTS versions, and some deprecated features (e.g., finalization) persist for decades. Compare with Python: Python 2→3 was a one-time breaking change with no interoperability between versions, requiring the entire ecosystem to migrate simultaneously — the exact failure mode that Rust's editions prevent. |
| 2.2 | 25 min | **Edition 2018 and 2021 changes.** Read the Rust Edition Guide online (2018 Edition, 2021 Edition). Read Gjengset Ch.5 pp. 67–84 (feature flags, conditional compilation, MSRV policy — the infrastructure that supports edition evolution). Key insight: Edition 2018 introduced: the module path system redesign (uniform paths, replacing the inconsistent 2015 `extern crate` and `use` path rules), async/await syntax (the most impactful addition, enabling the Tokio-based async ecosystem), NLL (Non-Lexical Lifetimes, making the borrow checker smarter about when borrows actually end), and the `dyn Trait` syntax requirement (previously bare `Trait` was allowed for trait objects). Edition 2021 introduced: disjoint capture in closures (closures capture individual fields rather than entire structs, reducing borrow checker friction), `IntoIterator` for arrays (arrays now implement `IntoIterator` directly), and `panic!("{}")` formatting by default (panic messages are now formatted strings). ### Session 3 — Java's Six-Month Cadence & Preview Features

**Goal:** understand Java's modern evolution model — the six-month release cadence, preview feature mechanism, LTS release strategy, and how the major OpenJDK projects (Amber, Panama, Loom, Valhalla, Leyden) are transforming the language and platform incrementally.

| Step | Time | Activity |

| 3.1 | 30 min | **Java's evolution arc: from stagnation to velocity.** Read Horstmann Ch.1 pp. 24–31 (A Short History of Java: Green Project through Java 21). Java 8 was the most transformative single release, adding functional programming idioms that changed how Java is written. A preview feature is fully specified and implemented but not yet permanent — it requires the `--enable-preview` flag to use, and its design may change in the next release. | 3.3 | 25 min | **The LTS strategy and version selection.** Read Evans et al Ch.18 section 18.6 (Selecting your Java version). LTS releases receive security updates and bug fixes for years (Oracle: minimum 5 years premier support; Eclipse Temurin: minimum 4 years). Current LTS versions: Java 8 (2014, still widely used), Java 11 (2018), Java 17 (2021), Java 21 (2023), and Java 25 (2025). Non-LTS versions receive updates only until the next release (6 months). The practical implication: most organizations run on LTS versions and skip 5–11 non-LTS releases between upgrades. Compare with Python: all minor versions receive security fixes for 5 years, similar to Java's LTS model but applied to every release. |
| 3.4 | 20 min | **Java Platform Module System as a case study.** Read Horstmann Ch.12 pp. 150–167 (JPMS: module declarations, migration strategies, automatic modules, unnamed module, command-line flags for migration). Seven years after JPMS's introduction, adoption remains mixed: most libraries have added `module-info.java`, but many applications still rely on the classpath (unnamed module). The lesson: even with Java's resources and governance, a backwards-incompatible platform change takes a decade to fully absorb. JPMS influenced Java's subsequent approach: Loom's virtual threads are designed as drop-in replacements for platform threads, Panama's FFI replaces JNI without breaking existing JNI code, and Valhalla's value classes are designed to be backwards-compatible with existing generics. Each subsequent project learned from JPMS's migration difficulty. |

**Goal:** examine the major features that are shaping Rust's near-term future — understanding what each feature enables, why it was difficult to implement, and how the RFC/stabilization pipeline brought each from proposal to reality.

| 5.1 | 30 min | **GATs and async traits: completing the async story.** Read Gjengset Ch.13 pp. 223–243 (the Rust ecosystem: staying up to date, what's out there). Key insight: Generic Associated Types (GATs, stabilized in Rust 1.65/2022) and async functions in traits (stabilized in Rust 1.75/2023) are the two features that complete Rust's async programming model. GATs allow associated types to be generic over lifetimes or types: `type Item<'a>: Iterator<Item = &'a Self::Data>` — enabling the `LendingIterator` pattern where an iterator can yield references tied to its own lifetime. The current limitation: async trait methods are not yet fully compatible with `dyn Trait` (dynamic dispatch) — `trait_variant::make` provides a workaround by generating a separate trait for dynamic dispatch. | 5.2 | 25 min | **Polonius: the next-generation borrow checker.** Read the Polonius repository documentation online. Key insight: Polonius is the planned replacement for Rust's current borrow checker (NLL — Non-Lexical Lifetimes, stabilized in edition 2018). The current borrow checker sometimes rejects valid programs because it tracks borrows at the variable level rather than the access-path level. Polonius uses a different algorithm (based on Datalog queries over program facts) that tracks borrows more precisely, accepting more valid programs while maintaining all safety guarantees. The classic example: with NLL, a function that borrows from a HashMap and then inserts into it is rejected even when the borrow has ended, because the borrow checker tracks the borrow at the variable level. Polonius can determine that the borrow has ended and allow the subsequent mutation. Polonius has been in development since 2018 and is available on nightly behind a feature flag. Its stabilization is a multi-year effort because the borrow checker is the most correctness-critical component of the Rust compiler — any regression could introduce unsoundness. This demonstrates a key difference from Java and Python: Rust's evolution is constrained by the need to maintain formal safety guarantees, making changes to core language semantics orders of magnitude more difficult to validate. |

Instead of writing a struct with `next()` method implementation, a gen block uses `yield` to produce values — the compiler transforms the block into a state machine that implements `Iterator`. Gen blocks were part of the 2024 edition's keyword reservation (`gen` became a reserved keyword) and are progressing through nightly stabilization. The broader context: Rust's coroutine design has been evolving since 2016, with generators as a building block for both iterators (sync) and async/await (already stable, implemented via a similar state machine transformation). The current evolution is making the generator-as-iterator pattern available directly in the language syntax, rather than only as the hidden implementation mechanism behind async. Future directions include async generators (`async gen { yield value; }`) for asynchronous streaming iterators, and possibly algebraic effects. Compare with Python: Python's generator syntax (`yield`) has been stable since Python 2.2 (2001) — Rust is adding a similar capability 25 years later, but with full type safety and zero-cost abstraction (no heap allocation, the state machine is sized at compile time). The Rust Foundation (established 2021) provides organizational infrastructure and funding but does not control language direction — technical decisions remain with the volunteer teams. ### Session 7 — Major Features: Python (free-threaded mode, JIT, pattern matching)

**Goal:** examine the three transformative features reshaping Python's identity — the free-threaded (no-GIL) mode, the experimental JIT compiler, and structural pattern matching — understanding how each addresses a fundamental limitation of CPython and what they signal about Python's future direction.

| Step | Time | Activity |
|------|------|----------|
| 7.1 | 35 min | **Free-threaded Python (PEP 703): removing the GIL.** Read Shaw Ch.8 pp. 221–283 (the GIL, multiprocessing, threading limitations, subinterpreters). Read PEP 703 online (Making the Global Interpreter Lock Optional). Read the Python free-threading documentation online. Key insight: the GIL (Global Interpreter Lock) has been CPython's most debated design decision since its inception. Shaw's coverage explains why it exists: CPython's reference counting memory management is not thread-safe, so the GIL ensures that only one thread executes Python bytecode at a time. This simplifies CPython's internals and C extension development but prevents true multi-threaded parallelism for CPU-bound Python code. PEP 703 (accepted 2023, experimental in Python 3.13+) makes the GIL optional through a specialized CPython build (`python3.13t`). The implementation is extraordinarily complex: (1) biased reference counting (objects are owned by a thread and use thread-local refcounts, falling back to atomic operations only when shared), (2) deferred reference counting (immortal objects and frequently-accessed globals avoid refcount operations entirely), (3) per-object locking (fine-grained locks replace the single GIL), (4) safe memory reclamation for lock-free data structures. The migration challenge is immense: every C extension that assumes the GIL protects its internal state must be audited and potentially rewritten. The Python community is taking a three-phase approach: Phase 1 (3.13): opt-in experimental build; Phase 2 (3.14–3.15): increasing stability, ecosystem testing; Phase 3 (future): potentially making free-threaded the default. The performance impact on single-threaded code is a key concern: the free-threaded build may be 5–10% slower for single-threaded workloads due to the overhead of per-object locking and biased reference counting. Compare with Rust: Rust's ownership model makes data races a compile-time error — no runtime locking overhead needed. Compare with Java: the JVM never had this constraint — Java has supported true multi-threading since version 1.0. |
| 7.2 | 25 min | **The CPython JIT compiler (PEP 744).** Read Shaw Ch.5 pp. 151–175 (the evaluation loop: bytecode dispatch — the performance bottleneck that the JIT addresses). Read PEP 744 online. Read the Faster CPython project documentation online. Key insight: CPython has always been an interpreter, executing bytecode through a dispatch loop (Shaw Ch.5 describes the `ceval.c` main loop in detail). PEP 744 (experimental in Python 3.13+) introduces a "copy-and-patch" JIT compiler: instead of interpreting bytecode one instruction at a time, it copies pre-compiled machine code templates ("stencils") and patches in the specific operands for each bytecode instruction. This approach is simpler than traditional JIT compilers (HotSpot, V8, PyPy): no intermediate representation, no optimization passes, no register allocation — just template copying and patching. The initial performance improvement is modest (0–9% on benchmarks) because the copy-and-patch JIT is only the first tier: future CPython versions will add optimization passes on top of this foundation, progressively improving performance. The Faster CPython project (led by Mark Shannon, funded by Microsoft) has a multi-year roadmap: Python 3.11 introduced adaptive interpreter specialization (specializing bytecode for observed types), 3.12 added further specialization, 3.13 added the copy-and-patch JIT, and future versions aim for more aggressive optimization. Compare with Java: HotSpot's JIT is vastly more sophisticated (tiered compilation, profile-guided optimization, escape analysis, inlining — decades of engineering). Compare with PyPy: PyPy's tracing JIT achieves 4–10x speedups over CPython by recording and optimizing hot execution traces, but PyPy has a separate ecosystem and limited C extension compatibility. CPython's JIT strategy is different: incremental improvements to the standard interpreter that maintain full C extension compatibility. |
| 7.3 | 25 min | **Structural pattern matching and other recent features.** Read Martelli et al Ch.26 pp. 661–668 (version migration: significant changes 3.7–3.11). Read Martelli et al Appendix pp. 669–685 (feature changelog 3.7–3.11). Read the Python 3.13 and 3.14 What's New documentation online. Key insight: Python 3.10 introduced structural pattern matching (PEP 634), the most significant syntax addition since async/await (Python 3.5). `match` statements support: literal patterns, capture patterns, sequence patterns, mapping patterns, class patterns (deconstructing objects by attributes), OR patterns, guard clauses, and wildcard patterns. The design is more expressive than Java's switch pattern matching and similar in power to Rust's match, though without exhaustiveness checking by default (since Python lacks sum types). Other significant recent features: Python 3.12 — `type` statement for type aliases, simplified generic syntax (`def first[T](lst: list[T]) -> T`), per-interpreter GIL (subinterpreters); Python 3.13 — free-threaded build (experimental), JIT compiler (experimental), improved error messages, `typing.ReadOnly` for TypedDict; Python 3.14 — deferred evaluation of annotations (PEP 649, replacing PEP 563), template strings (PEP 750), `@warnings.deprecated` decorator. The trajectory is clear: Python is simultaneously becoming more expressive (pattern matching, type syntax), faster (JIT, specializing interpreter), and more concurrent (free-threading, subinterpreters). Compare with Rust and Java: all three languages are converging on pattern matching as a core language feature, arriving from different starting points (Rust always had it, Java and Python added it 2021–2022). |
| 7.4 | 15 min | **Python's identity evolution.** Read Slatkin Item 124 pp. 613–620 (static analysis via typing). Key insight: Python is undergoing the most dramatic identity shift of the three languages. In 2014 (pre-PEP 484), Python was an untyped, interpreted, GIL-constrained scripting language optimized for developer productivity. In 2026, Python has: gradual static typing with multiple production-quality type checkers (mypy, pyright, pytype), an experimental JIT compiler, an experimental free-threaded build, structural pattern matching, and protocols for structural subtyping. The language is evolving toward a more capable systems-level tool while maintaining its accessibility. The key tension: every "advanced" feature (typing, pattern matching, free-threading) risks making Python more complex for beginners, undermining the simplicity that made it dominant. The Python community manages this through the "optional complexity" philosophy: typing is optional, pattern matching can be ignored in favor of if/elif chains, and free-threading is an opt-in build. Compare with Rust: Rust was designed from the start with a complex type system — the complexity is the point. Compare with Java: Java has accumulated complexity over 30 years (generics, modules, records, sealed classes, pattern matching, virtual threads) — it manages this through strong backwards compatibility and incremental preview features. Python's unique challenge: remaining the best language for beginners while becoming powerful enough for production systems. |

### Session 8 — Deprecation, Migration & Cross-Language Synthesis

**Goal:** compare deprecation and migration strategies across all three languages, synthesize the evolution patterns observed in Sessions 1–7, and build a framework for understanding how programming languages evolve over decades.

| Step | Time | Activity |
|------|------|----------|
| 8.1 | 30 min | **Deprecation mechanisms compared.** Read Klabnik & Nichols Ch.14 pp. 295–313 (publishing crates, deprecating versions). Read Horstmann Ch.11 pp. 140–149 (@Deprecated annotation). Read Slatkin Item 123 pp. 605–612 (warnings for migration). Read the Rust `#[deprecated]` documentation, Java jdeprscan tool, and Python pyupgrade documentation online. Key insight: each language's deprecation mechanism reflects its evolution philosophy. **Rust**: `#[deprecated(since = "1.0.0", note = "use bar instead")]` emits compiler warnings, and `cargo fix` can automatically apply some migrations. Editions provide the ultimate deprecation mechanism: old syntax can be removed in a new edition while remaining valid in previous editions. Cargo's dependency resolution means that a deprecated crate API only needs to be fixed in the crate that directly uses it — transitive dependencies are unaffected. Crater provides ecosystem-wide impact analysis before any deprecation. **Java**: `@Deprecated(since = "9", forRemoval = true)` marks methods/classes for eventual removal. `jdeprscan` scans JAR files for deprecated API usage. The `forRemoval` flag (added in Java 9) distinguishes "discouraged" from "will be removed." In practice, Java almost never removes deprecated APIs — `Thread.stop()` was deprecated in Java 1.2 (1998) and still exists in Java 21. The JPMS migration (Ch.12) showed that even with deprecation, removing functionality requires command-line flags (`--add-opens`) for years. **Python**: `warnings.warn("msg", DeprecationWarning)` is the standard mechanism, with `FutureWarning` for changes that will affect end users. Python is the most aggressive at actual removal: deprecated features are removed after 2+ versions (typically 2–4 years). The `@warnings.deprecated` decorator (Python 3.14+, PEP 702) standardizes deprecation for functions and classes. pyupgrade automates syntax modernization. The spectrum: Python removes most aggressively, Rust removes via editions (old syntax preserved in old editions), Java almost never removes. |
| 8.2 | 25 min | **Migration tooling and automation.** Read Klabnik & Nichols Appendix D pp. 511–514 (rustfix, Clippy). Read Gjengset Ch.5 pp. 67–84 (versioning, MSRV, changelogs). Read the Cargo semver-checks, Java SE Migration Guide, and pyupgrade documentation online. Key insight: automated migration tooling is increasingly critical as languages evolve faster. **Rust**: `cargo fix --edition` migrates code between editions automatically by applying compiler suggestions. `cargo-semver-checks` detects semver violations in library API changes. Clippy provides lint-based migration suggestions (e.g., suggesting edition-2024-idiomatic patterns). The Rust ecosystem's approach: migration should be boring — `cargo fix` handles 95% of cases, leaving only ambiguous lifetime captures for manual resolution. **Java**: the Java SE Migration Guide documents migration from each LTS version. Key tools: `jdeps` (dependency analysis, detecting internal API usage), `jdeprscan` (deprecated API detection), `jlink` (custom runtime images for modular applications), and IDE migration assistants (IntelliJ IDEA provides automated migrations between Java versions). Multi-release JARs allow a single JAR to contain code for multiple Java versions. **Python**: pyupgrade is the primary automated migration tool, rewriting syntax to use newer features (f-strings, simplified type annotations, walrus operator). `2to3` (historical, for Python 2→3 migration) established the pattern of automated code transformation. `ruff` (Rust-based Python linter) includes pyupgrade rules for fast execution. tox and nox enable testing across multiple Python versions. The trend across all three: migration is becoming increasingly automated, with tooling that can update millions of lines of code with minimal human intervention. |
| 8.3 | 25 min | **Cross-language evolution patterns.** Key insight: synthesize the patterns observed across all eight sessions into general principles of language evolution. (1) **The backwards-compatibility tax**: every successful language accumulates features that cannot be removed. Java carries 30 years of backwards-compatible APIs; Rust manages this through editions; Python accepted the 2→3 trauma to reset but now avoids breaking changes. The cost of backwards compatibility is language surface area: Java 21 has records AND classes, virtual threads AND platform threads, pattern matching AND if/else chains — developers must know which to use when. (2) **Convergent evolution**: all three languages are adopting pattern matching, algebraic data types (Rust enums, Java sealed interfaces + records, Python match/case), and gradual typing (Rust always had it, Java's type system has grown richer, Python is adding it). This suggests that these features address fundamental programming needs regardless of language philosophy. (3) **The governance-velocity correlation**: faster release cadences (Rust > Java > Python) require stronger migration tooling and backwards-compatibility mechanisms. Rust's six-week releases would be untenable without edition-based evolution and crater testing. (4) **Platform vs language evolution**: Java uniquely separates platform (JVM) and language (Java) evolution — the JVM improves performance for all Java code without any code changes, while language features require explicit adoption. Rust and Python couple implementation and language more tightly. (5) **The complexity spiral**: as languages add features to address limitations, they increase cognitive load, which creates pressure for simplification (Kotlin simplifying Java, Rust edition cleanups, Python's "there should be one obvious way to do it" struggling under feature accumulation). |
| 8.4 | 20 min | **Future trajectories and cross-language influence.** Read Gjengset Ch.13 pp. 223–243 (staying up to date with Rust). Key insight: project the evolution trajectories forward. **Rust's trajectory**: stabilizing more async features, improving compile times (the primary user complaint), expanding const evaluation, formalizing the specification, growing adoption in safety-critical domains (automotive, aerospace via Ferrocene). Rust's main evolutionary challenge: managing complexity as features accumulate — the language is already considered difficult to learn, and each new feature adds to the learning curve. **Java's trajectory**: completing Valhalla (value types, eliminating boxing), Leyden (startup optimization), and continuing Amber's syntax modernization. Java's main evolutionary challenge: competing with Kotlin for developer mindshare while maintaining its enterprise reliability brand. Virtual threads (Loom) may be Java's most transformative feature since generics, potentially reversing the trend of Java developers moving to reactive frameworks or other languages for concurrent workloads. **Python's trajectory**: maturing the free-threaded build (potentially defaulting in Python 3.17+), improving the JIT compiler (targeting 2–5x speedups over current CPython), and deepening the typing system. Python's main evolutionary challenge: the GIL removal is the riskiest evolution any of the three languages is attempting — it requires the entire C extension ecosystem to adapt, and failure could fragment the community. **Cross-language influence**: Rust's ownership model has influenced Java (Valhalla's value semantics) and Python (the awareness of zero-copy patterns). Java's virtual threads may influence Python's concurrency evolution post-GIL. Python's typing evolution draws directly from TypeScript's experience. All three languages are watching WebAssembly as a potential universal compilation target. The overarching conclusion: programming languages are living systems that co-evolve with their communities, and understanding their evolution trajectory is essential for making informed technology choices. |
