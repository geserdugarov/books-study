# Layer 5 · Topic 14 — Unsafe Code & Low-Level Access

> The escape hatches each language provides when safety guarantees must be intentionally bypassed — Rust's `unsafe` blocks that unlock raw pointers and direct memory manipulation, Java's evolution from `sun.misc.Unsafe` through `VarHandle` to the Foreign Function & Memory API (Project Panama), and Python's CPython C API with the buffer protocol and ctypes/cffi for native code integration. Understanding when and why to reach for these mechanisms is as important as knowing how they work.

### Coverage Gaps

The owned books **do not** cover:

- **Rust inline assembly (`asm!` macro)** — stabilized in Rust 1.59 (RFC 2873), no owned book covers this. The `asm!` macro allows embedding assembly instructions directly in Rust code with input/output operands and clobber specifications. The Rust Reference and Rust By Example (online) provide the definitive documentation
- **Rust SIMD: portable `std::simd` API** — Matthews Ch.11 mentions SIMD conceptually but uses platform-specific examples. The nightly-only `std::simd` module (portable SIMD) and the stable `std::arch` module (platform-specific intrinsics like `_mm256_add_ps`) need the Rust docs and RFC 2948
- **Miri as a practical UB detection tool** — Gjengset Ch.6 mentions Miri for test augmentation, but no owned book provides a practical walkthrough of running Miri on unsafe code, interpreting its diagnostics, or understanding its limitations (e.g., it cannot detect all UB, does not model hardware). The Miri GitHub repository and its README are the primary resource
- **The Rustonomicon** — the definitive online reference for writing unsafe Rust, covering aliasing rules, variance, PhantomData, drop check, `#[may_dangle]`, Send/Sync implementation, and more. No owned book replicates its depth on these topics
- **`sun.misc.Unsafe` historical API** — Evans & Gough Ch.13 covers VarHandle as the modern replacement, but the historical `sun.misc.Unsafe` API itself (allocateMemory, getInt/putInt, compareAndSwapInt, park/unpark) is not directly covered. Understanding the historical API is important because it explains why VarHandle and the Foreign Memory API were created, and because legacy code and older libraries (Netty, Kafka, off-heap stores) still reference it
- **Foreign Function & Memory API full coverage (JEP 454, finalized Java 22)** — Horstmann Ch.13 provides a brief preview, Evans & Gough Ch.15 discusses Panama at a high level. The full API surface (MemorySegment, Arena, MemoryLayout, FunctionDescriptor, Linker, SymbolLookup) requires the JEP specification and Javadoc
- **Java Vector API code-level walkthrough (JEP 460+)** — Beckwith Ch.9 discusses the Vector API conceptually in the context of exotic hardware, but no owned book provides a code-level tutorial for using the Vector API (VectorSpecies, Vector.fromArray, SIMD lane operations). The JEP and Javadoc are needed
- **Buffer protocol deep dive (PEP 3118)** — Ramalho covers memoryview basics (Ch.2 pp. 67–71), Beazley Ch.15 covers C-side array passing, but no owned book explains the full buffer protocol specification (Py_buffer struct, format strings, shape/strides/suboffsets for multi-dimensional arrays, contiguity flags). PEP 3118 and the CPython C API docs are needed
- **PyO3 / maturin for Rust-Python interop** — not in any owned book. PyO3 enables writing Python extension modules in Rust with automatic reference counting, GIL management, and type conversions. Overlaps with topic 15 (FFI) but contextually relevant here as a modern alternative to ctypes/cffi/Cython

### Session 1 — Philosophy of "Unsafe" and When to Break the Rules

**Goal:** understand what "unsafe" means in each language's context — Rust's formal safety contract with the compiler (unsafe as opt-out from specific guarantees), Java's evolution from unofficial backdoors (sun.misc.Unsafe) to sanctioned APIs (VarHandle, Panama), and Python's pragmatic approach where "unsafe" means "crossing the C boundary." Establish the mental framework for when and why each ecosystem reaches for these escape hatches.

| Step | Time | Activity |
|------|------|----------|
| 1.1 | 35 min | **Rust: the safety contract.** Read Klabnik & Nichols Ch.19 pp. 419–425 (unsafe superpowers: dereference raw pointers, call unsafe functions/methods, access/modify mutable static variables, implement unsafe traits, access union fields, when to use unsafe). Read the Rustonomicon "Meet Safe and Unsafe" and "How Safe and Unsafe Interact" chapters online. Key insight: Rust's `unsafe` keyword does not turn off the borrow checker or type system — it unlocks exactly five additional capabilities. The fundamental contract: `unsafe` code promises the compiler "I have manually verified the invariants that you cannot check." The compiler still enforces all other rules inside `unsafe` blocks. The key mental model: `unsafe` does not mean "dangerous" — it means "the programmer takes responsibility for upholding safety invariants that the compiler cannot verify." A correct `unsafe` block can be wrapped in a safe API (`unsafe` implementation detail, safe public interface). This encapsulation is the core pattern for unsafe Rust: minimize the unsafe surface, maximize the safe abstraction. Contrast with C/C++ where everything is "unsafe" — there is no compiler-enforced boundary between safe and unsafe operations. |
| 1.2 | 30 min | **Java: the evolution of low-level access.** Read Evans & Gough Ch.13 pp. 348–355 (Method and Var Handles, Atomics and CAS, Locks and Spinlocks). Skim the Baeldung "Guide to sun.misc.Unsafe" article online for historical context. Key insight: Java's "unsafe" story has three eras. Era 1 (Java 1.1–8): `sun.misc.Unsafe` — an internal, undocumented class providing raw memory access (allocateMemory, getInt/putInt), CAS operations (compareAndSwapInt), object instantiation without constructors (allocateInstance), and thread parking (park/unpark). It was never part of the public API, yet became a de facto standard — used by Netty, Kafka, Cassandra, Spark, and most high-performance Java libraries. Era 2 (Java 9+): `VarHandle` (JEP 193) and `MethodHandle` — sanctioned replacements for `Unsafe`'s CAS and memory access operations, with proper safety checks and API guarantees. Era 3 (Java 22+): Foreign Function & Memory API (JEP 454) — a complete, safe API for off-heap memory management and native function calls, replacing both `sun.misc.Unsafe` memory operations and JNI. The philosophical difference from Rust: Java's "unsafe" was never a language feature — it was an implementation leak that the platform is now systematically replacing with sanctioned alternatives. |
| 1.3 | 30 min | **Python: crossing the C boundary.** Read Shaw pp. 285–295 (Objects and types: PyObject structure, PyVarObject, the type type). Read Gorelick & Ozsvald Ch.7 pp. 161–168 (JIT vs AOT compilers, why type information helps, the compilation landscape). Read the Python docs "Extending Python with C or C++" introduction online. Key insight: Python has no `unsafe` keyword because CPython itself is written in C — the entire interpreter is "unsafe" from a memory safety perspective. The concept of "unsafe" in Python means crossing the boundary from Python code into C code, where you lose Python's automatic memory management (reference counting) and must manually manage `Py_INCREF`/`Py_DECREF`, handle NULL returns, and avoid dereferencing freed memory. Every C extension module operates in this unsafe territory. The CPython C API provides a structured way to cross this boundary, but it is fundamentally a C API with all of C's dangers. The practical consequence: bugs in C extensions cause segfaults, memory leaks, and corruption rather than Python exceptions. The Python community's approach to "unsafe" is pragmatic: use ctypes/cffi for simple FFI, Cython for performance-critical code with some safety, and raw C extensions only when necessary. |
| 1.4 | 20 min | **Cross-language comparison.** Map the spectrum of "unsafe" across languages. Rust: a formal, compiler-level concept — `unsafe` blocks have precise semantics, the compiler tracks which operations require `unsafe`, and the language community has developed tools (Miri, cargo-careful) and conventions (safety comments, unsafe auditing) to manage unsafe code. Java: an evolving platform concept — started as an unofficial backdoor (`sun.misc.Unsafe`), now being systematically replaced with sanctioned APIs that provide similar capabilities with better safety guarantees (VarHandle for CAS, Foreign Memory API for off-heap, Panama for FFI). Python: a pragmatic boundary — "unsafe" is wherever you write or call C code; the safety boundary is the Python/C API itself, which provides helper functions to manage references and types but cannot prevent C-level bugs. The meta-insight: Rust makes the safe/unsafe boundary a first-class language concept with compiler support; Java makes it a platform evolution story (removing unofficial escape hatches, providing official replacements); Python makes it a runtime boundary (safe Python code vs unsafe C extensions). |

### Session 2 — Rust Unsafe: Raw Pointers and Unsafe Blocks

**Goal:** master Rust's `unsafe` keyword in practice — raw pointer creation and dereferencing, calling unsafe functions/methods, accessing mutable statics, implementing unsafe traits, and accessing union fields. Understand undefined behavior (UB) and the invariants that unsafe code must uphold.

| Step | Time | Activity |
|------|------|----------|
| 2.1 | 35 min | **Unsafe blocks and the five superpowers.** Read Klabnik & Nichols Ch.19 pp. 425–437 (detailed coverage of each superpower with code examples). Read Gjengset Ch.9 pp. 141–148 (the `unsafe` keyword: what it means and what it unlocks, juggling raw pointers). Key insight: creating a raw pointer (`*const T`, `*mut T`) is safe — you can create raw pointers in safe code. Dereferencing a raw pointer is the unsafe operation. This distinction is critical: `let ptr = &x as *const i32;` is safe, but `unsafe { *ptr }` is unsafe. Raw pointers differ from references in four ways: (1) they can be null, (2) they can dangle (point to freed memory), (3) they can alias (`*mut T` and `*const T` to the same location), (4) they are not bound by the borrow checker. Unsafe functions (`unsafe fn`) signal that the caller must uphold certain invariants — the function's documentation should specify exactly what those invariants are (the "safety contract"). Mutable statics are unsafe to access because the compiler cannot verify that no data races occur — this is the only place in Rust where global mutable state exists. Unsafe traits (like `Send` and `Sync`) mean that implementing the trait requires the implementor to uphold invariants the compiler cannot check. Unions allow storing multiple types at the same memory location — reading a union field is unsafe because the compiler cannot verify which variant is currently stored. |
| 2.2 | 35 min | **Raw pointer operations in depth.** Read Blandy & Orendorff Ch.21 pp. 525–555 (unsafe blocks, undefined behavior, unsafe traits, raw pointers — dereferencing, nullable pointers, type casting with `as`, pointer arithmetic, pointer-to-reference conversion). Read McNamara Ch.6 pp. 175–196 (pointers: exploring Rust's reference and pointer types, raw pointer ecosystem, smart pointer building blocks, providing programs with memory). Key insight: raw pointers unlock several operations. `ptr.offset(n)` and `ptr.add(n)` perform pointer arithmetic (both are unsafe — you must ensure the result points within the same allocation). `ptr.read()` and `ptr.write(value)` read/write through raw pointers without moving the pointed-to value. `std::ptr::copy` and `std::ptr::copy_nonoverlapping` (Rust's equivalents of C's `memmove` and `memcpy`). `ptr as *const U` performs type punning (reinterpreting the bits). `ptr.as_ref()` converts a raw pointer to an `Option<&T>` (unsafe — you must ensure the pointer is valid and properly aligned). `std::ptr::null()` and `std::ptr::null_mut()` create null pointers. `NonNull<T>` is a wrapper that guarantees non-null — useful as a building block for unsafe abstractions. The key danger: dereferencing a dangling pointer, a misaligned pointer, or creating two `&mut T` references to the same memory are all UB, and the compiler may optimize code assuming UB never happens — this can cause action-at-a-distance bugs. |
| 2.3 | 30 min | **Undefined behavior and what can go wrong.** Read Gjengset Ch.9 pp. 148–159 (great responsibility: what can go wrong — aliasing violations, validity invariants, panics across FFI boundaries, casting errors, the drop check). Read the Rust Reference "Behavior considered undefined" page online. Key insight: Rust defines specific categories of undefined behavior. (1) Data races: two threads accessing the same memory, at least one writing, without synchronization. (2) Dereferencing null, dangling, or misaligned pointers. (3) Breaking the aliasing rules: creating two `&mut T` to the same location, or `&mut T` and `&T` to the same location (Stacked Borrows model). (4) Producing invalid values: a `bool` that is not 0 or 1, a `char` outside Unicode range, a `&T` or `&mut T` that is null, a `str` that is not valid UTF-8. (5) Unwinding across `extern "C"` function boundaries (may corrupt the stack). (6) Using `std::mem::uninitialized()` or reading uninitialized memory. The practical consequence: UB is not "just a crash" — the compiler assumes UB never happens and optimizes accordingly. This means UB can cause the compiler to delete code, reorder operations, or merge branches in ways that produce correct-seeming output in tests but fail in production with different optimization levels. |
| 2.4 | 20 min | **The Rustonomicon deep dive.** Read the Rustonomicon chapters "Working with Unsafe" and "What Unsafe Can Do" online. Key insight: the Rustonomicon introduces the concept of "unsafe boundary" — the surface area where safe code relies on unsafe code to uphold invariants. A well-designed unsafe abstraction has a small unsafe boundary: the unsafe code is isolated in a few functions, and the public API is entirely safe. The standard library is full of examples: `Vec<T>` uses unsafe internally for raw pointer manipulation, but its public API is safe. The Rustonomicon also introduces the aliasing model (Stacked Borrows / Tree Borrows) — the formal model that defines which pointer operations are legal. Understanding this model is necessary for writing correct unsafe code because violations of the aliasing model are UB even if they "work" on current compilers. |

### Session 3 — Rust Unsafe: Practical Patterns and Auditing

**Goal:** study real-world unsafe Rust patterns — building safe abstractions over unsafe primitives, UnsafeCell as the interior mutability foundation, practical examples (RefWithFlag, GapBuffer), no_std and bare-metal Rust, and tools for verifying unsafe code (Miri, sanitizers).

| Step | Time | Activity |
|------|------|----------|
| 3.1 | 35 min | **Practical unsafe patterns: RefWithFlag and GapBuffer.** Read Blandy & Orendorff Ch.21 pp. 555–575 (example: RefWithFlag — packing a bool flag into a reference's alignment bits; example: GapBuffer — a text editor data structure using raw pointer manipulation for O(1) insertion at the cursor). Key insight: RefWithFlag demonstrates a common unsafe pattern — exploiting type layout knowledge for optimization. Since references are aligned, the lowest bit of a reference's address is always 0 and can be used to store a boolean flag. This is unsafe because it requires pointer arithmetic and bit manipulation, but it is wrapped in a safe API. GapBuffer demonstrates a more complex pattern — managing a dynamically allocated buffer with a gap (uninitialized region) that can be moved to the cursor position for efficient insertion/deletion. This requires raw pointer operations for initialization, copying, and drop. Both examples follow the core pattern: unsafe implementation, safe public API, documented safety invariants. |
| 3.2 | 30 min | **UnsafeCell and interior mutability.** Read Bos Ch.1 pp. 32–33 (interior mutability: Cell, RefCell, Mutex, RwLock, Atomics, UnsafeCell). Read Gjengset Ch.9 pp. 155–158 (validity, casting, the drop check — how UnsafeCell interacts with the compiler's alias analysis). Read the Rust docs for `std::cell::UnsafeCell` online. Key insight: `UnsafeCell<T>` is the only legal way to obtain a `&mut T` from a `&T` in Rust. It is the primitive building block for all interior mutability types: `Cell<T>` wraps `UnsafeCell<T>` with copy semantics, `RefCell<T>` wraps it with runtime borrow checking, `Mutex<T>` wraps it with OS-level locking, `AtomicU32` wraps it with hardware atomic operations. The compiler treats `UnsafeCell` specially: it disables the "shared references are immutable" optimization for the wrapped value. This means `&UnsafeCell<T>` does not imply that `T` won't be mutated — the compiler will not cache reads through shared references to `UnsafeCell`. Understanding `UnsafeCell` is essential because it is the foundation upon which all safe interior mutability abstractions are built. |
| 3.3 | 30 min | **no_std and bare-metal unsafe.** Read Gjengset Ch.12 pp. 211–222 (opting out of the standard library, dynamic memory allocation, the Rust runtime, low-level memory accesses with volatile, misuse-resistant hardware abstraction, cross-compilation). Read McNamara Ch.11 pp. 365–387 (building FledgeOS kernel: boot, CPU interaction, exception handling, VGA output, panic handling). Key insight: `#![no_std]` removes the standard library, leaving only `core` (and optionally `alloc`). This is necessary for bare-metal, embedded, and kernel programming where no OS provides heap allocation or I/O. In no_std contexts, unsafe code is more prevalent: you interact directly with hardware registers (using volatile reads/writes via `std::ptr::read_volatile`/`write_volatile`), manage memory manually, and implement interrupt handlers. Volatile operations prevent the compiler from optimizing away reads/writes to memory-mapped I/O registers (without volatile, the compiler might eliminate a "dead" write to a hardware register). The pattern for hardware abstraction in Rust: define typed wrappers around memory-mapped registers, use volatile operations internally, expose a safe API that prevents misuse (e.g., writing to read-only registers). |
| 3.4 | 25 min | **Tools for auditing unsafe code: Miri and sanitizers.** Read Gjengset Ch.6 pp. 85–100 (testing: Miri and sanitizers mentioned as test augmentation tools). Read the Miri GitHub repository README online. Read the Rustonomicon "Send and Sync" chapter online. Key insight: Miri is an interpreter for Rust's MIR (Mid-level Intermediate Representation) that detects undefined behavior at runtime. It catches: use-after-free, out-of-bounds access, invalid pointer alignment, aliasing violations (Stacked Borrows), data races, invalid enum discriminants, and more. Running `cargo +nightly miri test` executes your test suite under Miri. Limitations: Miri is slow (10–100x slower than native), cannot run all code (no raw syscalls, limited FFI), and only detects UB that is actually executed (not all code paths). `cargo-careful` is a lighter-weight tool that enables extra runtime checks in the standard library. Address Sanitizer (ASan) and Thread Sanitizer (TSan) are LLVM-based tools (`RUSTFLAGS="-Z sanitizer=address" cargo test`) that detect memory errors and data races in compiled code. Best practice: combine Miri (for unsafe-specific UB), sanitizers (for memory/thread bugs), and property-based testing (proptest/quickcheck) to maximize coverage of unsafe code. |
| 3.5 | 15 min | **The Rustonomicon: advanced unsafe topics.** Read the Rustonomicon "PhantomData", "Dropck", and "Send and Sync" chapters online. Key insight: `PhantomData<T>` tells the compiler that a type logically owns a `T` even though it does not physically contain one — necessary for raw pointer wrappers that should drop their pointee. The drop check (`#[may_dangle]`) controls whether the compiler allows dropping a type that contains dangling references — needed for container types like `Vec<T>` that destroy their elements before deallocating memory. Implementing `Send` and `Sync` is unsafe because the compiler cannot verify thread safety for types that contain raw pointers — you must manually ensure that sending the type across threads or sharing it between threads is safe. These are the advanced building blocks that standard library authors use to create safe abstractions. |

### Session 4 — Java Low-Level Access: Unsafe → VarHandle → Panama

**Goal:** trace Java's evolution from the unofficial `sun.misc.Unsafe` through `VarHandle` (Java 9+) to the Foreign Function & Memory API (Java 22+). Understand off-heap memory management, CAS operations, and how Project Panama replaces JNI for native interop.

| Step | Time | Activity |
|------|------|----------|
| 4.1 | 30 min | **`sun.misc.Unsafe` — the historical backdoor.** Read the Baeldung "Guide to sun.misc.Unsafe" article online. Read Evans & Gough Ch.7 pp. 177–179 (hardware memory models — context for why Unsafe was needed). Key insight: `sun.misc.Unsafe` provided capabilities that no public Java API offered: (1) `allocateMemory`/`freeMemory` for off-heap allocation (bypassing GC), (2) `getInt`/`putInt`/`getLong`/`putLong` for reading/writing memory at arbitrary offsets (bypassing field access), (3) `compareAndSwapInt`/`compareAndSwapLong` for CAS operations (the foundation of `java.util.concurrent.atomic`), (4) `allocateInstance` for creating objects without calling constructors, (5) `park`/`unpark` for thread scheduling (the foundation of `LockSupport`). The problem: Unsafe was an internal JDK class with no compatibility guarantees, no safety checks (you can corrupt the heap by writing to wrong offsets), and no documentation. Yet it was used by almost every high-performance Java library because there was no alternative. JEP 471 (Java 23) deprecated Unsafe's memory-access methods for removal, completing the transition to sanctioned APIs. |
| 4.2 | 35 min | **VarHandle and MethodHandle — the sanctioned replacements.** Read Evans & Gough Ch.13 pp. 348–355 (Method Handles and VarHandles, Atomics and CAS, Locks and Spinlocks). Read Beckwith Ch.2 pp. 43–50 (variable handle typed reference, Java 9+). Read the JEP 193 (Variable Handles) specification online. Read the `java.lang.invoke.VarHandle` Javadoc online. Key insight: `VarHandle` (Java 9, JEP 193) provides type-safe access to variables with memory ordering semantics — replacing `Unsafe` for field access and CAS operations. `VarHandle` supports five access modes: (1) plain read/write (no ordering guarantees, like Rust's `Relaxed`), (2) opaque (no reordering with respect to other accesses to the same variable), (3) acquire/release (like Rust's `Acquire`/`Release`), (4) volatile (like Rust's `SeqCst`), (5) CAS operations (`compareAndSet`, `compareAndExchange`, `getAndAdd`). You obtain a VarHandle via `MethodHandles.lookup().findVarHandle(Class, fieldName, fieldType)`. Unlike `Unsafe`, VarHandle performs type checking and access control at lookup time, and memory ordering semantics are explicit in the API. `MethodHandle` is the analogous replacement for `Unsafe`'s reflective method invocation — a typed, directly invokable reference to a method, constructor, or field, optimized by the JIT compiler to be as fast as direct calls. Together, VarHandle + MethodHandle provide everything `Unsafe` did for field/method access, but with safety and performance guarantees. |
| 4.3 | 40 min | **Foreign Function & Memory API (Project Panama).** Read Horstmann Ch.13 (JNI overview and the Foreign Function API preview — read the full chapter for context on JNI's complexity, then focus on section 13.11 "Foreign Functions: A Glimpse into the Future"). Read Evans & Gough Ch.15 pp. 412–414 (Panama). Read JEP 454 (Foreign Function & Memory API) online. Read the `java.lang.foreign` package Javadoc and the Baeldung "Guide to Java Project Panama" article online. Key insight: the Foreign Function & Memory API (finalized in Java 22, JEP 454) provides two capabilities. (1) **Foreign Memory API**: `MemorySegment` represents a contiguous region of memory (on-heap or off-heap). `Arena` manages the lifecycle of memory segments — when an arena is closed, all its segments are freed (deterministic deallocation, like Rust's RAII). `MemoryLayout` describes the structure of data in memory (struct layouts with alignment and padding). This replaces `Unsafe.allocateMemory`/`freeMemory` and `ByteBuffer.allocateDirect`. (2) **Foreign Function API**: `Linker` binds Java method handles to native functions. `FunctionDescriptor` describes a native function's signature (parameter types and return type). `SymbolLookup` finds native functions in loaded libraries. This replaces JNI with a pure-Java API — no C header files, no `javah`, no JNI boilerplate. Example: calling C's `strlen` is approximately 5 lines of Java code with Panama vs 50+ lines with JNI. The performance advantage: Panama avoids JNI's overhead of parameter marshaling and the JNI bookkeeping per call. |
| 4.4 | 20 min | **Comparison with Rust's unsafe.** Key insight: Java's low-level access story differs fundamentally from Rust's. Rust's `unsafe` is a permanent language feature — it will always exist because Rust needs it to build safe abstractions over hardware. Java's `sun.misc.Unsafe` was a temporary escape hatch that the platform is actively removing — the goal is to provide sanctioned APIs (VarHandle, Panama) that offer the same capabilities with safety guarantees. Rust's raw pointers give you unlimited access to memory; Java's `MemorySegment` gives you bounded, lifetime-scoped access to memory with automatic cleanup. Rust's unsafe code can cause UB; Java's `MemorySegment` throws `IllegalStateException` if you access a closed arena (safe failure mode). The tradeoff: Rust gives maximum flexibility and zero overhead at the cost of potential UB; Java gives safety checks at the cost of some overhead (bounds checking, arena lifecycle management). For CAS operations: Rust's `AtomicU64` is a compile-time abstraction with zero overhead; Java's `VarHandle.compareAndSet` is a runtime-resolved handle with JIT-optimized fast paths. |

### Session 5 — Python Low-Level Access: CPython C API, Buffer Protocol, ctypes/cffi

**Goal:** understand how Python programs access low-level capabilities — the CPython C API for writing extension modules, the buffer protocol for zero-copy memory sharing, and ctypes/cffi for calling native libraries without writing C code.

| Step | Time | Activity |
|------|------|----------|
| 5.1 | 35 min | **CPython object model and the C API.** Read Shaw pp. 285–315 (objects and types: PyObject structure, PyVarObject, the `type` type, built-in type implementations). Read Shaw pp. 177–195 (memory management: C memory allocation, CPython memory allocator design, the layered domains). Read the Python docs "Python/C API Reference Manual" introduction online. Key insight: every Python object is a `PyObject` struct in C, containing a reference count (`ob_refcnt`) and a pointer to its type object (`ob_type`). Variable-size objects (lists, strings, tuples) use `PyVarObject` which adds `ob_size`. The CPython C API provides functions to create, manipulate, and destroy Python objects from C code. The critical discipline: every C function that receives a Python object must manage reference counts correctly. `Py_INCREF(obj)` increments the count (you are keeping a reference), `Py_DECREF(obj)` decrements it (you are releasing a reference). A mismatched INCREF/DECREF leads to either memory leaks (forgotten DECREF) or use-after-free (extra DECREF). There are two kinds of references: "borrowed" (you don't own it, don't DECREF) and "new" (you own it, must DECREF when done). The API documentation specifies which kind each function returns. This is the Python equivalent of Rust's ownership/borrowing — but enforced by convention, not the compiler. |
| 5.2 | 35 min | **Writing C extension modules.** Read Beazley & Jones Ch.15 Recipe 15.2 pp. 605–608 (writing a simple C extension module). Read Beazley & Jones Ch.15 Recipe 15.3 pp. 609–611 (extension function operating on arrays). Read Beazley & Jones Ch.15 Recipe 15.4 pp. 612–613 (managing opaque pointers in C extensions). Read Beazley & Jones Ch.15 Recipe 15.5 pp. 614–618 (defining and exporting C APIs from extension modules). Read Shaw pp. 364–369 (writing C extensions for CPython — brief overview). Key insight: a C extension module follows a standard pattern: (1) define module methods using `PyMethodDef` array, (2) define the module using `PyModuleDef`, (3) implement the init function `PyInit_modulename`, (4) parse arguments with `PyArg_ParseTuple` (format strings like `"ii"` for two ints, `"s"` for string), (5) build return values with `Py_BuildValue`. Error handling: set an exception with `PyErr_SetString` and return NULL. The GIL (Global Interpreter Lock) is held by default in C extensions — you must explicitly release it with `Py_BEGIN_ALLOW_THREADS`/`Py_END_ALLOW_THREADS` for long-running C operations to allow other Python threads to run (Beazley Recipe 15.7 pp. 625). Managing opaque pointers (Recipe 15.4): when a C extension needs to return a pointer to a C struct that Python code cannot directly access, use `PyCapsule_New` to wrap the pointer — Python treats it as an opaque object, and the capsule can include a destructor for automatic cleanup. |
| 5.3 | 30 min | **Buffer protocol and memoryview.** Read Ramalho Ch.2 pp. 67–71 (memory views: handling slices of arrays without copying bytes). Read PEP 3118 (Revising the Buffer Protocol) online. Read the Python docs "Buffer Protocol" C API page online. Read the Python docs on `memoryview` online. Key insight: the buffer protocol (PEP 3118) allows Python objects to expose their internal memory as a contiguous buffer that other code can read/write without copying. The `Py_buffer` struct describes the buffer: `buf` (pointer to memory), `len` (total size), `itemsize` (size per element), `format` (struct-style format string like `"d"` for double), `ndim` (number of dimensions), `shape`/`strides`/`suboffsets` (for multi-dimensional arrays). Objects implementing `__buffer__` (or the C-level `bf_getbuffer` slot) can be used with `memoryview()`. `memoryview` provides a Python-level view into the buffer: you can slice it (creating a view, not a copy), cast it to different formats (`mv.cast('B')` to view as bytes), and read/write individual elements. This is Python's closest analogue to Rust's slice (`&[T]`) — a non-owning view into contiguous memory. The practical use: NumPy arrays, `array.array`, `bytes`, `bytearray`, `mmap` objects, and PIL images all support the buffer protocol, enabling zero-copy data sharing between libraries. |
| 5.4 | 30 min | **ctypes and cffi for calling native code.** Read Beazley & Jones Ch.15 Recipe 15.1 pp. 599–604 (accessing C code using ctypes). Read Gorelick & Ozsvald Ch.7 pp. 196–203 (FFI: ctypes, cffi). Read Beazley & Jones Ch.15 Recipes 15.13–15.16 pp. 644–656 (passing strings: NULL-terminated, Unicode, converting C strings to Python, strings of dubious encoding). Read the Python docs on `ctypes` online. Key insight: `ctypes` (stdlib) and `cffi` (third-party) let you call C functions from Python without writing C code. `ctypes` approach: `lib = ctypes.CDLL("libfoo.so")`, then call `lib.foo(arg)`. You must declare argument and return types: `lib.foo.argtypes = [ctypes.c_int, ctypes.c_double]`, `lib.foo.restype = ctypes.c_double`. For complex types: `ctypes.Structure` for C structs, `ctypes.POINTER(ctypes.c_int)` for pointers, `ctypes.CFUNCTYPE` for callback function pointers. `cffi` offers two modes: ABI (load shared library, like ctypes) and API (compile a C extension from C declarations). `cffi`'s API mode is generally preferred because it handles function signatures automatically from C header declarations and produces faster calls. String passing is a common source of bugs: Python 3 strings are Unicode, but C expects `char*` (bytes) — use `bytes` objects or `.encode('utf-8')` when passing to C, and handle the return encoding carefully. Both ctypes and cffi operate outside the GIL for the duration of the C call (unless the C code calls back into Python). |
| 5.5 | 15 min | **Cython and wrapping C with Cython.** Read Beazley & Jones Ch.15 Recipes 15.10–15.11 pp. 632–642 (wrapping C code with Cython, high-performance array operations with Cython). Read Gorelick & Ozsvald Ch.7 pp. 168–186 (Cython: type annotations, NumPy integration, OpenMP parallelization). Key insight: Cython is a superset of Python that compiles to C extension modules. It bridges the gap between Python and C: you write mostly-Python code with optional C type declarations (`cdef int x`, `cdef double[:] arr`), and Cython generates the C extension code automatically, handling all the PyArg_ParseTuple/Py_BuildValue/reference counting boilerplate. For wrapping existing C libraries, Cython can `cdef extern from "header.h"` to declare C functions and types, then write Python wrapper functions that call them. For NumPy integration, typed memoryviews (`double[:, :]`) provide C-speed access to array data with bounds checking. Cython can also release the GIL (`with nogil:`) for parallel execution. Compare with ctypes/cffi: ctypes is simplest (no compilation), cffi is more robust (handles edge cases better), Cython is most powerful (can mix Python and C semantics) but requires a build step. |

### Session 6 — Inline Assembly, SIMD, and Cross-Language Synthesis

**Goal:** explore the deepest level of low-level access — inline assembly (Rust) and SIMD intrinsics across languages — then synthesize all six sessions into a comparative framework for when and why to use unsafe/low-level code in each ecosystem.

| Step | Time | Activity |
|------|------|----------|
| 6.1 | 30 min | **Rust inline assembly (`asm!`).** Read the Rust Reference "Inline Assembly" page online. Read the Rust By Example "Inline Assembly" chapter online. Skim Rust RFC 2873 (Inline Assembly) for design rationale online. Key insight: `asm!` (stabilized in Rust 1.59) allows embedding assembly instructions directly in Rust code. Syntax: `unsafe { asm!("mov {}, 42", out(reg) x) }` — the template string contains assembly instructions, with Rust variables mapped to registers via operands. Operand types: `in(reg)` (input register), `out(reg)` (output register), `inout(reg)` (input+output), `const` (constant), `sym` (symbol reference). `options(nostack, nomem, pure)` communicate to the compiler what the assembly block does, enabling optimization. Use cases: (1) access CPU-specific instructions not available through SIMD intrinsics, (2) implement very specific performance-critical sequences (custom prefetch patterns, cache line operations), (3) interface with hardware in kernel/embedded code (interrupt enable/disable, special registers). `global_asm!` allows defining entire functions in assembly (for boot code, ISR trampolines). Inline assembly is inherently unsafe and platform-specific — it is the absolute lowest level of abstraction Rust provides. Compare with C's `__asm__`: Rust's `asm!` is more structured, with explicit operand binding instead of string-based constraint syntax. |
| 6.2 | 30 min | **SIMD intrinsics in Rust.** Read Matthews Ch.11 pp. 219–231 (SIMD, zero-cost abstractions). Read the Rust docs for `std::arch` module online. Skim the `std::simd` (portable SIMD, nightly) documentation online. Key insight: Rust provides SIMD at two levels. (1) `std::arch` (stable): platform-specific intrinsics — `std::arch::x86_64::_mm256_add_ps` for AVX, `std::arch::aarch64::vaddq_f32` for NEON. These are `unsafe` because the compiler cannot verify that the target CPU supports the instruction (use `#[target_feature(enable = "avx2")]` and runtime detection with `is_x86_feature_detected!("avx2")`). (2) `std::simd` (nightly): portable SIMD — `Simd<f32, 4>` provides platform-independent vector operations that the compiler maps to the best available instructions. Portable SIMD is safe (no `unsafe`) because it abstracts away hardware details. The practical pattern: use auto-vectorization first (the compiler vectorizes loops automatically when possible), then portable SIMD for explicit control, then platform-specific intrinsics only when the compiler's code generation is suboptimal. SIMD is one of the key use cases for `unsafe` in practice — high-performance math, serialization (simdjson), and string processing (memchr) libraries use SIMD intrinsics extensively. |
| 6.3 | 25 min | **Java Vector API (SIMD) and hardware access.** Read Beckwith Ch.9 pp. 307–336 (exotic hardware, TornadoVM, Project Panama, Vector API). Read Evans & Gough Ch.7 pp. 170–179 (hardware: memory caches, TLB, branch prediction, hardware memory models). Read JEP 460 (Vector API) and the Inside.java "Introduction to the Vector API" article online. Key insight: the Java Vector API (incubating since Java 16, JEP 460 in Java 22) provides explicit SIMD operations within the JVM. The API is centered on `VectorSpecies<Float>` (specifying vector type and lane count), `FloatVector.fromArray(species, array, offset)` to load data, and lane-wise operations (`add`, `mul`, `fma`, `compare`, `blend`). The JIT compiler maps these to actual SIMD instructions (`SSE`, `AVX2`, `AVX-512`, `NEON`). Unlike Rust's `std::arch`, Java's Vector API is safe — no `unsafe` required, bounds checked, and the JIT chooses the best instruction set for the running CPU. The tradeoff: less control (cannot access specific instructions), but portable and safe. Java does not support inline assembly — the JVM abstraction prevents direct CPU instruction access. For hardware access beyond SIMD, Java relies on JNI or Panama to call native code, or projects like TornadoVM for GPU offloading. This represents a fundamentally different philosophy from Rust: Java trusts the JVM/JIT to optimize hardware access, while Rust gives the programmer direct control. |
| 6.4 | 15 min | **Python and low-level computation.** Key insight: Python has no inline assembly or direct SIMD support — the interpreter abstraction prevents it. Python's approach to SIMD is indirect: NumPy's internal C/Fortran code uses SIMD (via compiler auto-vectorization and hand-tuned kernels), Cython can generate SIMD-friendly C code, and Numba JIT-compiles Python to LLVM IR which auto-vectorizes. The practical consequence: for compute-intensive work, Python delegates to C/Rust extensions — the buffer protocol (session 5) provides the zero-copy interface for passing data to native SIMD code. This is why Rust is increasingly used as the backend for Python libraries (Polars, Pydantic v2, Ruff, cryptography) — Rust provides unsafe-level SIMD performance with memory safety, and PyO3 provides ergonomic Python bindings. |
| 6.5 | 25 min | **Grand synthesis: when and why to go unsafe.** Build a comprehensive comparison across all six sessions. **Rust**: `unsafe` is a precisely scoped language feature — five superpowers, formal safety contract, compiler-assisted auditing (Miri). Use cases: implementing data structures (Vec, HashMap, linked lists), FFI, SIMD, hardware access, performance-critical hot paths. Philosophy: minimize unsafe surface, maximize safe abstractions, document invariants. **Java**: low-level access evolved from unofficial backdoors to sanctioned APIs — `sun.misc.Unsafe` → `VarHandle` (CAS, memory ordering) + Foreign Function & Memory API (off-heap memory, native calls) + Vector API (SIMD). Use cases: high-performance libraries (off-heap caches, serializers, network frameworks), native interop, hardware acceleration. Philosophy: the platform provides safe alternatives to every low-level operation; prefer sanctioned APIs over Unsafe. **Python**: "unsafe" means crossing the C boundary — C extensions, ctypes/cffi, Cython, buffer protocol. Use cases: wrapping native libraries, performance-critical code, hardware access, scientific computing. Philosophy: Python is the glue — delegate unsafe work to C/Rust and expose safe Python APIs. The meta-insight: all three ecosystems converge on the same pattern — unsafe low-level code wrapped in safe high-level APIs. They differ in where the boundary lives: Rust puts it at the `unsafe` keyword, Java puts it at the API version (Unsafe → VarHandle → Panama), Python puts it at the language boundary (Python → C). |
