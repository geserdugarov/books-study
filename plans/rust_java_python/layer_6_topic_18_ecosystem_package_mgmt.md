# Layer 6 · Topic 18 — Ecosystem & Package Management

### Coverage Gaps

The owned books **do not** cover:

- **crates.io governance and trust model in depth** — Klabnik Ch.14 covers the publishing workflow, but no owned book discusses crates.io governance (owned by the Rust Foundation, policies around name squatting, crate ownership transfer, the crates.io team and their responsibilities), the trust model (any GitHub account can publish, no review process, immutable once published except for yanking), or the recent initiatives around crate signing and Trusted Publishing; the crates.io policies documentation and Rust Foundation blog posts are needed
- **Maven Central governance, publishing process, and Sonatype Nexus** — Evans Ch.11 covers Maven as a build tool, but no owned book explains Maven Central's governance (operated by Sonatype, namespace verification via reverse-DNS and domain ownership, GPG signature requirements, the Central Portal), the publishing process (creating a Sonatype OSSRH account, staging repositories, release promotion), or how Maven Central's namespace model prevents typosquatting more effectively than crates.io or PyPI; the Sonatype Central Portal documentation and Maven Central publishing guides are needed
- **PyPI governance, trust model, and Trusted Publishers** — Slatkin Item 116 and Martelli Ch.7 describe using pip to install from PyPI, but no owned book covers PyPI governance (operated by the Python Software Foundation, sustained by the Python Packaging Authority team), the trust model (any account can publish any unclaimed name, no code review), Trusted Publishers (OIDC-based publishing from CI/CD without API tokens, introduced 2023), or the PyPI Malware Checks project; the PyPI documentation, Warehouse source code, and PSF blog posts are needed
- **Supply chain security tools: cargo-audit, cargo-vet, pip-audit, safety, Dependabot, Renovate** — no owned book covers these tools in depth. `cargo-audit` checks `Cargo.lock` against the RustSec Advisory Database. `cargo-vet` (Mozilla) provides auditing of third-party dependencies. `pip-audit` (Google/Trail of Bits) checks Python dependencies against the OSV database. `safety` (Safety/PyUp) checks against a curated vulnerability database. Dependabot (GitHub) and Renovate (Mend) automate dependency updates across all three ecosystems. The respective tool documentation and GitHub repositories are needed
- **Supply chain attack case studies** — no owned book analyzes real-world supply chain attacks: event-stream (npm, 2018, malicious dependency injection), colors.js/faker.js (npm, 2022, maintainer sabotage), ua-parser-js (npm, 2021, account compromise), PyPI malware campaigns (typosquatting, dependency confusion), crates.io incidents (rust_decimal supply chain attempt). Security blog posts, post-mortem analyses, and the OpenSSF documentation are needed
- **Package signing and verification across all three ecosystems** — Horstmann covers Java JAR signing, but no owned book compares package signing across ecosystems: Sigstore/cargo-audit (Rust), GPG signatures on Maven Central (Java), PEP 740 digital attestations on PyPI (Python), or the broader Sigstore/in-toto/SLSA framework; the respective specification documents and ecosystem-specific guides are needed
- **Standard library philosophy comparison (batteries-included vs minimal core)** — no owned book provides a systematic comparison of why Python ships 300+ stdlib modules (reducing external dependencies but creating maintenance burden), why Rust ships a minimal stdlib and delegates to crates.io (enabling faster iteration but creating ecosystem fragmentation risk), and why Java sits in between (large JDK but modularized via JPMS since Java 9); this requires synthesis from language design documents and community blog posts
- **Key ecosystem libraries comparative survey** — no single owned book surveys the major libraries across all three languages for common tasks: HTTP clients/servers (reqwest/hyper/axum vs Apache HttpClient/OkHttp/Spring WebClient vs requests/httpx/aiohttp), serialization (serde vs Jackson/Gson vs json/msgpack/pickle), CLI (clap vs picocli vs argparse/click/typer), database access (sqlx/diesel/sea-orm vs JDBC/Hibernate/jOOQ vs SQLAlchemy/asyncpg/psycopg); individual libraries are used in owned books but no comparative survey exists
- **Lock file formats and dependency resolution algorithms** — Matthews Ch.2 and Blandy Ch.8 mention `Cargo.lock`, Evans Ch.11 covers Maven dependency resolution, and Martelli Ch.7 covers pip, but no owned book compares the lock file formats (`Cargo.lock` TOML, Maven lacks a native lock file, `requirements.txt`/`poetry.lock`/`uv.lock`), resolution algorithms (Cargo's SAT-solver-based resolver, Maven's nearest-wins strategy, pip's backtracking resolver introduced in 20.3), or reproducibility guarantees
- **Namespace squatting, typosquatting prevention, and package naming policies** — no owned book addresses these registry-level security concerns: how crates.io handles squatted names (crates.io policy allows reclamation), Maven Central's namespace verification (prevents claims on `com.google.*`), PyPI's lack of namespace enforcement (any name is first-come-first-served), or detection tools for typosquatting
- **SBOMs (Software Bill of Materials) and compliance frameworks** — no owned book covers SBOM generation (CycloneDX, SPDX), SLSA (Supply-chain Levels for Software Artifacts), or how `cargo sbom`, Maven CycloneDX plugin, and `cyclonedx-python` generate SBOMs for compliance (Executive Order 14028, EU Cyber Resilience Act)

### Session 1 — Package Registries: Governance, Trust & Publishing

**Goal:** understand how crates.io, Maven Central, and PyPI are governed, who operates them, what trust model each registry provides, how packages are published and verified, and what structural differences in their design affect security posture — particularly namespace policies, signing requirements, and immutability guarantees.

| Step | Time | Activity |
|------|------|----------|
| 1.1 | 30 min | **Rust: publishing to crates.io.** Read Klabnik & Nichols Ch.14 pp. 295–313 (complete chapter: documentation comments, `pub use` for re-exports, crates.io account, metadata, publishing, yanking, workspaces, `cargo install`). Key insight: crates.io is the single centralized registry for the Rust ecosystem — there is no alternative public registry (unlike Java's Maven Central vs JCenter vs GitHub Packages). Publishing requires a GitHub-authenticated API token. Once published, a version is **immutable** — it cannot be deleted, only yanked (which prevents new projects from depending on it but does not remove it from existing `Cargo.lock` files). This immutability is a critical security property: it prevents a malicious actor from replacing a known-good version with a compromised one. The `Cargo.lock` file pins exact versions for reproducible builds. Crate names are globally flat (no namespaces) — `serde`, not `org.serde.serde` — which makes the namespace simpler but more vulnerable to squatting and typosquatting. |
| 1.2 | 25 min | **Rust: crates.io governance and trust model.** Read the crates.io Policies page online. Read the Rust Foundation blog posts on crates.io security. Read The Cargo Book — Publishing on crates.io and Registries sections. Key insight: crates.io is governed by the Rust Foundation and maintained by the crates.io team (a subteam of the Rust project). The trust model is permissive: any GitHub user can create an account and publish any crate under any unclaimed name. There is no code review, no manual approval, and no namespace verification. The crates.io team handles name squatting disputes on a case-by-case basis (the policy allows reclamation of squatted names for legitimate projects). Recent security improvements include: rate limiting on publishing, crate index served via sparse protocol (RFC 2789) for faster and more secure index updates, and ongoing work on Trusted Publishing (OIDC-based publishing from CI/CD, eliminating long-lived API tokens). Compare with Maven Central: crates.io has no namespace verification and no signing requirement, making it structurally more vulnerable to typosquatting but simpler to publish to. Compare with PyPI: similar permissive model (flat namespace, no review), but PyPI has implemented Trusted Publishers ahead of crates.io. |
| 1.3 | 30 min | **Java: Maven Central and the Sonatype ecosystem.** Read Evans et al Ch.11 pp. 345–375 (Maven: lifecycle, POM structure, dependencies, repositories). Read the Sonatype Central Portal documentation online (Getting Started, Publishing Requirements, GPG Signing). Read Baeldung "Guide to Publishing to Maven Central" online. Key insight: Maven Central is the most locked-down of the three registries. Publishing requires: (1) a verified namespace — you must prove ownership of the reverse-DNS groupId (e.g., `com.google.guava` requires proof of `google.com` domain ownership or `io.github.username` via GitHub account verification), (2) GPG-signed artifacts — every JAR, POM, and source artifact must be signed with a GPG key and the public key must be available on a keyserver, (3) mandatory metadata — `groupId`, `artifactId`, `version`, SCM URL, license, developer info, (4) Javadoc and source JARs alongside the compiled JAR. The publishing workflow goes through a staging repository on Sonatype OSSRH (OSS Repository Hosting) before promotion to Central. This multi-step verification makes Maven Central significantly harder to typosquat — an attacker cannot claim `com.google.guava2` without proving ownership of `google.com`. The trade-off: publishing is complex and slow (first-time setup takes hours), which raises the barrier for both legitimate developers and attackers. Maven does not have a native lock file — `pom.xml` specifies version ranges and Maven uses a "nearest-wins" resolution strategy, which can lead to non-reproducible builds without the Maven Dependency Lock plugin or Gradle's lock file mechanism. |
| 1.4 | 30 min | **Python: PyPI, Trusted Publishers, and the packaging ecosystem.** Read Slatkin Ch.14 pp. 575–581 (Item 116: finding community modules on PyPI; Item 117: virtual environments). Read Martelli et al Ch.7 pp. 221–245 (pip, wheels, pyproject.toml, virtual environments). Read the PyPI Trusted Publishers documentation online. Read the Python Packaging User Guide online (packaging-projects tutorial). Key insight: PyPI (Python Package Index) is operated by the Python Software Foundation and maintained by the Python Packaging Authority (PyPA) team. Like crates.io, PyPI has a flat namespace — any authenticated user can publish any unclaimed package name with no namespace verification. Publishing historically used long-lived API tokens or username/password, but since 2023, Trusted Publishers (OIDC-based publishing from GitHub Actions, GitLab CI, Google Cloud Build) is the recommended method — it eliminates stored credentials entirely. PyPI now supports digital attestations (PEP 740, using Sigstore) that cryptographically link a published package to its source repository and CI/CD build. Python's packaging landscape is fragmented: `setuptools` (legacy, still dominant), `hatch` (modern, PEP 517 native), `flit` (simple, for pure Python), `maturin` (for Rust-Python extensions), all using `pyproject.toml` (PEP 518) as the unified configuration file. Lock files are not standardized: `requirements.txt` (pip freeze output), `poetry.lock` (Poetry), `uv.lock` (uv), `pdm.lock` (PDM) — each tool has its own format. pip's dependency resolver (backtracking resolver since pip 20.3) is substantially slower than Cargo's SAT-solver-based resolver. |
| 1.5 | 25 min | **Cross-language comparison: registry trust models.** Build a comparison table. **crates.io**: flat namespace, GitHub auth, no signing required, immutable versions, name squatting handled by policy; most permissive. **Maven Central**: namespace-verified (reverse-DNS), GPG signing mandatory, staging + promotion workflow, domain ownership proof; most restrictive. **PyPI**: flat namespace, Trusted Publishers (OIDC), PEP 740 digital attestations (Sigstore), no signing requirement for upload (attestations are optional); evolving rapidly. The structural insight: Maven Central's namespace verification is the strongest defense against typosquatting — an attacker cannot claim a similar-looking groupId without owning a similar domain. crates.io and PyPI rely on detection (automated scanning for malicious packages) rather than prevention (namespace verification). The resolution algorithm comparison: Cargo uses a SAT solver (PubGrub) that finds a globally consistent solution or reports a clear conflict; Maven uses "nearest-wins" (first version encountered in the dependency tree wins, potentially leading to silent version conflicts); pip uses a backtracking resolver (correctness-first, can be slow on complex dependency graphs). **Hands-on activity**: create a trivial package in each ecosystem (a `hello-world` crate/artifact/package) and walk through the publishing flow locally without actually publishing — examine the metadata requirements, signing steps, and lock file formats. |
| 1.6 | 20 min | **Lock files and reproducibility.** Read Matthews Ch.2 pp. 17–25 (Cargo.lock, dependency management). Read Blandy & Orendorff Ch.8 pp. 175–185 (versions, Cargo.lock). Read Gjengset Ch.5 pp. 75–84 (versioning, minimal dependency versions). Key insight: reproducible builds require pinning exact dependency versions. Cargo.lock is the gold standard: it pins every transitive dependency to an exact version and is checked into version control for binaries (but not for libraries, since libraries should be compatible with a range of dependency versions). Maven has no native lock file — the Dependency Lock Plugin or Gradle's `--write-locks` provides this. Python's ecosystem is fragmented: `pip freeze > requirements.txt` captures a snapshot but includes no integrity hashes; `poetry.lock` and `uv.lock` include content hashes for integrity verification. **Hands-on activity**: examine a real `Cargo.lock` file, a `pom.xml` with dependency management section, and a `poetry.lock` or `uv.lock` — compare what metadata each captures (versions, hashes, source URLs, resolution strategy). |

### Session 2 — Standard Library Philosophy & Ecosystem Shape

**Goal:** understand how each language's standard library philosophy — Python's "batteries included" approach shipping 300+ modules, Rust's deliberately minimal `std` that delegates to the crate ecosystem, and Java's large but now modularized JDK — shapes the structure, fragmentation risk, and innovation speed of the third-party ecosystem.

| Step | Time | Activity |
|------|------|----------|
| 2.1 | 30 min | **Rust: the minimal standard library.** Read Gjengset Ch.13 pp. 223–243 (the Rust ecosystem: tools, major libraries, patterns in the wild, what a Rust standard library "really" looks like in practice). Read Gjengset Ch.5 pp. 67–74 (features, conditional dependencies — how the ecosystem composes). Key insight: Rust's standard library is intentionally minimal. It provides: primitive types, collections (`Vec`, `HashMap`, `BTreeMap`), I/O traits, threading primitives, networking basics (`TcpStream`, `UdpSocket`), and the `Future` trait — but NOT: an HTTP client, JSON parsing, async runtime, CLI argument parsing, database drivers, regular expressions, or random number generation. This is a deliberate design choice: including something in `std` means it must maintain backward compatibility forever (Rust's stability guarantee). By keeping `std` small, the ecosystem can iterate faster — serde can release breaking changes between major versions, something impossible for std types. Gjengset's Ch.13 describes the "real standard library" that experienced Rust developers treat as near-essential: `serde` (serialization), `tokio` (async runtime), `tracing` (logging/diagnostics), `rand` (random numbers), `regex` (regular expressions), `clap` (CLI), `reqwest` (HTTP). The trade-off: ecosystem fragmentation risk (multiple competing crates for the same task), discovery burden (newcomers must learn which crates are "blessed"), and supply chain surface area (every crate is a dependency to audit). Blessed.rs and lib.rs help with discovery. |
| 2.2 | 30 min | **Java: the large but modularized JDK.** Read Horstmann Core Java II Ch.12 pp. 293–320 (JPMS: module declarations, exporting, services, jlink). Read Evans et al Ch.2 pp. 26–40 (Jigsaw motivation, module graph, platform modules). Key insight: Java takes a middle path. The JDK ships a large standard library (~4,000 classes across ~30 modules as of JDK 21), including: collections, I/O, networking, HTTP client (`java.net.http`, since JDK 11), JDBC (database connectivity), XML parsing, cryptography, logging (`java.util.logging`), concurrency utilities, and regular expressions. Since Java 9, the JDK is modularized via JPMS — applications can use `jlink` to create a custom runtime image containing only the modules they use, reducing footprint. The large stdlib means Java developers have fewer external dependencies for basic tasks (HTTP, JSON via `javax.json`, XML, crypto are all in the JDK), but the JDK's slower release cycle means stdlib APIs often lag behind third-party alternatives (the JDK HTTP client is less ergonomic than OkHttp; `java.util.logging` is less flexible than SLF4J/Logback). The JDK's backward compatibility guarantee (Java code from 2004 still compiles on JDK 21) means deprecated APIs linger for decades. The JDK module system provides encapsulation (`exports` directive controls which packages are accessible), but the `--add-opens` escape hatch is widely used by frameworks (Spring, Hibernate) that rely on deep reflection. |
| 2.3 | 30 min | **Python: batteries included and the dead batteries problem.** Read Martelli et al Ch.1 pp. 1–19 (Python ecosystem overview: governance via PEPs, the PSF, standard library philosophy). Read Slatkin Ch.14 pp. 575–576 (Item 116: community modules — when the stdlib is not enough). Read Brett Cannon's blog post "The Python standard library's problematic size" online. Read PEP 594 — Removing dead batteries online. Key insight: Python's "batteries included" philosophy means the standard library ships over 300 modules covering: file I/O, networking, HTTP (`urllib`), email (`smtplib`, `email`), JSON, XML, CSV, SQLite (`sqlite3`), regular expressions, logging, unittest, argparse, and much more. This reduces the dependency count for simple projects — a basic HTTP request, JSON parsing, and argument parsing require zero third-party packages. However, the stdlib has become a double-edged sword. PEP 594 (Python 3.11–3.13) removes 19 "dead battery" modules (`aifc`, `cgi`, `telnetlib`, etc.) that were unmaintained and outdated. Brett Cannon's analysis argues that the large stdlib creates maintenance burden (every module must be maintained by the CPython core team), can block innovation (third-party `requests` is vastly superior to `urllib`, but `urllib` cannot be removed), and creates a false sense of quality (presence in stdlib does not guarantee active maintenance). The trend is toward a leaner stdlib with the ecosystem filling gaps. Compare: Python ships `argparse` and `json` in stdlib; Rust requires `clap` and `serde` as external crates; Java ships `java.net.http` and `javax.json` in the JDK but most projects use OkHttp and Jackson. |
| 2.4 | 25 min | **How standard library size shapes ecosystem dynamics.** Key insight: the standard library philosophy has cascading effects on the entire ecosystem. **Rust (minimal)**: forces a vibrant crate ecosystem, enables rapid iteration (serde has gone through multiple major versions), creates discovery burden (newcomers ask "which HTTP crate?"), increases supply chain surface area, and leads to "crate review" culture (`cargo-vet`, security auditing). **Java (large JDK)**: reduces dependency count for basic tasks, slows innovation for JDK-included functionality (logging, HTTP), creates "framework" culture (Spring/Quarkus aggregate many libraries into opinionated stacks), and the modular JDK (jlink) allows trimming unused modules. **Python (batteries included)**: lowest barrier to entry for beginners (everything just works out of the box), creates stdlib vs third-party tension (`urllib` vs `requests`, `unittest` vs `pytest`, `argparse` vs `click`), and the dead batteries problem forces ongoing curation. The meta-insight: each philosophy optimizes for a different value. Rust optimizes for ecosystem velocity and minimal coupling. Java optimizes for enterprise stability and backward compatibility. Python optimizes for beginner accessibility and self-contained deployability. None is objectively better — the right choice depends on the project's longevity, team size, and security posture. **Hands-on activity**: for a task like "parse a JSON config file, make an HTTP GET request, and write the result to a SQLite database" — count how many external dependencies each language requires: Python needs zero (json, urllib, sqlite3 are all in stdlib), Java needs one or two (Jackson for better JSON, or zero if using JDK classes), Rust needs three or more (serde + serde_json, reqwest, rusqlite). |
| 2.5 | 20 min | **Ecosystem discovery and quality signals.** Read Matthews Ch.3 pp. 43–50 (tooling ecosystem, cargo-tree). Browse Blessed.rs, lib.rs (Rust), mvnrepository.com (Java), and PyPI (Python) online. Key insight: each ecosystem has different quality signals for choosing libraries. **Rust**: downloads on crates.io, GitHub stars, last updated date, `#![forbid(unsafe_code)]` in crate metadata, RustSec advisory status, Blessed.rs recommendations. **Java**: Maven Central download counts (via mvnrepository.com), the Spring/Quarkus endorsement ecosystem (being part of a major framework stack is a strong signal), Baeldung tutorial coverage, Apache/Eclipse Foundation stewardship. **Python**: PyPI download stats (via pypistats.org), GitHub stars, "awesome-python" lists, whether the package is well-typed (has py.typed marker), integration with major frameworks (Django, FastAPI). The ecosystem shape also affects discoverability: Rust's single registry (crates.io) makes discovery simple; Java's multiple registries (Maven Central, JCenter until 2021, GitHub Packages, corporate Nexus instances) fragment discovery; Python's single registry (PyPI) is simple but the naming chaos (multiple packages for the same purpose with similar names) creates confusion. **Hands-on activity**: for each language, find the most-downloaded HTTP client library, serialization library, and CLI library — note the download counts, last publish date, and maintenance signals. |

### Session 3 — Key Ecosystem Libraries for Common Tasks

**Goal:** survey the major libraries that each language's ecosystem has converged on for four common tasks — HTTP clients, serialization, CLI argument parsing, and database access — understanding the design philosophy behind each library, its API ergonomics, and how it reflects the language's type system and idioms.

| Step | Time | Activity |
|------|------|----------|
| 3.1 | 35 min | **HTTP: reqwest/hyper (Rust) vs HttpClient/OkHttp (Java) vs requests/httpx (Python).** Read McNamara Ch.8 pp. 251–291 (reqwest for HTTP in Rust: async requests, error handling, practical patterns). Browse the reqwest and hyper crate documentation online. Browse the OkHttp documentation and the Java `java.net.http.HttpClient` Javadoc online. Browse the requests and httpx documentation online. Key insight: HTTP libraries reflect each language's core strengths. **Rust**: `reqwest` is the high-level HTTP client (built on `hyper` which provides the low-level HTTP implementation). reqwest provides async-by-default `Client::get(url).send().await?` with connection pooling, TLS via `rustls` or `native-tls`, automatic redirect following, cookie management, and JSON deserialization via serde (`response.json::<MyStruct>().await?`). hyper is the foundation — used directly when you need a custom HTTP stack or server. The type system enables zero-cost abstractions: a `Response` object owns its body bytes, preventing use-after-read bugs at compile time. **Java**: the JDK ships `java.net.http.HttpClient` (since JDK 11) supporting sync/async, HTTP/2, WebSocket. However, OkHttp (Square) remains more popular for its interceptor pattern, simpler API, and connection pooling. Spring's `WebClient` adds reactive HTTP on top of Netty. **Python**: `requests` (sync, the most-downloaded Python package ever) provides the gold-standard developer experience: `requests.get(url).json()`. `httpx` (encode.io) extends this to async: `async with httpx.AsyncClient() as client: r = await client.get(url)`. Python's HTTP libraries prioritize developer ergonomics over performance. **Hands-on activity**: write a simple HTTP GET request and JSON parse in all three languages — compare the code verbosity, error handling patterns, and dependency count. |
| 3.2 | 35 min | **Serialization: serde (Rust) vs Jackson (Java) vs json/msgpack (Python).** Read McNamara Ch.7 pp. 212–249 (serde for serialization: JSON, bincode, derive macros for Serialize/Deserialize). Read Bloch Ch.12 pp. 339–365 (Java serialization dangers, alternatives: JSON, Protocol Buffers). Browse the serde documentation (serde.rs) online. Browse the Jackson documentation and Baeldung Jackson tutorial online. Key insight: serialization is where each language's type system shines or struggles. **Rust**: `serde` is arguably the most important crate in the Rust ecosystem. `#[derive(Serialize, Deserialize)]` on a struct generates zero-cost serialization code at compile time via proc macros. serde is format-agnostic — the same derive works with `serde_json` (JSON), `serde_yaml` (YAML), `bincode` (binary), `toml`, `csv`, and dozens more. Attribute macros customize field names (`#[serde(rename = "...")]`), default values, and flattening. serde's compile-time code generation means no runtime reflection — serialization is as fast as hand-written code. **Java**: Jackson is the dominant JSON library. `ObjectMapper.readValue(json, MyClass.class)` uses runtime reflection to map JSON fields to Java object fields. Annotations (`@JsonProperty`, `@JsonIgnore`, `@JsonCreator`) customize mapping. Jackson supports streaming (SAX-like), tree model (DOM-like), and data binding (POJO mapping). Java's native serialization (`Serializable` interface) is discouraged due to security vulnerabilities (Bloch Items 85–90) — deserialization can execute arbitrary code. **Python**: `json.dumps(obj)` and `json.loads(text)` are in the stdlib. For classes, `dataclasses.asdict()` + `json.dumps()` provides basic serialization. `msgpack` provides efficient binary serialization. `pydantic` (third-party) provides validation + serialization with type hints — it is the Python equivalent of serde + validation, and its v2 core is written in Rust. **Hands-on activity**: define a data structure with nested fields and optional values — serialize/deserialize it in all three languages, noting how each handles missing fields, type mismatches, and unknown fields. |
| 3.3 | 30 min | **CLI: clap (Rust) vs picocli (Java) vs argparse/click (Python).** Read Gjengset Ch.13 pp. 230–235 (ecosystem libraries including clap). Browse the clap crate documentation online (derive API). Browse the picocli documentation online. Browse the Python argparse and click documentation online. Key insight: CLI argument parsing reveals each language's approach to compile-time vs runtime validation. **Rust**: `clap` (Command Line Argument Parser) supports both a derive API (`#[derive(Parser)]` on a struct) and a builder API. The derive API is the preferred approach: each struct field becomes a CLI argument/flag, types are validated at parse time, and `--help` output is auto-generated from doc comments. `clap` enforces at compile time that argument types are parseable (a `u32` field will only accept valid unsigned integers). Subcommands are modeled as enums. **Java**: `picocli` is the leading CLI library. It uses annotations (`@Command`, `@Option`, `@Parameters`) on classes or methods. picocli generates ANSI-colored help, supports subcommands, tab completion, and GraalVM native-image compilation. Java's annotation processing generates code at compile time, but argument validation is still at runtime. **Python**: `argparse` (stdlib) provides the baseline — `parser.add_argument('--name', type=str, required=True)`. `click` (third-party) uses decorators: `@click.command() @click.option('--name')`. `typer` (built on click, by FastAPI's author) uses type hints: `def main(name: str)` — the closest Python equivalent to clap's derive API. Python CLI libraries rely entirely on runtime validation. **Hands-on activity**: build a CLI tool that accepts a `--url` flag and a `--format` flag (json/yaml) — compare how each language validates arguments, generates help text, and handles errors. |
| 3.4 | 35 min | **Database: sqlx/diesel (Rust) vs JDBC/Hibernate (Java) vs SQLAlchemy/asyncpg (Python).** Read Evans et al Ch.11 pp. 375–399 (Gradle dependencies, build tool integration — relevant for understanding how Java projects pull in JDBC drivers and Hibernate). Browse the sqlx, diesel, JDBC, Hibernate, SQLAlchemy, and asyncpg documentation online. Key insight: database access reveals the deepest type system differences. **Rust**: `sqlx` checks SQL queries at compile time against a live database — a typo in a column name is a compile error, not a runtime crash. `diesel` provides a type-safe query builder where SQL is expressed as Rust types (table schemas are defined in Rust, and the compiler verifies that joins, filters, and projections are valid). The trade-off: Rust database libraries require more setup (diesel requires schema migrations, sqlx requires a database connection at compile time or cached metadata). **Java**: JDBC is the low-level standard (part of the JDK). `DriverManager.getConnection()` + `PreparedStatement` + `ResultSet` — manual mapping, SQL injection prevention via parameter binding. Hibernate (JPA implementation) provides ORM: entity classes annotated with `@Entity`, `@Table`, `@Column` are mapped to database tables, and HQL/JPQL queries operate on objects. jOOQ provides a type-safe DSL: `dsl.select(AUTHOR.NAME).from(AUTHOR).where(AUTHOR.ID.eq(1))` — compile-time-checked SQL similar to diesel. **Python**: `sqlite3` is in stdlib. SQLAlchemy provides both Core (SQL expression language) and ORM (declarative mapping). `asyncpg` (for PostgreSQL) provides the fastest async database driver. `psycopg3` provides sync/async PostgreSQL access. Python database libraries prioritize developer ergonomics and rapid prototyping. **Hands-on activity**: compare a "query all users where age > 30" operation in sqlx (compile-time checked), Hibernate (runtime entity mapping), and SQLAlchemy (expression language) — note how each language catches errors (compile time, startup time, query execution time). |
| 3.5 | 20 min | **Cross-language ecosystem survey.** Build a comprehensive comparison table across all four domains (HTTP, serialization, CLI, database): library name, approach (compile-time vs runtime), primary strength, main trade-off. Key insight: the ecosystem library landscape mirrors the language's philosophy. Rust libraries maximize compile-time safety (serde's zero-cost codegen, sqlx's compile-time SQL checking, clap's derive-based validation) at the cost of higher initial complexity. Java libraries leverage runtime reflection and annotations (Jackson, Hibernate) for convenience, backed by enterprise-grade reliability and decades of battle-testing. Python libraries optimize for developer experience and minimal boilerplate (`requests.get(url).json()` is unbeatable for simplicity) at the cost of runtime-only error detection. The ecosystem maturity also differs: Java's library ecosystem is the oldest and most stable (Hibernate has been production-ready since 2003). Python's is the most accessible and fastest-growing. Rust's is the youngest but innovating most rapidly (compile-time SQL checking, zero-cost serialization are unique to Rust). |

### Session 4 — Supply Chain Security

**Goal:** understand the supply chain security landscape across all three ecosystems — the advisory databases (RustSec, NVD/OSV, GitHub Advisory), the auditing tools (`cargo-audit`, `cargo-vet`, OWASP Dependency-Check, `pip-audit`, `safety`), the automated update tools (Dependabot, Renovate), real-world attack case studies, and the emerging standards for SBOMs and package provenance (SLSA, Sigstore, CycloneDX).

| Step | Time | Activity |
|------|------|----------|
| 4.1 | 30 min | **Rust: cargo-audit, cargo-vet, and cargo-deny.** Read Matthews Ch.3 pp. 50–55 (cargo-fuzz, CI/CD integration — security tooling context). Read the cargo-audit GitHub README online. Read the cargo-vet documentation online (Introduction, How It Works, Getting Started). Read the cargo-deny documentation online. Key insight: Rust has the most structured approach to dependency auditing. **`cargo-audit`** scans `Cargo.lock` against the RustSec Advisory Database (a community-maintained database of Rust crate vulnerabilities). Usage: `cargo audit` — reports known vulnerabilities with severity, advisory ID, and recommended upgrade version. RustSec advisories are filed as TOML files in a GitHub repository, enabling pull-request-based review. **`cargo-vet`** (by Mozilla) goes further: it maintains a list of which crate versions have been audited by your organization and trusted peers. When a dependency updates, `cargo vet` flags unaudited versions. Audits can be shared between organizations via import chains — if Mozilla has audited `serde 1.0.195`, your organization can import that audit. This creates a web-of-trust model for dependency auditing. **`cargo-deny`** is a linter for `Cargo.toml` dependencies: it checks licenses (deny GPL in a proprietary project), bans specific crates, detects duplicated dependencies (two versions of the same crate), and checks advisories. These three tools compose: `cargo-audit` for known vulnerabilities, `cargo-vet` for audit provenance, `cargo-deny` for policy enforcement. **Hands-on activity**: run `cargo audit` on an existing Rust project (or create one with an intentionally outdated dependency) — examine the advisory output, understand the severity levels, and practice `cargo audit fix`. |
| 4.2 | 30 min | **Java: OWASP Dependency-Check and Maven/Gradle security plugins.** Read Bloch Ch.12 pp. 339–345 (Java serialization security — a supply chain vector). Read the OWASP Dependency-Check documentation online. Read Baeldung "OWASP Dependency Check with Maven" online. Read the CycloneDX Maven Plugin documentation online. Key insight: Java's supply chain security relies primarily on the National Vulnerability Database (NVD) and the GitHub Advisory Database. **OWASP Dependency-Check** is the most widely used tool: it analyzes project dependencies (from Maven/Gradle), identifies them using Common Platform Enumeration (CPE), and checks against the NVD for known CVEs. Integration: `mvn org.owasp:dependency-check-maven:check` generates an HTML report listing vulnerable dependencies with CVE IDs, CVSS scores, and upgrade recommendations. False positives are common (CPE matching is heuristic-based) — suppression XML files exclude known false positives. **Sonatype OSS Index** provides an alternative vulnerability database with better accuracy for Java artifacts. **Maven Enforcer Plugin** can ban specific dependency versions or require minimum versions. The CycloneDX Maven Plugin generates SBOM files in CycloneDX JSON/XML format — listing all dependencies, their versions, licenses, and hashes for compliance and auditing. **Hands-on activity**: add `dependency-check-maven` to a Java project's `pom.xml`, run `mvn verify`, and examine the generated vulnerability report — compare the output format with `cargo audit`'s output. |
| 4.3 | 30 min | **Python: pip-audit, safety, and Bandit.** Read Viafore Ch.20 pp. 285–296 (security analyzers including Bandit). Read the pip-audit GitHub README online. Read the safety GitHub README online. Read the Bandit documentation online. Key insight: Python's supply chain security tools address two distinct problems: vulnerable dependencies and insecure code patterns. **`pip-audit`** (maintained by PyPA/Trail of Bits) checks installed packages against the OSV (Open Source Vulnerabilities) database and PyPI's own advisory feed. Usage: `pip-audit` — lists vulnerable packages with vulnerability IDs and fix versions. `pip-audit` can also audit `requirements.txt` files without installing packages (`pip-audit -r requirements.txt`). **`safety`** (by Safety/PyUp) checks against a curated vulnerability database (partially commercial — the free database has a 30-day delay). Safety was historically the dominant tool but `pip-audit`'s open database (OSV) and PyPA backing have made it the recommended choice. **Bandit** is a code scanner (not a dependency scanner) — it finds security issues in Python source code: hardcoded passwords, use of `eval()`, insecure `pickle.loads()`, SQL injection via string formatting, weak cryptography. Bandit complements dependency scanners by catching in-code vulnerabilities. The CycloneDX Python tool generates SBOMs from `requirements.txt`, `poetry.lock`, or `pipenv` lock files. **Hands-on activity**: run `pip-audit` on a Python virtual environment with an intentionally outdated package (e.g., an old version of `requests` with a known CVE) — examine the advisory output, then run `bandit -r src/` on a Python project to see code-level security findings. |
| 4.4 | 30 min | **Cross-ecosystem automated updates: Dependabot and Renovate.** Read the Dependabot documentation online (Configuring Dependabot version updates, security updates). Read the Renovate documentation online (Getting Started, Configuration). Key insight: automated dependency update tools work across all three ecosystems and are the most impactful supply chain security improvement an organization can make. **Dependabot** (GitHub-native) monitors `Cargo.toml`/`Cargo.lock`, `pom.xml`/`build.gradle`, and `requirements.txt`/`pyproject.toml` for outdated or vulnerable dependencies and automatically creates pull requests with version bumps. Configuration via `.github/dependabot.yml` specifies update frequency (daily/weekly), package ecosystems, and version update strategy. Dependabot security updates are triggered immediately when a new advisory is published. **Renovate** (by Mend, open source) provides the same functionality with more flexibility: custom grouping of updates, auto-merge for minor/patch updates with passing CI, regex-based manager support for custom file formats, and a dashboard issue for tracking pending updates. Both tools represent the same philosophy: keeping dependencies up to date is the single most effective defense against known vulnerabilities. The challenge is alert fatigue — large projects may receive dozens of PRs per week. Strategies: group related updates, auto-merge patch versions, schedule updates for specific days, and use lock file maintenance (Renovate) to keep transitive dependencies current. **Hands-on activity**: examine a `.github/dependabot.yml` configuration from a popular open-source project (e.g., the `serde` repo, Spring Boot, or FastAPI) — understand the update schedule, ecosystem configuration, and ignore rules. |
| 4.5 | 25 min | **Supply chain attacks: case studies and prevention.** Read the OpenSSF "Package Repository Security" report online. Read post-mortem analyses of major supply chain attacks online. Key insight: understanding real attacks illuminates why supply chain security tools exist. **event-stream (npm, 2018)**: a maintainer transferred ownership of a popular package to an unknown contributor who added a malicious dependency (`flatmap-stream`) targeting the Bitcoin wallet Copay — demonstrates the risk of maintainer transfer in single-maintainer packages. **ua-parser-js (npm, 2021)**: an attacker compromised the maintainer's npm account and published versions with a cryptocurrency miner and credential stealer — demonstrates the risk of account compromise (mitigated by 2FA and Trusted Publishers). **PyPI malware campaigns (ongoing)**: typosquatted packages (`python-dateutil` → `python3-dateutil`, `requests` → `request`) that install info-stealers — demonstrates why namespace verification (Maven Central) or typosquatting detection (PyPI Malware Checks) is essential. **colors.js/faker.js (npm, 2022)**: the maintainer intentionally sabotaged their own packages as a protest — demonstrates that even trusted maintainers can become threat actors. **Dependency confusion (2021)**: Alex Birsan's research showed that internal package names can be hijacked on public registries (if a company uses `internal-utils` as a private package, an attacker publishes `internal-utils` to PyPI/npm with a higher version number) — mitigated by scoped packages (npm), private registries, and Cargo's registry configuration. Prevention strategies: enable 2FA on all registry accounts, use Trusted Publishers (eliminate stored tokens), pin dependencies with lock files and hash verification, run `cargo-audit`/`pip-audit`/OWASP DC in CI, adopt Dependabot/Renovate for automated updates, and consider `cargo-vet` for high-security projects. |
| 4.6 | 25 min | **SBOMs, SLSA, and the future of supply chain security.** Read the SLSA specification online (slsa.dev — Levels, Requirements, Provenance). Read the Sigstore documentation online. Read the CycloneDX and SPDX specifications online. Key insight: the software industry is converging on standards for supply chain transparency, driven by government mandates (US Executive Order 14028, EU Cyber Resilience Act). **SBOM (Software Bill of Materials)**: a machine-readable inventory of all components in a software product. Two competing standards: CycloneDX (OWASP, security-focused) and SPDX (Linux Foundation, license-focused). Generation tools: `cargo-sbom` (Rust), CycloneDX Maven Plugin (Java), `cyclonedx-python` (Python). **SLSA (Supply-chain Levels for Software Artifacts)**: a framework with four levels (L1–L4) of increasing supply chain integrity guarantees. L1 requires build provenance (documenting how an artifact was built). L3 requires a hardened build platform (builds run on an isolated, tamper-proof system). GitHub Actions' build provenance and Sigstore attestations enable SLSA L3 compliance. **Sigstore**: provides free code signing for open source — `cosign` signs container images, `rekor` provides a transparency log (append-only, publicly auditable), and `fulcio` issues short-lived certificates tied to OIDC identities. PyPI's PEP 740 attestations use Sigstore under the hood. Cargo's signing story is still evolving (RFC discussions ongoing). Maven Central has required GPG signing since its inception — the oldest signing requirement of the three. **Hands-on activity**: generate an SBOM for a project in each language using the respective CycloneDX tool — compare the output formats, examine what metadata is captured, and verify that transitive dependencies are included. |
