# Layer 6 · Topic 17 — Performance Characteristics

> How each language's execution model — Rust's AOT compilation via LLVM with zero-cost abstractions, Java's tiered JIT compilation with profile-guided optimization, and Python's CPython interpreter with bytecode dispatch — produces fundamentally different performance profiles across startup time, steady-state throughput, memory footprint, and parallel scalability. Builds on compilation (§3), memory (§5), and concurrency (§12) to reason about performance holistically.

### Coverage Gaps

The owned books **do not** cover:

- **Cross-language benchmarking methodology for equivalent programs** — Gorelick & Ozsvald, Oaks, and Beckwith each discuss benchmarking within a single language, but no owned book provides a systematic methodology for writing equivalent programs in Rust, Java, and Python and comparing their performance fairly while controlling for algorithmic differences, warm-up effects, and measurement methodology; the Computer Language Benchmarks Game (benchmarksgame-team.pages.debian.net) and cross-language benchmark suites are needed
- **Linux `perf` and hardware performance counters for Rust profiling** — Bos Ch.9 discusses benchmarking and Criterion, but no owned book covers Linux `perf stat`, `perf record`, hardware performance counters (cache misses, branch mispredictions, IPC), or integration with `cargo-flamegraph`; the `perf` wiki and Brendan Gregg's performance analysis resources are needed
- **Flamegraph generation and interpretation across all three languages** — no owned book explains flamegraph methodology (Brendan Gregg's invention), how to generate flamegraphs from Rust (`cargo-flamegraph`, `inferno`), Java (`async-profiler`, JFR + JMC), and Python (`py-spy --flame`), or how to interpret them for cache-miss vs CPU-time profiles
- **GraalVM Native Image startup performance vs Rust native binaries** — Beckwith Ch.8 covers GraalVM and CRaC for Java, but no owned book provides a direct comparison of GraalVM Native Image startup characteristics (closed-world assumption, reflection configuration, build-time initialization) versus Rust native binary startup; GraalVM documentation and benchmark blog posts are needed
- **Python 3.13+ experimental JIT compiler (copy-and-patch)** — Shaw's CPython Internals predates the experimental copy-and-patch JIT compiler introduced in CPython 3.13; no owned book covers this or compares it to PyPy's tracing JIT or Java's HotSpot JIT; PEP 744 and the Faster CPython project documentation are needed
- **SIMD vectorization comparison across all three languages** — Matthews Ch.11 covers Rust SIMD briefly, Beckwith Ch.9 covers Java's Vector API, and Gorelick & Ozsvald Ch.6 covers NumPy vectorization, but no single source compares auto-vectorization (Rust via LLVM), explicit SIMD intrinsics (`std::arch`), Java's Vector API (incubating), and NumPy/Numba SIMD code generation side-by-side
- **Continuous profiling and production performance monitoring** — no owned book covers continuous profiling tools like Pyroscope or Datadog Continuous Profiler, or how JFR recordings are used in production environments; the discussion is limited to development-time profiling
- **Memory allocator tuning (jemalloc, mimalloc, tcmalloc)** — no owned book covers replacing the default memory allocator in Rust (`#[global_allocator]` with jemalloc/mimalloc), JVM allocator tuning (TLAB sizing), or Python's `pymalloc` vs system allocator trade-offs; allocator documentation and benchmark comparisons are needed

### Session 1 — Execution Model Fundamentals

**Goal:** understand the fundamental performance characteristics that arise from each language's execution model — Rust's AOT compilation via LLVM with zero-cost abstractions, Java's tiered JIT compilation (interpreter → C1 → C2) with profile-guided optimization, and Python's CPython interpreter loop with bytecode dispatch overhead — and why these models produce radically different steady-state and cold-start performance profiles.

| Step | Time | Activity |
|------|------|----------|
| 1.1 | 30 min | **Rust: AOT compilation and zero-cost abstractions.** Read Klabnik & Nichols Ch.13 pp. 273–294 (iterators, zero-cost abstractions — loops vs iterators compile to identical machine code). Read Blandy & Orendorff Ch.11 pp. 235–263 (traits/generics, static dispatch, monomorphization). Read Gjengset Ch.11 pp. 193–209 (zero-cost abstractions via FFI — Rust calling C has zero overhead beyond the call itself). Key insight: Rust compiles to native machine code via LLVM ahead of time. The "zero-cost abstraction" principle, borrowed from C++, means that high-level constructs (iterators, closures, trait dispatch, generics) compile down to code that is as efficient as hand-written low-level equivalents. Monomorphization eliminates generic overhead at compile time by generating specialized code for each concrete type — `Vec<i32>` and `Vec<String>` become distinct types with distinct optimized code. The LLVM backend applies aggressive optimizations: inlining, loop unrolling, autovectorization, dead code elimination, constant propagation. Klabnik demonstrates this with the iterator vs loop benchmark: Rust's iterator chain `v.iter().filter(\|x\| ...).map(\|x\| ...).sum()` compiles to the same assembly as a manual loop, with zero runtime overhead from the abstraction layer. The cost is paid at compile time (longer builds, larger binaries from monomorphization) rather than at runtime. |
| 1.2 | 35 min | **Java: tiered JIT compilation and hotpath optimization.** Read Beckwith Ch.1 pp. 1–42 (HotSpot VM compilation strategies: interpreter, JIT, tiered compilation, adaptive optimization, deoptimization). Read Oaks Ch.4 pp. 89–120 (JIT compilation: C1/C2, code cache tuning, inlining, escape analysis, deoptimization, GraalVM). Read Evans Ch.7 pp. 207–230 (JIT compilation behavior, inlining, monomorphic calls, deoptimization). Key insight: Java uses a fundamentally different performance strategy — the JVM starts by interpreting bytecode, then progressively compiles hot methods through tiered compilation. Tier 0 is the interpreter. Tiers 1–3 use the C1 (client) compiler with increasing levels of profiling instrumentation. Tier 4 uses the C2 (server) compiler for maximum optimization, guided by the profiling data collected in earlier tiers. This profile-guided optimization (PGO) enables optimizations that AOT compilers cannot easily perform: speculative inlining based on observed call targets (if a virtual call always dispatches to one implementation, inline it directly), uncommon trap deoptimization (optimize for the common case, bail out if assumptions are violated), and escape analysis (if an object never escapes the method, allocate it on the stack instead of the heap). The trade-off: Java pays a warmup cost (thousands of invocations before a method reaches peak performance) but can potentially match or exceed AOT performance for long-running applications because it optimizes based on actual runtime behavior rather than static analysis. |
| 1.3 | 30 min | **Python: interpreter overhead and the evaluation loop.** Read Shaw Ch.5 pp. 151–175 (the evaluation loop: thread state, frame objects, frame execution, the value stack — the heart of CPython that determines baseline interpreter performance). Read Gorelick & Ozsvald Ch.1 pp. 1–20 (understanding performant Python: fundamental computer system, CPU/memory/cache hierarchy, how Python's abstractions layer over hardware creating overhead). Key insight: CPython is the slowest execution model of the three by a significant margin. Source code is compiled to bytecode (.pyc files), which is then interpreted by the CPython evaluation loop (`ceval.c`). Each bytecode instruction requires: fetching the opcode, dispatching through a giant `switch` statement (or computed goto on supported platforms), manipulating the value stack, and updating the instruction pointer. This per-instruction overhead means that even a simple integer addition involves multiple C function calls, type checks, and memory allocations (every integer is a heap-allocated `PyObject`). The dynamic typing tax is severe: the interpreter cannot know at compile time that `a + b` means integer addition, so it must check types at runtime for every operation. CPython 3.11 introduced the specializing adaptive interpreter, which caches type information for hot bytecodes (e.g., `BINARY_OP` specializes to `BINARY_OP_ADD_INT` after seeing integer operands), reducing dispatch overhead by 10–25%. However, even specialized CPython is typically 10–100x slower than equivalent Rust or Java for CPU-bound computation. |
| 1.4 | 25 min | **Cross-language comparison: the performance spectrum.** Read Matthews Ch.11 pp. 219–222 (zero-cost abstractions overview). Read Oaks Ch.1 pp. 1–13 (performance optimization philosophy — algorithms, code volume, common case). Key insight: the three execution models occupy distinct positions on the performance spectrum. Rust (AOT native) achieves peak performance from the first instruction but cannot adapt to runtime behavior — all optimization decisions are made at compile time based on static analysis. Java (JIT) starts slow (interpreter) but reaches competitive peak performance after warmup, with the unique advantage of runtime profile-guided optimization — it can deoptimize and reoptimize as program behavior changes. Python (interpreted) has the slowest baseline but compensates through a rich ecosystem of C extensions (NumPy, Polars), alternative runtimes (PyPy tracing JIT achieves 5–10x speedup over CPython), and compilation tools (Cython, Numba) that selectively accelerate hot code. The practical implication: Rust is fastest for all workloads. Java is competitive for long-running server applications (after warmup). Python is competitive for workloads dominated by C-extension computation (data science, ML) but slow for pure Python loops and computation. The choice of language rarely determines application performance for most real-world systems — algorithm selection, I/O patterns, and architecture matter more than the execution model. |

### Session 2 — Benchmarking Methodology

**Goal:** learn rigorous benchmarking methodology for each language — Criterion for Rust, JMH for Java, timeit/pytest-benchmark/pyperf for Python — understanding the pitfalls of microbenchmarking (dead code elimination, constant folding, JIT warmup artifacts, GC pauses) and when macrobenchmarks provide more actionable data.

| Step | Time | Activity |
|------|------|----------|
| 2.1 | 30 min | **Rust: Criterion and `cargo bench`.** Read Bos Ch.9 pp. 181–211 (optimization techniques, benchmarking — includes practical benchmarking of lock implementations). Read Matthews Ch.11 pp. 219–231 (optimizations, including measurement considerations). Key insight: Rust benchmarking faces the challenge that LLVM is extremely aggressive at eliminating "useless" computation. If a benchmark computes a value but never uses it, LLVM will simply eliminate the computation entirely, producing misleadingly fast results. `std::hint::black_box()` prevents this by telling the compiler the value is used (opaque identity function). Criterion.rs is the standard benchmarking framework: it runs benchmarks multiple times, performs statistical analysis (confidence intervals, regression detection), and produces HTML reports with comparison against previous runs. Criterion handles warmup, outlier detection, and significance testing automatically. Unlike Java, Rust has no JIT warmup concern — the first run is representative of steady-state performance. The main measurement pitfalls in Rust: (1) dead code elimination without `black_box`, (2) constant folding if inputs are compile-time constants, (3) CPU frequency scaling (turbo boost) affecting reproducibility, (4) memory allocator variability. `#[bench]` in nightly Rust is simpler but less statistically rigorous than Criterion. |
| 2.2 | 35 min | **Java: JMH and the science of JVM benchmarking.** Read Beckwith Ch.5 pp. 115–175 (end-to-end performance optimization, JMH benchmarking, performance methodology, perfasm profiling). Read Oaks Ch.2 pp. 15–48 (performance testing methodology: micro/macro/mesobenchmarks, throughput vs response time, JMH). Read Goetz Ch.12 pp. 245–276 (testing concurrent programs: performance testing, avoiding pitfalls). Key insight: JVM benchmarking is notoriously difficult because the JIT compiler actively undermines naive benchmarks. JMH (Java Microbenchmark Harness), developed by the OpenJDK team, is the only reliable way to benchmark JVM code. JMH addresses multiple hazards: (1) warmup iterations — JMH runs configurable warmup iterations so the JIT compiler reaches steady state before measurement begins; (2) dead code elimination — JMH's `Blackhole.consume()` prevents the JIT from eliminating benchmark computation; (3) constant folding — JMH's `@State` annotation ensures inputs are loaded from non-constant fields; (4) loop optimization — JMH prevents the JIT from hoisting invariant computation out of benchmark loops; (5) fork isolation — JMH forks a new JVM per benchmark to prevent profile pollution between benchmarks (profiling data from one benchmark should not influence JIT decisions for another). Goetz Ch.12 provides the complementary perspective: how to test the performance of concurrent data structures, avoiding the pitfalls of timing code that contains synchronization (contention varies with thread count, CPU architecture, and JVM version). Oaks emphasizes that microbenchmarks should be interpreted with extreme caution — application-level benchmarks with realistic workloads are far more informative. |
| 2.3 | 25 min | **Python: timeit, pytest-benchmark, and pyperf.** Read Shaw Ch.11 (benchmarking, profiling, timeit, cProfile, perf counters). Read Slatkin Items 92–93 pp. 448–457 (profile before optimizing, optimize with timeit microbenchmarks). Read Martelli Ch.17 (testing/optimization, profiling). Key insight: Python benchmarking is comparatively straightforward because CPython's interpreter has no JIT (in standard builds) — there is no warmup concern, and the interpreter does not aggressively optimize away dead code. `timeit` is the standard module: it runs a statement many times and reports the minimum time, automatically disabling garbage collection during measurement. `pytest-benchmark` integrates benchmarking into the pytest workflow with statistical analysis. `pyperf` (from the PSF) provides the most rigorous benchmarking: it spawns separate processes for each benchmark, tunes the system for reproducibility (CPU affinity, frequency pinning), detects calibration issues, and performs statistical significance testing. The main Python benchmarking pitfalls: (1) GC pauses — `timeit` disables the cycle collector by default, but this may not represent real workload performance; (2) import overhead — module imports can dominate benchmark time for small benchmarks; (3) warm cache vs cold cache — CPython caches compiled bytecode (.pyc files), but first-import overhead is real; (4) measuring I/O-bound vs CPU-bound — Python's performance characteristics differ dramatically between these two modes. |
| 2.4 | 20 min | **Cross-language comparison: microbenchmarks vs macrobenchmarks.** Key insight: microbenchmarks measure isolated operations (sorting an array, hashing a string, iterating a collection) while macrobenchmarks measure end-to-end application performance (HTTP request latency, batch processing throughput, memory consumption under load). Each language has different microbenchmark-to-macrobenchmark translation fidelity. Rust: microbenchmarks translate well to macrobenchmarks because there is no runtime that changes behavior at scale (no GC, no JIT). Java: microbenchmarks frequently mislead because JIT optimizations in isolation (escape analysis eliminating allocations) may not occur in the full application (objects escape to callers), and GC behavior at scale differs from small benchmarks. Python: microbenchmarks understate Python's disadvantage because real applications compound interpreter overhead across millions of bytecode dispatches, and the GIL prevents thread-based scaling. The Computer Language Benchmarks Game provides a useful (if imperfect) cross-language comparison framework — each program is algorithmically equivalent, implementations are community-optimized, and measurements include both CPU time and memory consumption. However, even this benchmark suite favors languages that allow low-level optimization (Rust excels, Python programs often use C extensions like NumPy). |
| 2.5 | 15 min | **Establishing a personal benchmarking practice.** Key insight: construct a personal set of equivalent benchmark programs across all three languages: (1) a CPU-bound task (e.g., matrix multiplication, prime sieve, mandelbrot), (2) an I/O-bound task (e.g., HTTP request fan-out, file processing), (3) a memory-intensive task (e.g., building a large hash map, graph traversal). For Rust, use Criterion with `--bench` and `black_box`. For Java, use JMH with `@Warmup(iterations = 5)`, `@Measurement(iterations = 10)`, and `@Fork(3)`. For Python, use `pyperf` with process isolation. Always report: wall-clock time, RSS memory, and CPU utilization. Always test on the same hardware with consistent system configuration (CPU governor set to `performance`, background processes minimized). |

### Session 3 — Profiling Tools and Techniques

**Goal:** master the profiling tool ecosystem for each language — `perf`/`cargo-flamegraph`/DHAT for Rust, async-profiler/JFR/Java Mission Control for Java, cProfile/py-spy/line_profiler/memory_profiler/Scalene for Python — understanding the difference between sampling vs instrumentation profilers, CPU vs allocation profiling, and how to interpret flamegraphs.

| Step | Time | Activity |
|------|------|----------|
| 3.1 | 35 min | **Rust: perf, flamegraphs, and DHAT.** Read Bos Ch.9 pp. 181–211 (optimization techniques, benchmarking — includes profiling in the context of building locks). Key insight: Rust profiling leverages Linux system tools directly because Rust compiles to native code with standard DWARF debug information. `perf record -g ./target/release/myapp` samples the call stack at regular intervals (typically 99Hz to avoid aliasing with timer interrupts); `perf report` shows the sampled call stacks as a tree. `cargo-flamegraph` wraps `perf` and produces SVG flamegraphs — interactive visualizations where the x-axis is the percentage of samples and the y-axis is the call stack depth. | 3.2 | 35 min | **Java: async-profiler, JFR, and Java Mission Control.** Read Oaks Ch.3 pp. 49–88 (Java performance toolbox: OS tools, JVM monitoring, profiling approaches — sampling vs instrumented, JFR and Java Mission Control). Key insight: Java profiling is uniquely rich because the JVM provides built-in profiling infrastructure. The safepoint bias problem: the JVM can only walk thread stacks at "safepoints" (GC-safe points in generated code). Naive profilers that sample at safepoints miss CPU time spent between safepoints, producing biased results. async-profiler avoids this by using OS-level signals. Additionally, JFR can profile allocations (`jdk.ObjectAllocationInNewTLAB`, `jdk.ObjectAllocationOutsideTLAB` events) to find allocation-heavy code paths without instrumenting every allocation. |
| 3.3 | 30 min | **Python: cProfile, py-spy, line_profiler, memory_profiler, and Scalene.** Read Gorelick & Ozsvald Ch.2 pp. 21–64 (profiling to find bottlenecks: Julia Set benchmark, cProfile/SnakeViz, line_profiler, memory_profiler, PySpy, bytecode analysis with dis). Read Slatkin Item 92 pp. 448–452 (profile before optimizing with cProfile). Key insight: Python has a rich profiling ecosystem because performance optimization is frequently needed. `cProfile` is the built-in deterministic profiler: it instruments every function call and return, recording call count, total time, and cumulative time. `python -m cProfile -s cumulative script.py` sorts by cumulative time. `cProfile` adds 30–50% overhead but provides exact call counts. `py-spy` is a sampling profiler that attaches to a running Python process without modifying it — it reads the process memory to extract Python call stacks, producing flamegraphs with minimal overhead (~5%). `py-spy` is invaluable for profiling production systems where you cannot restart with instrumentation. `line_profiler` (`@profile` decorator + `kernprof`) provides line-by-line timing within a function — essential for identifying which specific line is slow in a hot function. `memory_profiler` shows line-by-line memory consumption changes. Scalene is the newest and most comprehensive: it simultaneously profiles CPU time (distinguishing Python vs C code), GPU utilization, and memory (allocation rate, copy volume) without requiring code modification. Scalene separates "Python time" from "C extension time," which is critical for understanding whether optimization should target Python code or the interface to C libraries. |
| 3.4 | 20 min | **Cross-language comparison: profiling philosophy.** Key insight: each language's profiling ecosystem reflects its performance model. Rust: profiling uses OS-level tools (perf, flamegraph) because Rust compiles to native code — there is no runtime to introspect, so the OS provides the profiling infrastructure. Java: profiling uses JVM-provided infrastructure (JFR, JMX, `AsyncGetCallTrace`) because the JVM has deep visibility into execution — it knows which methods are JIT-compiled, where GC pauses occur, and how threads interact. Python: profiling uses Python-level tools (cProfile, line_profiler) because the interpreter is the bottleneck — profiling the C runtime of CPython (via perf) would show the interpreter loop, not the Python functions. The key difference in what profilers reveal: Rust flamegraphs show your actual code (hot loops, algorithm choices). Java flamegraphs show a mix of your code and JVM internals (GC, JIT compilation, class loading). Python flamegraphs show the interpreter overhead alongside your code (CPython evaluation loop). This is why Python profilers like Scalene distinguish "Python time" from "C time" — in data science workloads, most CPU time is in C extensions (NumPy, pandas), and the Python interpreter overhead is negligible. |

### Session 4 — Memory Footprint and Layout

**Goal:** understand how memory layout and object representation differ across the three languages — Rust's compact struct layout with predictable alignment, Java's JVM object headers and compressed oops overhead, Python's PyObject headers and per-object reference counting cost — and compare the memory footprint of equivalent programs.

| Step | Time | Activity |
|------|------|----------|
| 4.1 | 30 min | **Rust: struct layout, alignment, and cache efficiency.** Read Gjengset Ch.2 pp. 19–35 (types in memory: alignment, layout, complex types, DSTs). Read McNamara Ch.5 pp. 137–173 (data in depth: bit patterns and types, endianness, floating-point internals). Read Bos Ch.7 pp. 127–159 (processor architecture, cache coherence with MESI protocol, impact on performance). Key insight: Rust gives programmers explicit control over memory layout. By default, the Rust compiler is free to reorder struct fields to minimize padding (unlike C, which must preserve field order). `#[repr(C)]` forces C-compatible layout for FFI. `#[repr(packed)]` eliminates padding entirely (at the cost of unaligned access). `std::mem::size_of::<T>()` and `std::mem::align_of::<T>()` reveal the layout. A `u8` followed by a `u64` followed by a `u8` in a struct requires 24 bytes with C layout (padding to align the u64), but the Rust compiler may reorder to `u64, u8, u8` requiring only 16 bytes. Bos Ch.7 explains why layout matters for performance: CPU caches operate on cache lines (typically 64 bytes). Data that is accessed together should be adjacent in memory (spatial locality). False sharing occurs when two threads modify data on the same cache line, causing the cache line to ping-pong between CPU cores via the MESI cache coherence protocol. Rust's `#[repr(align(64))]` can pad structs to cache line boundaries to prevent false sharing. Enums use tagged unions: the tag (discriminant) plus the largest variant determine the size, with niche optimization eliminating the tag when possible (`Option<&T>` is the same size as `&T` because the null pointer value serves as the `None` discriminant). |
| 4.2 | 30 min | **Java: object headers, compressed oops, and Valhalla.** Read Oaks Ch.7 pp. 203–248 (heap memory best practices: heap analysis, reducing object size, lazy initialization, soft/weak references, compressed oops). Read Beckwith Ch.6 pp. 177–217 (advanced memory management: TLABs/PLABs, regionalized heap). Read Bloch Item 6 (avoid creating unnecessary objects). Key insight: every Java object on the heap carries a 12-byte header (with compressed oops enabled, which is the default for heaps under 32GB): 8 bytes for the "mark word" (hash code, lock state, GC age) and 4 bytes for the class pointer (compressed from 8 bytes). Arrays add a 4-byte length field (16 bytes total header). This means an `Integer` object (wrapping a 4-byte `int`) occupies 16 bytes — 4x the raw data size. An `int[]` of 1000 elements occupies 4016 bytes (4000 data + 16 header), while an `ArrayList<Integer>` of 1000 elements occupies approximately 20,000 bytes (16 header for ArrayList + 4016 for backing `Object[]` + 1000 × 16 for boxed Integer objects). This boxing overhead is why Java performance guides emphasize primitive arrays and `IntStream` over `Stream<Integer>`. Project Valhalla (JEP 401, in progress) aims to introduce value types that eliminate object headers for small, identity-free types — `Complex(double re, double im)` would be 16 bytes instead of 32 bytes, and arrays of value types would be flattened (contiguous memory, no pointers). Tools: JOL (Java Object Layout, by Aleksey Shipilev) reveals the exact memory layout of any Java object: `ClassLayout.parseInstance(obj).toPrintable()`. |
| 4.3 | 25 min | **Python: PyObject overhead and reference counting cost.** Read Shaw Ch.7 pp. 177–219 (memory management: CPython allocator design, reference counting, garbage collection, memory pools/arenas). Read Gorelick & Ozsvald Ch.11 pp. 341–390 (using less RAM: Python object memory overhead, array module, NumPy optimization, probabilistic data structures). Key insight: CPython has the highest per-object memory overhead of the three languages. Every Python object starts with a `PyObject` header: 8 bytes for the reference count (`Py_ssize_t ob_refcnt`) and 8 bytes for the type pointer (`PyTypeObject *ob_type`) — 16 bytes minimum. Variable-size objects (`PyVarObject`) add 8 bytes for the size field. A Python `int` (arbitrary precision) occupies 28 bytes minimum (PyObject header + size + one 30-bit digit). A Python `float` occupies 24 bytes (PyObject header + 8-byte double). A Python `list` of 1000 integers occupies approximately 36,000 bytes (list header + pointer array + 1000 boxed int objects). Compare with Rust: a `Vec<i64>` of 1000 elements occupies approximately 8,024 bytes (24-byte Vec header + 8,000 bytes of contiguous data). That is a 4.5× memory overhead in Python. `__slots__` eliminates the per-instance `__dict__` (which is itself a hash table), reducing object size significantly for classes with fixed attributes. `sys.getsizeof()` returns the shallow size of an object; `pympler.asizeof()` returns the deep size including all referenced objects. For large datasets, `array.array` (typed arrays, stored as C arrays) and NumPy arrays (contiguous typed memory) bypass the per-element PyObject overhead entirely. |
| 4.4 | 25 min | **Cross-language comparison: memory footprint of equivalent programs.** Read Gorelick & Ozsvald Ch.3–4 pp. 65–95 (data structure performance — lists, dicts, sets). Read Slatkin Item 99 pp. 485–492 (memoryview and bytearray for zero-copy). Key insight: construct the mental model for memory comparison. For an array of 1 million 64-bit integers: Rust `Vec<i64>` occupies approximately 8 MB (contiguous, zero overhead). Java `long[]` occupies approximately 8 MB (contiguous, 16-byte header negligible). Java `ArrayList<Long>` occupies approximately 24 MB (boxed Long objects + pointer array). Python `list` of `int` occupies approximately 36 MB (boxed int objects + pointer array). NumPy `np.array(dtype=np.int64)` occupies approximately 8 MB (contiguous, C-level storage). This comparison reveals the fundamental trade-off: Rust and Java (with primitives) achieve C-like memory density. Java with objects and Python with standard types pay a 3–4.5× memory tax. Python with NumPy recovers C-like density for numerical data. The memory tax compounds: a hash map of 1 million key-value pairs has even larger ratios because each entry requires boxed key, boxed value, hash, and pointer overhead. Memory footprint directly affects performance through cache utilization: 8 MB of contiguous data fits in L3 cache on modern CPUs; 36 MB does not, causing frequent cache misses and L3-to-RAM round trips (~100ns per miss vs ~4ns for L1 hit). |

### Session 5 — Startup Time and Warmup

**Goal:** understand the dramatically different startup characteristics of each language — Rust's near-instant native binary startup, Java's multi-phase warmup from interpreter through JIT tiers (and mitigation strategies: CDS, AOT, GraalVM Native Image, CRaC), and Python's interpreter startup with import overhead — and when startup time matters vs steady-state throughput.

| Step | Time | Activity |
|------|------|----------|
| 5.1 | 25 min | **Rust: instant startup.** Read Matthews Ch.11 pp. 219–231 (optimizations overview — binary characteristics). Key insight: Rust native binaries start in microseconds. The OS loads the binary into memory, the dynamic linker resolves shared library symbols (if any — Rust defaults to static linking for most dependencies), and `main()` executes. There is no runtime initialization, no interpreter startup, no JIT warmup. A "hello world" Rust binary starts in approximately 1–3 milliseconds. This makes Rust ideal for CLI tools (ripgrep starts instantly, unlike Java CLI tools that take 100–500ms), serverless functions (AWS Lambda cold starts), and short-lived processes. The only startup cost is dynamic linking (if used) and any explicit initialization code (lazy_static, once_cell). Static linking (`-C target-feature=+crt-static` or musl target) eliminates even dynamic linking overhead. Binary size affects startup through page loading: a 1MB binary loads faster than a 100MB binary because fewer pages must be faulted in from disk. `strip` removes debug symbols, and LTO (link-time optimization) reduces binary size. Compare with Java: a trivially simple Rust program is ready to do useful work before the JVM has even finished loading the class file containing `main`. |
| 5.2 | 40 min | **Java: JVM warmup and mitigation strategies.** Read Beckwith Ch.8 pp. 273–306 (accelerating time to steady state: JVM start-up phases, CDS, GraalVM native image, CRIU/CRaC, serverless optimization, Project Leyden). Read Oaks Ch.4 pp. 89–95 (JIT compilation tiers — understanding warmup). Read Evans Ch.7 pp. 215–225 (JIT compilation overview). Key insight: JVM startup is a multi-phase process. First, the JVM itself initializes (loading core classes from the system image, initializing the GC, JIT compiler threads, thread management). This takes approximately 50–100ms for a bare `java -version`. Then the application's classes are loaded, verified, and linked. Then the code runs in the interpreter (Tier 0), which is 5–10× slower than compiled code. As methods become "hot" (default threshold: ~10,000 invocations for C2), they are compiled in the background by JIT compiler threads. Full warmup to Tier 4 (C2-compiled) can take seconds to minutes depending on the application. For a typical Spring Boot web application, full warmup (all critical paths C2-compiled, all caches populated) takes 30–60 seconds. Mitigation strategies: (1) Class Data Sharing (CDS/AppCDS) — serialize loaded classes to a shared archive, reducing class loading time by 10–30%. (2) GraalVM Native Image — AOT compile the entire application to a native binary, achieving Rust-like startup (10–50ms), but with restrictions (closed-world assumption, limited reflection, no runtime class loading). (3) CRaC (Coordinated Restore at Checkpoint) — snapshot a running, fully warmed-up JVM and restore it later, achieving near-instant startup with full JIT optimization already applied. (4) Project Leyden (in progress) — shift JIT work to before runtime using "condensed" JVM images. |
| 5.3 | 25 min | **Python: interpreter startup and import overhead.** Read Slatkin Items 97–98 pp. 475–484 (bytecode caching for startup, lazy-load modules with dynamic imports). Read Gorelick & Ozsvald Ch.1 pp. 1–10 (understanding performant Python). Key insight: CPython startup is faster than JVM startup but slower than native binaries. `python -c "pass"` takes approximately 30–50ms on modern hardware. The startup cost comes from: (1) initializing the interpreter state (creating built-in modules, setting up the import machinery, initializing the GC), (2) importing the `site` module (which scans for `.pth` files and adds site-packages), and (3) importing user modules. Import time is the dominant startup cost for most Python applications: each `import` statement triggers file system lookups, bytecode compilation (if .pyc is missing or stale), and module initialization (executing module-level code). `python -X importtime` reveals the import tree and time for each module. Django startup imports hundreds of modules, taking 500ms–2s. Mitigation strategies: (1) `-S` flag skips site module import. (2) `.pyc` bytecode caching avoids recompilation (`compileall` precompiles). (3) PEP 690 proposes lazy imports (import modules only when first accessed, not at import statement). (4) `__all__` and careful import structure avoid importing unused submodules. (5) For serverless/CLI: `zipimport` from a compressed archive reduces file system lookups. Unlike Java, Python has no JIT warmup — performance is the same from the first bytecode instruction to the last (in standard CPython). |
| 5.4 | 20 min | **Cross-language comparison: when startup matters.** Key insight: startup time matters for: CLI tools (users expect instant response), serverless functions (cold start latency directly affects P99), microservices with frequent restarts (rolling deployments, autoscaling), and short-lived batch processes (if the process runs for 100ms, a 500ms startup is 5× overhead). Startup time does NOT matter for: long-running servers (amortized over hours of uptime), batch processing jobs (amortized over minutes of computation), and interactive applications (user waits for UI to load, not for language runtime). The startup spectrum: Rust 1–5ms (native binary) < Python 30–50ms (interpreter) < Java 100–500ms (JVM) < Java with Spring 2–10s (framework initialization) < Python with Django 0.5–2s (import overhead). GraalVM Native Image transforms Java from the slowest to competitive with Rust (10–50ms), but at the cost of AOT compilation constraints (no dynamic class loading, limited reflection, closed-world assumption). CRaC achieves the best of both worlds for Java (instant startup WITH full JIT optimization) but requires checkpoint/restore infrastructure. For Python, the key insight is that startup overhead is dominated by imports, not by the interpreter itself — a well-structured Python application with lazy imports can start in 50–100ms. |

### Session 6 — Data Structure and Algorithm Performance

**Goal:** compare the performance characteristics of standard collection types and iteration patterns across all three languages — understanding how data layout (contiguous arrays vs pointer-heavy structures), iterator optimization (zero-cost vs stream fusion vs generator overhead), cache effects, and SIMD vectorization create orders-of-magnitude performance differences for the same algorithmic work.

| Step | Time | Activity |
|------|------|----------|
| 6.1 | 30 min | **Rust: collection performance and iterator optimization.** Read Matthews Ch.11 pp. 222–231 (vector optimization, SIMD, fast copies with slices). Read Klabnik & Nichols Ch.13 pp. 273–294 (iterators, zero-cost abstractions — loops vs iterators benchmark). Read Blandy & Orendorff Ch.11 pp. 235–263 (static dispatch, monomorphization). Key insight: Rust collections achieve near-optimal performance because data is stored contiguously in memory (no boxing, no indirection for primitive types). A `Vec<i32>` is a contiguous block of 4-byte integers — CPU prefetchers predict sequential access patterns and load cache lines ahead of time. Rust's iterator chains compile to the same machine code as hand-written loops through monomorphization and inlining: `vec.iter().filter(\|x\| **x > 0).map(\|x\| x * 2).sum::<i32>()` becomes a single loop with no function call overhead, no allocation, and no dynamic dispatch. LLVM autovectorizes these loops when possible, processing 4–8 integers per CPU instruction using SIMD. `HashMap` in Rust's standard library uses the SwissTable algorithm (via `hashbrown`), which is designed for cache efficiency: metadata bytes for each slot fit in a single SIMD register, enabling 16-way parallel probing. Compare with Java: Java's `HashMap` uses linked-list chaining (or tree buckets for long chains), requiring pointer traversal and poor cache behavior. Compare with Python: Python's `dict` is highly optimized (compact dict since 3.6) but each key and value is a boxed `PyObject` with pointer indirection. |
| 6.2 | 30 min | **Java: Stream API optimization and escape analysis.** Read Oaks Ch.4 pp. 95–120 (JIT optimizations: inlining, escape analysis, CPU-specific code). Read Bloch Items 48, 61, 63 (streams, boxing, string concat). Read Beckwith Ch.7 pp. 219–270 (string interning, compact strings, lock optimization). Key insight: Java's Stream API introduces an abstraction layer over collections. `list.stream().filter(x -> x > 0).map(x -> x * 2).reduce(0, Integer::sum)` creates intermediate `Stream` objects and lambda invocations. The C2 JIT compiler optimizes this through: (1) inlining — lambda bodies are inlined into the stream pipeline loop; (2) escape analysis — intermediate `Stream` objects that do not escape the method are stack-allocated or eliminated entirely; (3) loop fusion — chained operations are fused into a single pass. However, boxing remains a significant overhead: `Stream<Integer>` boxes and unboxes at each step. `IntStream`, `LongStream`, and `DoubleStream` avoid boxing for primitive types and should always be preferred for numerical work. Bloch Item 48 warns against overusing parallel streams — the overhead of splitting, thread handoff, and combining results only pays off for large datasets with expensive per-element operations. String performance is a Java-specific concern: compact strings (Java 9+) store Latin-1 strings as byte arrays instead of char arrays, halving memory usage. String concatenation with `+` in loops creates O(n²) behavior; `StringBuilder` or `String.join()` should be used instead (Bloch Item 63). |
| 6.3 | 25 min | **Python: data structure performance and NumPy vectorization.** Read Gorelick & Ozsvald Ch.3–4 pp. 65–95 (lists, tuples, dicts, sets — data structure performance internals). Read Gorelick & Ozsvald Ch.5 pp. 97–107 (iterators/generators for memory efficiency). Read Gorelick & Ozsvald Ch.6 pp. 109–160 (NumPy vectorization, SIMD, numexpr). Read Ramalho Ch.2 pp. 21–76 (sequence performance: slicing, comprehensions, arrays, deques). Key insight: Python's built-in data structures are implemented in C and are highly optimized within the constraints of the object model. `list.append()` uses amortized O(1) over-allocation. `dict` uses open addressing with a compact layout (since Python 3.6, dicts maintain insertion order). However, every element in a `list` or `dict` is a pointer to a boxed `PyObject`, causing pointer chasing and cache misses. For numerical work, NumPy eliminates this overhead: `np.array([1, 2, 3], dtype=np.int64)` stores values contiguously without boxing. Vectorized operations (`a + b`, `np.sum(a)`) execute in compiled C/Fortran code with SIMD instructions, operating on contiguous memory. A NumPy sum of 1 million integers is approximately 100× faster than `sum(python_list)` because it avoids: interpreter dispatch, type checking, boxing/unboxing, and pointer chasing. Generators (`yield`) provide memory efficiency by producing values lazily — iterating over `range(1_000_000)` uses O(1) memory instead of O(n) — but each `yield` involves interpreter overhead (suspending/resuming the generator frame). |
| 6.4 | 20 min | **SIMD and vectorization comparison.** Read Matthews Ch.11 pp. 227–231 (SIMD in Rust). Read Beckwith Ch.9 pp. 307–336 (Vector API, GPU, SIMD in Java). Key insight: SIMD (Single Instruction, Multiple Data) processes multiple data elements with a single instruction — 4 floats at once with SSE, 8 with AVX2, 16 with AVX-512. Rust: LLVM autovectorization works on tight loops over contiguous data. Explicit SIMD via `std::arch` (e.g., `_mm256_add_ps` for AVX2). The `std::simd` (nightly) provides portable SIMD abstractions. Java: the Vector API (incubating since Java 16, JEP 448) provides explicit SIMD: `IntVector.fromArray(SPECIES_256, array, 0).add(other).intoArray(result, 0)`. Before the Vector API, the C2 JIT compiler performed limited autovectorization for simple loops. Python: NumPy's ufuncs (universal functions) use SIMD internally. Numba's `@njit` compiles Python functions to LLVM IR with autovectorization. Direct SIMD is not accessible from pure Python. The key insight: Rust has the most control (explicit intrinsics + reliable autovectorization), Java is catching up (Vector API provides portable abstractions), Python delegates entirely to C extensions (NumPy, Numba). For data-parallel workloads, all three languages can achieve similar SIMD-level performance — the difference is where the SIMD code lives (Rust: in your code, Java: in JIT-compiled code, Python: in C extensions). |

### Session 7 — Common Optimization Patterns

**Goal:** learn the language-specific optimization patterns that experienced developers use in each ecosystem — inlining and monomorphization in Rust, escape analysis and string interning in Java, Cython/Numba/C extensions in Python — understanding when each pattern applies and the expected performance improvement.

| Step | Time | Activity |
|------|------|----------|
| 7.1 | 30 min | **Rust: inlining, monomorphization, and allocation reduction.** Read Gjengset Ch.11 pp. 193–209 (zero-cost abstractions via FFI — demonstrates the principle in cross-language context). Read Blandy & Orendorff Ch.11 pp. 235–263 (static dispatch, monomorphization — how generics compile to specialized code). Read Klabnik & Nichols Ch.4 pp. 59–83 (ownership, zero-copy — avoiding allocation through borrowing). Key insight: Rust optimization follows a predictable pattern. First, ensure zero-cost abstractions are working: use iterators (not index-based loops), use generics (not trait objects) for hot paths, prefer `&str` over `String` (avoid allocation). Second, control inlining: `#[inline]` hints the compiler to inline across crate boundaries (within a crate, LLVM decides automatically); `#[inline(always)]` forces inlining; `#[inline(never)]` prevents it (useful for cold error paths to keep hot code compact). Third, reduce allocations: use `&[T]` instead of `Vec<T>` for read-only access, use `Cow<'_, str>` to avoid cloning when the borrowed case is common, use `SmallVec` for vectors that are usually small (avoids heap allocation). Fourth, enable LTO (Link-Time Optimization) in release builds (`lto = true` in Cargo.toml) for cross-crate inlining and dead code elimination. Fifth, target the specific CPU: `-C target-cpu=native` enables CPU-specific instructions (AVX2, etc.). Monomorphization is both an optimization and a potential problem: excessive monomorphization bloats binary size and instruction cache, hurting performance. `dyn Trait` (dynamic dispatch via vtable) trades per-call overhead (~2ns for indirect call) for reduced code size — use it for cold paths with many type variants. |
| 7.2 | 35 min | **Java: escape analysis, string optimization, and lock elision.** Read Oaks Ch.4 pp. 100–120 (escape analysis, inlining, on-stack replacement, deoptimization). Read Beckwith Ch.7 pp. 219–270 (string interning, deduplication, compact strings, lock types in HotSpot, contention optimization). Read Bloch Items 6, 61, 63 (avoid unnecessary objects, prefer primitives, avoid string concat in loops). Key insight: Java's C2 JIT compiler performs aggressive optimizations that are not possible at compile time. Escape analysis determines whether an object escapes its creating method: if not, it can be (1) stack-allocated (avoiding GC entirely), (2) scalar-replaced (broken into individual fields stored in registers), or (3) lock-elided (synchronization removed if the object is thread-local). For example, `synchronized(new Object()) { ... }` — the lock is eliminated because the Object never escapes. String optimization in Java is a performance-critical pattern: compact strings (Latin-1 encoding for ASCII strings) halve memory usage. String interning (`String.intern()`) deduplicates strings using a pool, saving memory for repeated string values. String deduplication with G1 GC (`-XX:+UseStringDeduplication`) automatically interns strings that have identical `char[]` backing arrays. Lock optimization: biased locking (now deprecated in recent JDKs), lock coarsening (merging adjacent synchronized blocks on the same monitor), and lock elision (removing locks on thread-local objects). Bloch Item 6 advises against unnecessary autoboxing: `Long sum = 0L; for (long i = 0; i < n; i++) sum += i;` creates approximately `n` Long objects (though modern JVMs may optimize this via escape analysis, the explicit `long sum` avoids the risk entirely). |
| 7.3 | 30 min | **Python: Cython, Numba, and C extension optimization.** Read Gorelick & Ozsvald Ch.7 pp. 161–211 (compiling to C: Cython, Numba, PyPy, GPU computing, FFI). Read Slatkin Items 94–96 pp. 458–474 (when to replace Python with C/Rust, ctypes for FFI, extension modules). Key insight: Python's primary optimization strategy is to move hot code out of the interpreter entirely. Cython: a Python superset that compiles to C. Adding type annotations to Cython code (`cdef int x = 0`) allows the Cython compiler to generate tight C loops without Python object overhead. A typed Cython loop can be 100–1000× faster than the equivalent pure Python. Numba: a JIT compiler for numerical Python. The `@numba.njit` decorator compiles a Python function to machine code via LLVM on first call. Numba supports NumPy arrays, loops, and mathematical operations. It cannot compile arbitrary Python (no dict/list/class support in `nopython` mode). A Numba-compiled function achieves C-like performance for numerical kernels. C extensions via PyO3 (Rust) or CPython C API: implement hot functions in Rust or C, callable from Python. PyO3 is increasingly popular because it provides memory safety (Rust) with Python interoperability. `ctypes`/`cffi`: call existing C libraries from Python without writing extension code. `memoryview`: access the buffer protocol for zero-copy data access — avoids copying data when passing arrays between Python and C extensions. The optimization decision tree: (1) profile to find the bottleneck, (2) try NumPy vectorization (easiest), (3) try Numba `@njit` (moderate effort, great for numerical loops), (4) try Cython (moderate effort, works for general Python code), (5) write a C/Rust extension (most effort, maximum performance). |
| 7.4 | 20 min | **Cross-language comparison: optimization philosophy.** Key insight: each language has a fundamentally different optimization philosophy. Rust: optimization is about removing abstractions that the compiler cannot see through. The compiler does most of the work — the programmer's job is to write idiomatic code and let monomorphization, inlining, and LLVM optimization handle performance. When that is insufficient, add explicit SIMD, control memory layout, or adjust allocator strategy. Java: optimization is about working with the JIT compiler, not against it. Write simple, readable code that the JIT can optimize (avoid overly complex inheritance hierarchies that defeat inlining, avoid reflection in hot paths that defeats escape analysis). When the JIT is insufficient, use primitive types, reduce GC pressure, and consider off-heap data structures. Python: optimization is about escaping the interpreter for hot code. Profile to find the 1–5% of code that consumes 90% of the time, then move that code to NumPy/Numba/Cython/C/Rust. The remaining 95% of Python code should prioritize readability and developer productivity over performance. The meta-insight: Rust programmers optimize at the system level (cache efficiency, memory layout, SIMD). Java programmers optimize at the runtime level (JIT-friendliness, GC tuning, object pooling). Python programmers optimize at the architecture level (choosing the right library, offloading to compiled extensions, parallelizing across processes). |

### Session 8 — Concurrency Performance

**Goal:** compare the performance characteristics of concurrent and parallel execution across the three languages — Rayon's work-stealing parallelism in Rust, ForkJoinPool/virtual threads in Java, multiprocessing/GIL workarounds in Python — including false sharing, lock contention, Amdahl's law limitations, and when parallel execution actually improves throughput.

| Step | Time | Activity |
|------|------|----------|
| 8.1 | 30 min | **Rust: Rayon parallelism and cache-aware concurrency.** Read Matthews Ch.11 pp. 227–231 (Rayon parallelism). Read Bos Ch.7 pp. 127–159 (processor architecture, cache coherence with MESI protocol, performance impact of false sharing). Read Gjengset Ch.10 pp. 167–192 (concurrency performance: atomics, memory ordering, compare-and-exchange, stress testing). Key insight: Rayon achieves near-linear parallel speedup for data-parallel workloads because it addresses the three main barriers to parallel performance. (1) Work distribution: Rayon's work-stealing scheduler dynamically balances work across threads — if one thread finishes early, it steals tasks from other threads' queues, avoiding idle cores. (2) Synchronization overhead: parallel iterators use `Send + Sync` bounds enforced at compile time, requiring zero runtime synchronization for read-only access. When mutation is needed, Rayon's `reduce` pattern partitions data into independent chunks processed lock-free, then combines results. (3) Cache coherence: Bos Ch.7 explains that on modern CPUs, the MESI/MOESI cache coherence protocol ensures that modified cache lines are synchronized across cores. False sharing (two threads modifying adjacent data on the same 64-byte cache line) causes cache line invalidation storms, degrading performance below single-threaded levels. Rust prevents accidental false sharing through ownership: two threads cannot simultaneously have mutable access to adjacent data unless explicitly arranged. Atomic operations (`AtomicUsize::fetch_add` with `Ordering::Relaxed`) are the cheapest synchronization primitive — approximately 10–20ns per operation on contended data, versus 200–500ns for a `Mutex::lock()` on contended locks. |
| 8.2 | 30 min | **Java: threading performance, false sharing, and Amdahl's law.** Read Oaks Ch.9 pp. 267–306 (threading and synchronization performance: thread pools, ForkJoinPool, synchronization costs, false sharing, biased locking, thread monitoring). Read Goetz Ch.11 pp. 221–243 (performance and scalability: thinking about performance, Amdahl's law, costs introduced by threads — context switching, memory synchronization, blocking — reducing lock contention). Read Goetz Ch.12 pp. 245–276 (testing concurrent programs: performance testing). Key insight: Goetz Ch.11 provides the theoretical framework for reasoning about concurrent performance. Amdahl's law: the maximum speedup from N processors is `1 / (S + (1-S)/N)` where S is the serial fraction. If 10% of work is serial, maximum speedup is 10× regardless of processor count. Lock contention is the primary serial bottleneck in Java programs: contended locks force threads to wait (context switch: ~10–20μs), and high contention creates convoys (threads queue behind the lock, reducing parallelism). Strategies to reduce lock contention: (1) reduce lock scope (hold locks for less time), (2) reduce lock frequency (use `ConcurrentHashMap` instead of `synchronized HashMap`), (3) use lock striping (partition data so different threads lock different stripes), (4) use lock-free algorithms (`AtomicInteger.compareAndSet`). False sharing in Java: the `@Contended` annotation (`@jdk.internal.vm.annotation.Contended`) pads fields to separate cache lines. `Thread.onSpinWait()` (Java 9+) hints the CPU during busy-wait loops. Virtual threads (Java 21) change the performance equation: instead of tuning a fixed thread pool size, spawn millions of virtual threads and let the JVM scheduler handle carrier thread utilization. Virtual threads eliminate the performance cost of blocking I/O but do not improve CPU-bound parallelism. |
| 8.3 | 25 min | **Python: GIL, multiprocessing, and parallel performance.** Read Gorelick & Ozsvald Ch.9 pp. 245–308 (multiprocessing: processes vs threads, Joblib, work queues, IPC mechanisms — Manager.Value, Redis, RawValue, mmap — sharing numpy arrays). Read Ramalho Ch.19–20 pp. 695–772 (concurrency models: GIL, threads vs processes vs coroutines, concurrent executors). Read Slatkin Item 79 pp. 393–398 (concurrent.futures for true parallelism). Key insight: Python's GIL means that threads cannot execute Python bytecode in parallel on multiple cores. For CPU-bound work, `threading` provides zero speedup and may actually degrade performance (GIL contention overhead). `multiprocessing.Pool.map(fn, data)` achieves parallelism by forking separate processes, each with its own GIL. The overhead is significant: data must be serialized (pickled) to cross process boundaries, and each process has its own memory space (no shared heap). For a parallelizable task, `ProcessPoolExecutor` provides speedup proportional to the number of cores minus the serialization overhead. The break-even point depends on task granularity: tasks shorter than ~1ms are dominated by inter-process communication overhead. NumPy operations automatically release the GIL during C-level computation, enabling threads to achieve true parallelism for numerical work: `np.dot(a, b)` uses multi-threaded BLAS (OpenBLAS, MKL) that runs independently of the GIL. This is why data science Python is not as GIL-constrained as it appears — the hot loops execute in C/Fortran, and the GIL only governs the Python-level orchestration. Free-threaded Python (PEP 703, Python 3.13+) aims to remove this limitation entirely, but at the cost of 5–10% single-threaded performance overhead (biased reference counting, per-object locks). |
| 8.4 | 25 min | **Cross-language comparison: parallel performance spectrum.** Key insight: for a CPU-bound embarrassingly parallel workload (e.g., processing 1 million independent items on 8 cores): Rust with Rayon achieves approximately 7.5–7.8× speedup (near-linear, limited only by Amdahl's law for the reduce phase and cache coherence overhead). Java with `ForkJoinPool` or parallel streams achieves approximately 6–7× speedup (close to Rust, slightly more overhead from GC pauses and JIT compilation of parallel code paths). Python with `multiprocessing.Pool` achieves approximately 4–6× speedup (limited by serialization overhead for data transfer between processes). Python with NumPy (GIL-releasing) achieves approximately 6–7× speedup for array operations (BLAS/LAPACK thread pool, no serialization overhead). For I/O-bound workloads (e.g., 10,000 HTTP requests): Rust with tokio achieves high throughput with ~4 OS threads. Java with virtual threads achieves comparable throughput with millions of virtual threads. Python with asyncio achieves comparable throughput on a single thread. The final insight ties performance back to language philosophy: Rust gives you the tools to achieve maximum performance and trusts you to use them correctly (the compiler prevents safety errors, not performance errors). Java gives you reasonable performance by default and provides JVM tuning knobs for optimization. Python gives you the fastest development time and delegates performance-critical work to compiled extensions. Performance is rarely the reason to choose a language — developer productivity, ecosystem maturity, and team expertise dominate the decision for most real-world systems. |
