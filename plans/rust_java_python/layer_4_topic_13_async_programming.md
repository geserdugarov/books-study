# Layer 4 · Topic 13 — Async Programming

> How each language enables high-concurrency I/O without one OS thread per task — Rust's zero-cost stackless coroutines compiled to state machines, Java's virtual threads (Project Loom) that make blocking code scalable, and Python's `asyncio` coroutines evolved from generators. Builds directly on §12 (Concurrency & Parallelism) — same problems, different execution model.

### Coverage Gaps

The owned books **do not** cover:

- **Rust `Future` trait internals and the state machine desugaring in depth** — Gjengset Ch.8 covers `Poll`, `Pin`, and waking at a conceptual level, but neither Gjengset nor Matthews provides the detailed state machine transformation that the compiler performs when desugaring `async fn` into an anonymous `enum` implementing `Future`; the Rust Async Book (online) and Rust RFC 2394 are needed for this level of detail
- **Tokio runtime architecture and internals (work-stealing scheduler, I/O driver, timer)** — Matthews Ch.8 uses Tokio practically and Gjengset mentions runtimes, but no owned book explains how Tokio's multi-threaded work-stealing scheduler operates, how its I/O driver integrates with `epoll`/`kqueue`/`io_uring`, or how its timer wheel works; the tokio.rs documentation, Tokio blog posts, and Alice Ryhl's articles fill this gap
- **`async-std` and alternative Rust async runtimes** — only Tokio is covered; `async-std`, `smol`, `glommio` (io_uring-based), and embassy (embedded) are not discussed in any owned book
- **Rust `Stream` trait and async iterators** — no owned book covers the `Stream` trait (the async equivalent of `Iterator`), `StreamExt` combinators, or the ongoing stabilization of `AsyncIterator` in the standard library; the `futures` crate documentation and the Rust Async Book are needed
- **Self-referential futures and `Pin` at the implementation level** — Gjengset Ch.8 pp. 126–132 introduces `Pin`/`Unpin` conceptually but does not provide a complete worked example of building a self-referential future, the `pin_project` crate, or why `Pin` is required specifically for self-referential types generated by `async` blocks; the `pin-project` documentation and blog posts by withoutboats and tmandry are needed
- **Java `CompletableFuture` in depth (combinators, composition patterns, exception handling)** — Rahman Ch.1 introduces it briefly, Evans Ch.6/Ch.16 covers it partially, but no owned book provides comprehensive coverage of all 50+ `CompletableFuture` methods, composition patterns (`thenCompose` vs `thenCombine` vs `allOf`), or exception handling strategies (`exceptionally`, `handle`, `whenComplete`); the `CompletableFuture` Javadoc and Baeldung tutorials are needed
- **Netty architecture and NIO-based async I/O in Java** — Rahman Ch.6 covers blocking vs non-blocking I/O conceptually and discusses event-driven architecture, but no owned book explains Netty's `EventLoopGroup`, `Channel`, `ChannelHandler` pipeline, or `ByteBuf` API; Netty documentation is needed
- **Python asyncio event loop internals (selector-based implementation, `call_soon`, `call_later`, scheduling)** — Ramalho Ch.21 and Slatkin Items 75–78 use asyncio extensively but at the user level; Hattingh Ch.3 covers the event loop and graceful shutdown patterns in more practical depth (signal handling, executor cleanup); Shaw pp. 265–278 covers coroutine internals at the CPython level; however, the low-level event loop implementation details (selector integration, callback scheduling internals) still require the CPython source code and `uvloop` documentation
- **Python asyncio cancellation semantics in depth (`Task.cancel()`, `CancelledError` propagation, `TaskGroup` cancellation)** — Hattingh Ch.3 covers graceful shutdown including `return_exceptions=True` in `gather()` and signal handling, but does not cover Python 3.11's `TaskGroup` and `ExceptionGroup` cancellation behavior (the book predates Python 3.11); the Python docs and PEP 654 are still needed for structured cancellation patterns
- **Cross-language comparison of the colored function problem** — no single owned book discusses the "colored function" problem (Bob Nystrom's "What Color is Your Function?" concept) and how each language's async model creates or avoids function coloring; this requires synthesis from the original blog post, Java Loom articles explaining why virtual threads avoid coloring, and Rust community discussions about the async/sync split

### Session 1 — Async Model Fundamentals

**Goal:** understand why async programming exists — the problem of blocking I/O in high-concurrency scenarios — and how each language solves it: Rust's stackless coroutines compiled to state machines, Java's virtual threads (Project Loom) plus the legacy reactive approach, and Python's `asyncio` coroutines built on generators.

| Step | Time | Activity |
|------|------|----------|
| 1.1 | 30 min | **Rust: why async and how it differs from threads.** Read Gjengset Ch.8 pp. 117–121 (what's the deal with asynchrony, synchronous vs asynchronous interfaces, multithreading limitations, the standardized polling approach). Read Matthews Ch.8 pp. 157–161 (runtimes, thinking asynchronously). Key insight: Rust's async model is based on stackless coroutines — `async fn` is compiled into a state machine that implements the `Future` trait. Unlike OS threads, futures do not own a stack; they store only the local variables needed across `await` points. This means thousands of concurrent tasks can exist with minimal memory overhead (each future is typically a few hundred bytes, vs 2–8 MB per OS thread stack). The critical design decision: Rust futures are lazy — they do nothing until polled. This contrasts with most other languages where creating a task immediately starts executing it. The executor (runtime) is not part of the standard library; users choose Tokio, async-std, or others. This separation of the `Future` trait (stdlib) from the executor (ecosystem) is unique to Rust. |
| 1.2 | 30 min | **Java: from threads to virtual threads to reactive.** Read Rahman Ch.1 pp. 1–30 (brief history of Java threads, hidden costs pp. 9–11, executor framework pp. 18–21, `CompletableFuture` pp. 23–25, reactive paradigm pp. 25–27, virtual threads promise pp. 28–30). Skim Lea Ch.4 pp. 291–310 (oneway messages and thread-based services) for the foundational design patterns that evolved into modern executors and virtual threads. Key insight: Java's async story has three eras. Era 1 (Java 1.0–4): one OS thread per task, limited scalability. Era 2 (Java 5–20): `ExecutorService` + `Future` + `CompletableFuture` — thread pools manage concurrency but callbacks create complexity; reactive frameworks (RxJava, Reactor, Netty) provide async I/O but with steep learning curves and stack trace loss. Era 3 (Java 21+): virtual threads (JEP 444) — lightweight threads scheduled by the JVM on carrier threads, making blocking code scalable without callbacks. Virtual threads are the JVM's answer to stackless coroutines, but they are stackful — each virtual thread has its own stack, managed by the JVM rather than the OS. The key philosophical difference: Java chose to make existing blocking code async-friendly (virtual threads look like regular threads), while Rust and Python chose explicit async/await syntax. |
| 1.3 | 30 min | **Python: asyncio coroutines and the evolution from generators.** Read Ramalho Ch.19 pp. 695–711 (the big picture, jargon, GIL, spinner with coroutines). Read Ramalho Ch.21 pp. 775–781 (definitions, asyncio example, Guido's trick for reading async code, awaitables). Read Hattingh Ch.1 pp. 1–8 (restaurant analogy for async thinking, what problem asyncio solves) and Ch.2 pp. 9–20 (threading vs asyncio tradeoffs, race condition case study). Key insight: Python's async model evolved from generators. PEP 255 (Python 2.2) introduced `yield`, PEP 342 (Python 2.5) added `send()` to generators (creating "classic coroutines"), PEP 380 (Python 3.3) added `yield from` for coroutine delegation, and PEP 492 (Python 3.5) introduced `async`/`await` as first-class syntax replacing `@asyncio.coroutine` + `yield from`. Native coroutines (`async def`) are syntactically distinct from generators but share the same underlying machinery — they are stackless (execution state is saved in the coroutine frame when suspended). Python's event loop is single-threaded by default, meaning all coroutines share one OS thread. This is both a simplification (no data races between coroutines) and a limitation (CPU-bound work blocks the event loop). Guido's trick: to understand async code, mentally remove `async` and `await` keywords and read it as synchronous code — the control flow is the same. |
| 1.4 | 20 min | **Cross-language comparison.** Map the three approaches: Rust — stackless coroutines, lazy futures, no built-in runtime, zero-cost abstraction, compile-time safety via `Send`/`Sync`; Java — stackful virtual threads (OS thread behavior without OS thread cost), reactive as legacy alternative, no special syntax needed, JVM-managed scheduling; Python — stackless coroutines, single-threaded event loop, explicit `async`/`await`, GIL makes coroutines safe but limits parallelism. Note the fundamental design tradeoff: Rust optimizes for zero overhead (no runtime, no allocations for futures), Java optimizes for developer ergonomics (write blocking code, get async behavior), Python optimizes for simplicity (single-threaded, no data races). Compare the memory cost of 10,000 concurrent tasks: Rust futures ~MB total, Java virtual threads ~tens of MB, Python coroutines ~MB (similar to Rust), but OS threads would require ~20–80 GB. |

### Session 2 — `Future` Trait and Future Types

**Goal:** understand the `Future` abstraction in each language — Rust's zero-cost `Future` trait with `poll()`, Java's `CompletableFuture` with callback chains, and Python's awaitable protocol — including how each encodes completion, error, and pending states.

| Step | Time | Activity |
|------|------|----------|
| 2.1 | 35 min | **Rust: the `Future` trait and poll-based execution.** Read Gjengset Ch.8 pp. 121–126 (ergonomic futures, `Future` trait, `Poll::Ready` vs `Poll::Pending`, the contract). Key insight: Rust's `Future` trait has a single method: `fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>`. This is a pull-based model — the executor calls `poll()` on the future, and the future returns `Poll::Pending` if not ready or `Poll::Ready(value)` if complete. The `Context` contains a `Waker` that the future uses to signal the executor when it should be polled again. This design is zero-cost: no heap allocation is required for the future itself (it can be stack-allocated), no reference counting, no garbage collection. The state machine generated by the compiler stores all local variables that need to survive across `.await` points as fields in an anonymous `enum`, where each variant represents a suspension point. Contrast with callback-based futures (Java, JavaScript): Rust's model inverts control — the executor drives the future, rather than the future scheduling callbacks. |
| 2.2 | 30 min | **Java: `Future`, `CompletableFuture`, and composability.** Read Goetz Ch.6 pp. 113–123 (task execution, Executor framework, `Future` interface). Read Evans Ch.6 pp. 197–205 (futures, `CompletableFuture`, tasks and execution). Read Rahman Ch.3 pp. 101–103 (`Callable` and `Future`). Key insight: Java has two `Future` types. `java.util.concurrent.Future` (Java 5) is a blocking handle: `get()` blocks until the result is available, `cancel()` requests cancellation, `isDone()` checks completion. `CompletableFuture` (Java 8) adds callback-based composition: `thenApply()` (map), `thenCompose()` (flatMap), `thenCombine()` (zip), `allOf()` (join all), `exceptionally()` (recover). `CompletableFuture` is push-based: completion triggers downstream callbacks automatically. With virtual threads (Java 21+), the blocking `Future.get()` is now cheap — virtual threads make blocking acceptable again, reducing the need for `CompletableFuture` callback chains. This is a philosophical reversal: Java spent years building `CompletableFuture` to avoid blocking, then virtual threads made blocking cheap again. |
| 2.3 | 25 min | **Python: awaitables and the coroutine protocol.** Read Ramalho Ch.21 pp. 781–785 (awaitables, native coroutines vs generators, the all-or-nothing problem). Read Slatkin Item 75 pp. 364–368 (achieve highly concurrent I/O with coroutines). Read Hattingh Ch.3 pp. 33–42 (tasks and futures: `create_task` vs `ensure_future`, the distinction between coroutines, tasks, and futures). Key insight: in Python, an "awaitable" is any object with an `__await__` method returning an iterator. Three things are awaitable: (1) coroutine objects (from `async def`), (2) `asyncio.Task` objects, (3) `asyncio.Future` objects. `asyncio.Future` is a low-level callback-based primitive (similar to `CompletableFuture`) — user code rarely creates these directly. `asyncio.Task` wraps a coroutine and schedules it on the event loop. The "all-or-nothing problem" (p. 785): you cannot `await` in a regular function — if any function in a call chain needs to `await`, every caller up to the event loop must be `async def`. This is the colored function problem in action. Compare: Rust has the same coloring constraint (`.await` only in `async` context), Java's virtual threads do not (any code can call blocking operations). |
| 2.4 | 20 min | **Cross-language comparison of future semantics.** Build a table: Rust `Future` — pull-based (executor polls), lazy (does nothing until polled), zero-cost, no heap allocation required, cancellation by dropping. Java `CompletableFuture` — push-based (callbacks triggered on completion), eager (computation starts immediately), heap-allocated, cancellation via `cancel()`. Python `asyncio.Future`/`Task` — pull-based at protocol level (event loop drives via `__await__`), eager when wrapped in `Task` (starts when scheduled), heap-allocated, cancellation via `cancel()` raising `CancelledError`. Note that Java `Future.get()` is synchronous blocking while Python `await future` is asynchronous suspension — the syntax looks different but the semantic intent (wait for a result) is the same. Rust's design is unique in making the future itself a value type that can be composed without allocation. |

### Session 3 — Runtime Architecture

**Goal:** understand what sits between the `Future`/coroutine and the operating system — Tokio's multi-threaded work-stealing scheduler and I/O driver (Rust), Netty's event loop and Project Loom's virtual thread scheduler (Java), and Python's `asyncio` single-threaded event loop.

| Step | Time | Activity |
|------|------|----------|
| 3.1 | 35 min | **Rust: Tokio runtime internals.** Read Matthews Ch.8 pp. 159–164 (runtimes, defining a runtime with `#[tokio::main]`, futures and handling results). Read Gjengset Ch.10 pp. 175–176 (asynchrony and parallelism). Read the Tokio tutorial online ("Hello Tokio", "Spawning", "I/O"). Key insight: Tokio is a multi-threaded runtime with three core components: (1) the task scheduler (work-stealing across a thread pool — each worker thread has a local queue plus a shared injection queue; idle workers steal tasks from busy workers), (2) the I/O driver (integrates with OS async I/O: `epoll` on Linux, `kqueue` on macOS — registers interest in I/O events and wakes the corresponding tasks), (3) the timer (hierarchical timer wheel for `tokio::time::sleep` and timeouts). `#[tokio::main]` is a macro that creates a `Runtime` and blocks on the provided `async fn main()`. `tokio::spawn()` creates a new task (like a lightweight thread) that runs concurrently. `Runtime::new()` creates a multi-threaded runtime; `runtime::Builder::new_current_thread()` creates a single-threaded one. The key design decision: Tokio tasks must be `Send` (can move between threads) unless using `LocalSet` (single-thread mode). This constraint enables work-stealing but prevents thread-local state in tasks. |
| 3.2 | 30 min | **Java: virtual thread scheduler and Netty's event loop.** Read Rahman Ch.2 pp. 46–51 (how virtual threads work under the hood: stack frames, carrier threads, OS involvement). Read Rahman Ch.3 pp. 113–123 (continuation mechanics, building virtual threads from scratch, I/O polling). Read Rahman Ch.6 pp. 249–276 (blocking vs non-blocking I/O, event-driven architecture). Skim Lea Ch.3.4 pp. 199–240 (concurrency control utilities: semaphores, latches, barriers — the foundational patterns that underpin modern runtime schedulers). Key insight: Java's virtual thread scheduler uses `ForkJoinPool` as the carrier thread pool — virtual threads are mounted on carrier threads (platform threads) for execution and unmounted when they block on I/O or synchronization. The JVM rewrites blocking operations (socket I/O, `Thread.sleep`, lock acquisition) to yield the carrier thread instead of blocking it. This is transparent to user code — `Thread.sleep()` in a virtual thread yields the carrier, while the same call in a platform thread blocks the OS thread. Netty takes a different approach: it uses a small number of event loop threads, each running an I/O multiplexer (`epoll`), and processes events via `ChannelHandler` pipelines — this is explicit async I/O, requiring callback-based programming. Virtual threads make Netty-style NIO programming unnecessary for most use cases — but Netty is still relevant for protocols requiring byte-level control (HTTP/2, WebSocket). |
| 3.3 | 25 min | **Python: the asyncio event loop.** Read Ramalho Ch.21 pp. 797–810 (delegating tasks to executors, writing asyncio servers — TCP server, FastAPI). Read Slatkin Item 78 pp. 389–393 (maximize responsiveness of asyncio event loops). Read Hattingh Ch.3 pp. 27–33 (event loop walkthrough) and pp. 55–73 (graceful startup and shutdown: signal handling, executor cleanup, `return_exceptions=True` in `gather()`). Read Beazley Ch.9.14 (blocking operations and concurrency: nonblocking I/O, I/O polling, threads, asyncio). Read the Python event loop docs online. Key insight: Python's asyncio event loop is single-threaded: it runs coroutines, handles I/O events (via `selectors` module, which wraps `epoll`/`kqueue`/`select`), manages timers, and schedules callbacks — all on one OS thread. `asyncio.run(main())` creates an event loop, runs the coroutine, and cleans up. `asyncio.create_task()` schedules a coroutine for concurrent execution. The loop repeatedly: (1) checks the ready queue for runnable callbacks/coroutines, (2) polls the OS for I/O events (with a timeout), (3) runs expired timers. CPU-bound work blocks the entire event loop (all other coroutines are starved). Mitigation: `loop.run_in_executor()` offloads blocking/CPU work to a `ThreadPoolExecutor` or `ProcessPoolExecutor`. `uvloop` replaces the default event loop with a `libuv`-based implementation for 2–4x throughput improvement. The single-threaded model means no locks are needed between coroutines — shared state is safe because only one coroutine runs at a time. |
| 3.4 | 20 min | **Cross-language comparison of runtime architecture.** Compare: Rust/Tokio — multi-threaded work-stealing scheduler, tasks must be `Send`, user chooses runtime; Java/Loom — `ForkJoinPool`-based scheduler built into JVM, virtual threads are transparently scheduled, user writes blocking code; Python/asyncio — single-threaded event loop, no thread safety concerns between coroutines, CPU-bound work requires executor delegation. Note the I/O integration: Tokio wraps `epoll`/`kqueue` in its I/O driver, Java's virtual thread scheduler intercepts blocking I/O calls at the JVM level, Python uses `selectors` module. The philosophical differences: Rust gives maximum control (choose your runtime, configure your thread pool), Java maximizes compatibility (existing blocking code "just works" on virtual threads), Python maximizes simplicity (single thread, no races). |

### Session 4 — `async`/`await` Syntax and Desugaring

**Goal:** understand how `async`/`await` syntax is compiled/interpreted in each language — Rust's state machine transformation, Java's deliberate absence of async syntax (replaced by virtual threads), and Python's evolution from `yield from` to `async`/`await`.

| Step | Time | Activity |
|------|------|----------|
| 4.1 | 35 min | **Rust: desugaring async into state machines.** Read Gjengset Ch.8 pp. 124–126 (async/await). Read the Rust Async Book "async/.await" chapter online. Read Tyler Mandry's "How Rust optimizes async/await" blog post online. Key insight: `async fn foo(x: i32) -> String` desugars to `fn foo(x: i32) -> impl Future<Output = String>`. The compiler transforms the function body into an anonymous `enum` (state machine) where each variant holds the local variables live at that `.await` point. For example, an `async fn` with two `.await` points becomes a 3-variant enum (Start, AfterFirstAwait, AfterSecondAwait). Each call to `poll()` resumes from the current state, executes until the next `.await`, saves state into the next variant, and returns `Poll::Pending`. This is a stackless transformation — no heap allocation for the coroutine frame unless the future is boxed (`Box::pin(future)`). The size of the generated future equals the maximum size of any state variant (plus discriminant). `tokio::spawn()` requires boxing (`Box<dyn Future>` or equivalent) because task ownership transfers to the runtime. The zero-cost claim: the generated state machine is at least as efficient as a hand-written state machine — the compiler can optimize across `await` points. |
| 4.2 | 25 min | **Java: no async/await — virtual threads instead.** Read Rahman Ch.2 pp. 37–49 (adapting to virtual threads, demonstrating virtual thread creation, simplifying async operations). Read Evans Ch.18 pp. 618–625 (Project Loom, virtual threads, programming with virtual threads). Key insight: Java deliberately chose NOT to add `async`/`await` syntax. Instead, virtual threads make every blocking call an implicit suspension point — `Thread.sleep()`, `socket.read()`, `lock.lock()` all suspend the virtual thread without blocking the carrier thread. This means existing code (libraries, frameworks, JDBC drivers) automatically benefits from virtual threads without modification. The tradeoff: Java's approach requires JVM-level support for intercepting every blocking call, which is complex to implement and has limitations (pinning: `synchronized` blocks and native methods pin the virtual thread to its carrier thread, preventing it from being unmounted). The benefit: no colored function problem — any function can call any other function, whether it performs I/O or not. There is no "sync" vs "async" split in the API. Compare this to Rust and Python where `async fn` creates a fundamentally different kind of function from regular `fn`. |
| 4.3 | 30 min | **Python: from generators to async/await.** Read Ramalho Ch.17 pp. 641–652 (classic coroutines, `send()`, `yield from`). Read Beazley Ch.6 pp. 139–152 (generators and the bridge to awaiting: `yield`, `yield from`, enhanced generators, the historical connection from generators to `async`/`await` — a concise, distilled treatment of the evolution). Read Ramalho Ch.21 pp. 782–785 (the secret of native coroutines vs generators). Read Shaw pp. 265–278 (asynchronous programming internals, generators, coroutines, async generators). Read Beazley Ch.5.23 (async functions and `await` — how async integrates with the rest of Python's function system). Read the Brett Cannon blog post "How the heck does async/await work in Python 3.5?" online. Key insight: Python's `async`/`await` evolved through four stages: (1) Generators (`yield` suspends, caller calls `next()`); (2) Enhanced generators (`send()` sends values into generator, enabling two-way communication); (3) `yield from` delegation (PEP 380, allows one generator to delegate to another, forming coroutine chains); (4) `async def` + `await` (PEP 492, native coroutines with dedicated syntax). Under the hood, `async def` creates a coroutine object whose `__await__` method returns a generator-like iterator. `await expr` is conceptually similar to `yield from expr.__await__()` — it suspends the coroutine and propagates the yielded value up to the event loop. The event loop catches the final `StopIteration` with the return value. Native coroutines differ from generators in one critical way: you cannot use `yield` in an `async def` (except in `async def` generators with `yield`). Shaw's CPython internals chapters show the bytecode: `GET_AWAITABLE`, `YIELD_FROM`, `SEND` opcodes that implement the protocol. |
| 4.4 | 20 min | **Comparison: desugaring strategies.** Map the spectrum: Rust — compile-time transformation into state machine `enum`, zero runtime overhead, explicit suspension points (`.await`); Python — interpreter-level coroutine frames, `await` suspends via `__await__` protocol (evolved from `yield from`), single-threaded so no thread safety concerns; Java — no source-level transformation, JVM intercepts blocking calls at runtime and unmounts virtual threads from carriers, every blocking call is an implicit suspension point. The key insight: Rust and Python took the "explicit async" path (you see `.await`/`await` at every suspension point), while Java took the "transparent async" path (blocking calls suspend transparently). Explicit async gives the programmer visibility into where context switches happen; transparent async gives backward compatibility with existing code. Both approaches have precedent: Go's goroutines are transparent (like Java), JavaScript/C#/Python/Rust use explicit `async`/`await`. |

### Session 5 — Pinning (`Pin<T>`) and Self-Referential Futures

**Goal:** understand why Rust requires `Pin<T>` for async programming — the problem of self-referential types that arise from compiler-generated state machines — and appreciate that Java and Python do not face this problem because of garbage collection and reference-based semantics.

| Step | Time | Activity |
|------|------|----------|
| 5.1 | 40 min | **Rust: why Pin exists and how it works.** Read Gjengset Ch.8 pp. 126–132 (Pin and Unpin). Read the Rust Async Book "Pinning" chapter online. Read the fasterthanlime blog post "Pin and suffering" online. Key insight: when the compiler desugars `async fn`, the generated state machine may contain self-references — a variable in one state may borrow another variable in the same state. Moving the state machine in memory would invalidate these internal pointers. `Pin<P>` is a wrapper that prevents the pointee from being moved after it has been pinned. `Pin<&mut T>` guarantees that `T` will not be moved. `Unpin` is an auto-trait: types that implement `Unpin` can be freely moved even when pinned (because they contain no self-references). Most types are `Unpin` — only compiler-generated futures and types with manual self-references are `!Unpin`. The `Future::poll()` method takes `Pin<&mut Self>` rather than `&mut Self` precisely to prevent the caller from moving the future between polls. Creating a pinned future: `Box::pin(future)` (heap pinning) or `tokio::pin!(future)` (stack pinning). The `pin_project` crate provides ergonomic projection from a pinned struct to its fields. This is the most Rust-specific topic in async programming — no other mainstream language needs this concept because GC handles object relocation transparently. |
| 5.2 | 25 min | **Practical Pin patterns.** Read the `pin-project` crate documentation online. Read the Rust RFC 2349 (Pin) online for design rationale. Key insight: in practice, most Rust programmers interact with `Pin` in three contexts: (1) `async fn` — the compiler handles pinning automatically; `.await` internally pins the sub-future. (2) Manual `Future` implementations — `poll()` receives `Pin<&mut Self>`, and you must use `unsafe` or `pin-project` to access fields. (3) Storing futures in collections — `Vec<Pin<Box<dyn Future>>>` allows dynamic collections of futures. The `pin-project` crate eliminates most `unsafe` by providing `#[pin_project]` derive macro that generates safe pin projections. Common pitfall: accidentally moving a pinned future (this is prevented at compile time by `Pin`'s API, but can be confusing). The mental model: once you `.await` a future, it is implicitly pinned — you cannot move it and `.await` it again from a different location. `Pin` is the compile-time enforcement of this invariant. |
| 5.3 | 15 min | **Why Java and Python don't need Pin.** Key insight: Java objects live on the GC heap. The GC may move objects (for compaction), but it transparently updates all references — user code never sees raw memory addresses, so self-references are not a problem. Virtual thread stacks are managed by the JVM and can be moved (stack frames are stored on the heap), but the JVM handles all pointer updates. Python objects use reference counting and are never moved in memory (CPython allocates objects with `PyObject_Malloc` and their address is stable for their lifetime). Therefore, coroutine frames in Python can safely contain self-references (internal pointers remain valid). The key insight: `Pin` is a consequence of Rust's ownership model — the language gives programmers raw pointers and move semantics, so it must also give them tools to prevent moves when they would be unsafe. Languages with managed memory (Java, Python) handle this transparently at the cost of GC overhead. |
| 5.4 | 15 min | **Summary and mental model.** Build a flowchart: Do you write `async fn` and use `.await`? Yes — `Pin` is handled automatically by the compiler and runtime, you rarely need to think about it. Do you implement `Future` manually? Yes — use `pin-project` crate to safely project through `Pin`. Do you store futures in collections or trait objects? Yes — use `Box::pin()` or `Pin<Box<dyn Future>>`. For Java/Python developers learning Rust: `Pin` is the price Rust pays for zero-cost abstractions in async code — it ensures safety without GC, but adds complexity to manual `Future` implementations. |

### Session 6 — Cancellation Semantics

**Goal:** compare how each language handles cancellation of in-progress async operations — Rust's drop-based cancellation (dropping a future cancels it), Java's interruption mechanism and structured concurrency scopes, and Python's `CancelledError` propagation.

| Step | Time | Activity |
|------|------|----------|
| 6.1 | 30 min | **Rust: cancellation by dropping.** Read Gjengset Ch.8 pp. 136–140 (tasks and subexecutors, spawn, tying it all together). Read the Tokio documentation on `select!` and graceful shutdown online. Key insight: in Rust, cancelling a future is simply dropping it. When a `Future` is dropped, its destructor runs, cleaning up any resources via RAII. If a future is suspended at an `.await` point and is dropped, the sub-future it was awaiting is also dropped recursively. `tokio::select!` enables racing multiple futures — the first to complete wins, and all other branches are dropped (cancelled). This is elegant but has a subtle implication: any code after an `.await` point in a dropped future will never execute. Resources acquired before the `.await` but released after it may leak if not protected by RAII. Best practice: use RAII guards (like `tokio::sync::MutexGuard`) rather than manual cleanup-after-await patterns. `JoinHandle::abort()` cancels a spawned task by marking it for cancellation at the next `.await` point. Compare with Java/Python: Rust's cancellation is synchronous and deterministic (drop runs immediately), while Java/Python cancellation is cooperative and asynchronous (the task must check for cancellation). |
| 6.2 | 30 min | **Java: interruption, `Future.cancel()`, and structured concurrency.** Read Goetz Ch.7 pp. 135–161 (task cancellation, interruption, stopping thread-based services, handling abnormal termination). Read Lea Ch.3.1 pp. 149–170 (dealing with failure: exceptions and cancellation in concurrent contexts — the foundational thinking behind Java's cooperative cancellation model). Read Rahman Ch.4 pp. 125–140 (challenge of unstructured concurrency, promise of structured concurrency, `StructuredTaskScope` API). Key insight: Java has multiple cancellation mechanisms. (1) `Thread.interrupt()` — sets the interrupt flag; blocking methods (`sleep`, `wait`, `read`) throw `InterruptedException`. Cooperative: the task must check `Thread.interrupted()` or handle `InterruptedException`. (2) `Future.cancel(boolean mayInterruptIfRunning)` — attempts to cancel the computation; if `mayInterruptIfRunning` is true, interrupts the thread. (3) Structured concurrency (`StructuredTaskScope` in Java 21+) — cancellation flows from parent to children: when a scope's `shutdown()` is called (e.g., by `ShutdownOnFailure` policy when one subtask fails), all other subtasks are cancelled (interrupted). This provides automatic, hierarchical cancellation. Compare with Rust: Java's cancellation is cooperative (task must respond to interrupt), while Rust's is non-cooperative (drop is unconditional). Java's structured concurrency is the closest to Rust's drop-based semantics — scope exit cancels children. |
| 6.3 | 25 min | **Python: `CancelledError` and `TaskGroup`.** Read Hattingh Ch.3 pp. 55–73 (graceful startup and shutdown: signal handling, `return_exceptions=True` in `gather()`, waiting for executor during shutdown — practical cancellation and cleanup patterns). Read the Python docs on `Task.cancel()` and `TaskGroup` online. Read PEP 654 (Exception Groups) online for background on `TaskGroup` cancellation. Key insight: Python's cancellation is cooperative. `task.cancel()` schedules a `CancelledError` to be thrown into the coroutine at the next `await` point. The coroutine can catch `CancelledError` (e.g., for cleanup), but should re-raise it — swallowing it prevents cancellation from propagating. `asyncio.TaskGroup` (Python 3.11+) provides structured concurrency: if any task in the group raises an exception, all other tasks are cancelled and the exceptions are collected into an `ExceptionGroup`. `asyncio.timeout()` and `asyncio.wait_for()` use cancellation internally — when a timeout expires, the wrapped task is cancelled. Cancellation pitfalls: `CancelledError` was changed from `Exception` to `BaseException` subclass in Python 3.9, so `except Exception:` no longer catches it — this was a deliberate fix to prevent accidentally swallowing cancellations. Compare with Rust: Python's cancellation is cooperative and asynchronous (next `await` point), Rust's is immediate (drop). Compare with Java: both are cooperative, but Python uses exception injection while Java uses thread interruption. |
| 6.4 | 15 min | **Comparison: cancellation spectrum.** Build a cancellation comparison: Rust — non-cooperative, immediate (drop), deterministic cleanup via RAII, no partial execution after cancellation. Java — cooperative, via interruption or `Future.cancel()`, structured concurrency provides hierarchical scope-based cancellation, cleanup via `finally` blocks. Python — cooperative, via `CancelledError` injection, `TaskGroup` provides structured cancellation, cleanup via `try`/`finally` or `async with`. The key insight: Rust's drop-based cancellation is the safest (no forgotten cleanup) but can cause surprising behavior (code after `.await` may never run). Java's and Python's cooperative cancellation gives the task a chance to clean up gracefully but requires discipline (always respond to interruption/cancellation, always re-raise `CancelledError`). |

### Session 7 — Backpressure and Flow Control

**Goal:** understand how each language handles the producer-consumer problem in async streams — Rust's `Stream` trait and pull-based iteration, Java's `Flow` API / Reactive Streams with explicit backpressure `request(n)`, and Python's async generators with implicit backpressure via single-consumer `async for`.

| Step | Time | Activity |
|------|------|----------|
| 7.1 | 30 min | **Rust: `Stream` and async iterators.** Read the `futures::stream::Stream` trait documentation and `tokio-stream` crate documentation online. Key insight: `Stream` is the async equivalent of `Iterator`: `fn poll_next(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Option<Self::Item>>`. Like `Future`, it is pull-based: the consumer calls `poll_next()`, which returns `Poll::Pending` (not ready), `Poll::Ready(Some(item))` (next item), or `Poll::Ready(None)` (stream ended). Backpressure is automatic: the consumer controls the rate by choosing when to call `poll_next()`. `StreamExt` provides combinators: `map`, `filter`, `buffered`, `throttle`, `chunks`. `tokio::sync::mpsc::channel` provides a bounded async channel that applies backpressure when the buffer is full (sender's `send()` returns `Poll::Pending`). `async-stream` crate provides a `stream!` macro for creating streams from `async` blocks with `yield`. The `Stream` trait is not yet in the standard library (stabilization in progress as `AsyncIterator`) — it lives in the `futures` crate. Compare with synchronous `Iterator`: same pull-based model, same combinators, but suspension at each step is possible. |
| 7.2 | 30 min | **Java: Reactive Streams and `Flow` API.** Read Rahman Ch.6 pp. 279–292 (understanding reactive streams, backpressure). Read the Reactive Streams specification online. Read the `java.util.concurrent.Flow` API Javadoc online. Key insight: Java's `Flow` API (JDK 9+) implements the Reactive Streams specification with four interfaces: `Publisher<T>` (produces items), `Subscriber<T>` (consumes items), `Subscription` (mediates between them — `request(n)` signals demand), `Processor<T,R>` (transforms items, is both `Subscriber` and `Publisher`). Backpressure is explicit: the subscriber calls `subscription.request(n)` to signal how many items it can handle. If the publisher produces faster than the subscriber consumes, it must buffer, drop, or signal error. This protocol is implemented by Project Reactor (`Flux`/`Mono`), RxJava (`Observable`/`Flowable`), and Netty. However, with virtual threads, reactive streams are less necessary for most use cases — blocking I/O on virtual threads with bounded queues provides natural backpressure without the reactive programming complexity. Rahman (pp. 292–294) discusses when reactive is still relevant: extremely high-throughput event processing where the reactive model's operator fusion and scheduler control provide benefits. |
| 7.3 | 25 min | **Python: async generators and async iteration.** Read Ramalho Ch.21 pp. 811–820 (async iterables and generators, async comprehensions). Read Hattingh Ch.3 pp. 43–55 (async context managers, async iterators, async generators, async comprehensions — practical walkthrough with examples). Read Hattingh Ch.4 pp. 75–90 (asyncio streams: TCP message queue case study demonstrating producer-consumer patterns with backpressure). Read PEP 525 (Async Generators) online. Read Slatkin Item 75 pp. 364–368 (coroutines for concurrent I/O). Key insight: Python's async equivalent of generators is `async def` with `yield` — async generators. `async for item in async_gen():` iterates asynchronously, suspending at each step. The `__aiter__`/`__anext__` protocol mirrors `__iter__`/`__next__`. Backpressure in Python is implicit and simple: `async for` drives the generator one item at a time — the producer (`__anext__`) is not called until the consumer is ready for the next item. This is naturally pull-based, like Rust's `Stream`. For multi-consumer scenarios, `asyncio.Queue` (bounded) provides explicit backpressure: `put()` blocks when the queue is full. Async comprehensions (`[x async for x in aiter]`, `{x async for x in aiter}`) provide compact syntax. Compare with Java: Python's async generators are much simpler than Reactive Streams — no `request(n)`, no `Subscription` object — but they do not support multi-subscriber, broadcast, or operator fusion patterns that reactive libraries provide. |
| 7.4 | 15 min | **Comparison: backpressure strategies.** Map the approaches: Rust `Stream` — pull-based (consumer polls), backpressure automatic, bounded channels for producer-consumer, not yet in stdlib. Java `Flow`/Reactive Streams — explicit `request(n)` backpressure protocol, push-based with demand signaling, rich operator libraries (Reactor, RxJava), being partially superseded by virtual threads + blocking queues. Python async generators — pull-based (`async for` drives), backpressure implicit (one item at a time), `asyncio.Queue` for decoupled producers/consumers. The spectrum: Python is simplest (implicit), Rust is zero-cost (pull-based, no allocation), Java is most flexible (explicit demand signaling) but also most complex. |

### Session 8 — Colored Function Problem and Synthesis

**Goal:** understand the "colored function" problem (Bob Nystrom, 2015) — that `async` functions and regular functions are different "colors" that cannot freely interoperate — and how each language deals with (or avoids) this fundamental tension. Synthesize all eight sessions into a comparative framework.

| Step | Time | Activity |

| 8.1 | 25 min | **The colored function problem defined.** Read Bob Nystrom's "What Color is Your Function?" blog post online. Key insight: in languages with `async`/`await`, functions come in two "colors": sync (blue) and async (red). Rules: (1) red functions can call blue functions, (2) blue functions cannot call red functions (cannot `await` in a non-`async` context), (3) red functions are contagious — calling a red function requires the caller to be red. This creates an ecosystem split: libraries must provide both sync and async APIs (doubling API surface), or choose one color and exclude the other. JavaScript, Python, Rust, C#, and Kotlin all have this problem to varying degrees. The only mainstream language that avoids it entirely is Go (goroutines are transparent — every function can block cheaply). Java (since Java 21) joins Go in avoiding the problem: virtual threads make blocking cheap, so there is no need for a separate "async" color. |
| 8.2 | 25 min | **How each language deals with coloring.** Read Matthews Ch.8 pp. 174–176 (mixing sync and async, when to avoid async). Read Slatkin Item 77 pp. 381–389 (mix threads and coroutines to ease transition). Read Hattingh Ch.4 pp. 90–100 (Janus queue: bridging sync and async worlds — a practical Python solution to the coloring problem). Read Rahman Ch.2 pp. 47–49 (simplifying asynchronous operations with virtual threads). Key insight: **Rust** — fully colored. `async fn` returns a `Future`, regular `fn` does not. Calling async from sync requires a runtime handle (`runtime.block_on()`). Calling sync from async risks blocking the executor (use `spawn_blocking()`). The ecosystem is split: `reqwest` (async) vs `ureq` (sync), `sqlx` (async) vs `diesel` (sync). Matthews advises: use async only when you have many concurrent I/O operations; for CPU-bound or simple programs, sync is simpler. **Python** — fully colored. `async def` is different from `def`. `asyncio.run()` bridges sync-to-async at the top level. `loop.run_in_executor()` bridges async-to-sync. Slatkin Item 77 shows the gradual migration pattern: wrap sync code in threads called from async, and wrap async code in `asyncio.run()` called from sync. The "all-or-nothing" nature means converting one function to async often cascades through the call stack. **Java** — not colored (since virtual threads). `Thread.sleep()` works in both platform and virtual threads. Libraries do not need async variants. However, the pre-Loom reactive ecosystem (Reactor, RxJava) created its own "color" with `Mono`/`Flux` types — migrating from reactive back to imperative is a current challenge. |
| 8.3 | 25 min | **Practical implications and workarounds.** Read Matthews Ch.8 pp. 176–180 (when to avoid async, tracing and debugging async code, dealing with async when testing). Read Ramalho Ch.21 pp. 825–827 (how async works and does not). Key insight: the colored function problem has practical consequences. (1) Debugging: async stack traces are harder to read because the state machine transformation (Rust) or event loop scheduling (Python) obscures the call chain. Rust: use `tracing` crate + `tokio-console`. Python: use `asyncio.get_event_loop().set_debug(True)`. Java: virtual thread stack traces look like normal thread stack traces — a major ergonomic advantage. (2) Testing: Rust requires `#[tokio::test]` instead of `#[test]`. Python requires `pytest-asyncio` or `unittest.IsolatedAsyncioTestCase`. Java: no special test infrastructure needed for virtual threads. (3) Library ecosystem: Rust and Python have parallel sync/async ecosystems. Java is converging back to a single synchronous ecosystem enabled by virtual threads. (4) Performance: Rust's async is zero-cost but adds compile-time complexity. Python's asyncio is limited to single-thread concurrency. Java's virtual threads scale to millions of concurrent tasks with simple blocking code. |
| 8.4 | 25 min | **Grand synthesis: the async programming spectrum.** Build a comprehensive comparison table across all eight subtopics. **Rust**: stackless coroutines, zero-cost `Future` trait, explicit `Pin` for safety, Tokio runtime (multi-threaded), drop-based cancellation, pull-based `Stream`, fully colored. Philosophy: maximum performance and safety at the cost of complexity. **Java**: stackful virtual threads (Loom), `CompletableFuture` for composition (legacy), JVM-managed scheduling, interruption-based cancellation, `Flow` reactive streams (legacy), NOT colored. Philosophy: backward compatibility and developer ergonomics, leveraging the JVM's ability to transform blocking into non-blocking transparently. **Python**: stackless coroutines (evolved from generators), `asyncio.Future`/`Task`, single-threaded event loop, `CancelledError`-based cancellation, async generators for streaming, fully colored. Philosophy: simplicity and readability, explicit concurrency points, limited by GIL to single-threaded async. The meta-insight: each language's async model reflects its core values. Rust chose zero-cost abstractions with explicit control. Java chose to extend its existing thread model rather than add new syntax. Python chose explicit coroutines with clear suspension points. There is no objectively "best" approach — each makes tradeoffs appropriate to its ecosystem, performance requirements, and target audience. |
