### Coverage Gaps

The owned books **do not** cover:

- **Rust `Cow<T>` in real-world API design patterns** — Blandy and Matthews introduce `Cow<T>` at the trait/type level, but none of the owned books provide extended examples of using `Cow` in public API signatures to allow callers to pass either borrowed or owned data, nor do they discuss when `Cow` is preferable to separate `&str`/`String` overloads; this requires blog posts and the Rust API Guidelines
- **Java `record` immutability limitations and workarounds** — Evans covers records as immutable data carriers, but the owned books do not explore the edge cases where records hold mutable fields (e.g., `record Pair(List<Integer> items)` is shallowly immutable), nor the community debate around "deep record immutability"; this requires JEP 395 discussions and blog analysis
- **Python's `types.MappingProxyType` for read-only dict views** — none of the owned Python books cover `MappingProxyType`, which provides a read-only view of a dictionary without copying it, analogous to Java's `Collections.unmodifiableMap`; this is documented in the Python standard library reference
- **Cross-language comparison of "effectively immutable" patterns** — no owned book systematically compares the concept of safe publication of effectively immutable objects across all three languages (Rust's ownership transfer, Java's JMM guarantees for final fields, Python's lack of a formal memory model); Goetz Ch.3 and Ch.16 cover the Java side thoroughly, but the cross-language synthesis must be constructed by the reader

### Session 1 — Immutable by Default vs Mutable by Default

**Goal:** understand the fundamental philosophical difference in how Rust, Java, and Python treat mutability at the variable and field level — Rust's opt-in mutability with `let mut`, Java's opt-out mutability with `final`, and Python's entirely mutable-by-default model with only advisory `typing.Final`.

| Step | Time | Activity |
|------|------|----------|
| 1.1 | 30 min | **Rust: immutable by default.** Read Klabnik Ch.3 pp. 31–42 (variables and mutability — `let` vs `let mut`, constants with `const`, shadowing). Read Blandy Ch.5 pp. 93–100 (shared references `&T` as immutable views). Read Gjengset Ch.1 pp. 1–5 (ownership fundamentals, the significance of the immutable-by-default design). Key insight: Rust variables are immutable by default — `let x = 5;` creates a binding that cannot be reassigned or mutated. You must explicitly opt in with `let mut x = 5;` to allow mutation. This is not just a convention; it is enforced by the compiler. The design rationale is that immutability is the safer default: it prevents accidental mutation, enables the compiler to make optimization assumptions (e.g., that a value behind `&T` will not change during the reference's lifetime), and makes code easier to reason about. Constants (`const`) go further — they are inlined at compile time and must have a type annotation. Shadowing (`let x = x + 1;`) creates a new binding rather than mutating the old one, which allows changing the type. The immutable-by-default philosophy permeates Rust's entire design: function parameters are immutable unless marked `mut`, struct fields follow the variable's mutability (you cannot have a partly mutable struct), and closures capture variables immutably unless they need `mut`. |
| 1.2 | 35 min | **Java: mutable by default, `final` as opt-out.** Read Horstmann Ch.3 pp. 41–52 (variables and constants, the `final` keyword for local variables and fields). Read Bloch Item 17 pp. 80–86 ("Minimize mutability" — five rules for making a class immutable: do not provide mutators, ensure the class cannot be extended, make all fields `final`, make all fields `private`, ensure exclusive access to mutable components). Read Evans Ch.3 pp. 55–68 (records as immutable data carriers — `record Point(int x, int y)` generates `final` fields, `equals`, `hashCode`, `toString` automatically). Read Goetz Ch.3 pp. 46–54 (§3.4 Immutability — immutable objects are always thread-safe, conditions for immutability, `final` field guarantees under the JMM; §3.5 Safe publication — how immutable objects can be freely shared without synchronization). Read Rahman Ch.5 pp. 217–230 (`ScopedValue` as an immutable alternative to `ThreadLocal` — bounded lifetime, no mutability, no memory leaks). Key insight: Java variables and fields are mutable by default — `int x = 5;` allows `x = 10;` freely. The `final` keyword is Java's opt-in immutability marker: `final int x = 5;` prevents reassignment. But `final` is shallow for reference types — `final List<String> list = new ArrayList<>();` prevents reassigning `list` but does not prevent `list.add("hello")`. Bloch's Item 17 advocates making classes immutable by default and identifies the benefits: thread safety without synchronization, free sharing (no defensive copies needed), safe HashMap keys. Java records (Java 16+) encode this pattern: all record fields are implicitly `final` and the class is implicitly `final`. Goetz deepens this: the JMM guarantees that if an object is properly constructed (no `this` escape), all threads will see the values of its `final` fields without synchronization — this is the formal basis for safe publication of immutable objects. Rahman's `ScopedValue` (Java 21+) represents the modern evolution: an explicitly immutable, bounded-lifetime context variable designed for virtual threads, replacing the mutable, unbounded `ThreadLocal`. |
| 1.3 | 30 min | **Python: mutable by default, immutability through type choice.** Read Ramalho Ch.6 pp. 201–212 ("Variables Are Not Boxes" — Python variables are labels/references, not containers; identity vs equality). Read Martelli Ch.3 pp. 33–60 (data types and their mutability characteristics — mutable: `list`, `dict`, `set`, `bytearray`; immutable: `int`, `float`, `str`, `tuple`, `frozenset`, `bytes`). Read Viafore Ch.4 pp. 45–55 (`typing.Final` — signaling that a name should not be rebound, enforced only by type checkers). Read Slatkin Item 30 pp. 135–137 ("Know That Function Arguments Can Be Mutated" — demonstrating how passing mutable objects to functions enables unintended side effects). Read Slatkin Item 36 pp. 157–160 ("Use None and Docstrings to Specify Dynamic Default Arguments" — the canonical mutable default argument bug: `def f(items=[])` shares the list across all calls). Key insight: Python has no language-level mechanism to prevent variable reassignment. `x = 5` followed by `x = "hello"` is perfectly valid — variables are just names bound to objects. Immutability in Python is a property of the *object*, not the *variable*: `tuple`, `str`, `int`, `frozenset`, and `bytes` are immutable (their state cannot change after creation), while `list`, `dict`, `set`, and most user-defined objects are mutable. There is no `const` or `final` keyword in the language syntax. `typing.Final` (PEP 591, Python 3.8+) provides an advisory annotation — `x: Final = 5` tells type checkers (mypy, pyright) that `x` should not be reassigned, but the runtime does not enforce it. Slatkin's items illuminate the practical consequences: mutable function arguments enable callers and callees to accidentally corrupt each other's state, and the mutable default argument bug (`def f(items=[])`) is one of Python's most common pitfalls — the default list is shared across all calls because default values are evaluated once at function definition time, not per call. The convention `def f(items=None): if items is None: items = []` is the defensive pattern. |
| 1.4 | 15 min | **Cross-language comparison.** Create a comparison table: rows = {variable reassignment default, mechanism to prevent reassignment, field mutability default, immutable class mechanism, compiler/runtime enforcement, parameter mutability}, columns = {Rust, Java, Python}. The fundamental spectrum: Rust treats mutability as a dangerous capability that must be explicitly requested; Java treats it as the default but provides `final` to restrict it; Python treats it as an intrinsic property of the object type rather than the binding. Note the downstream consequences: Rust's immutable-by-default feeds directly into the borrow checker (shared references `&T` are safe precisely because the data cannot be mutated through them); Java's `final` field semantics feed into the JMM (Goetz: safe publication guarantees for immutable objects); Python's object-level immutability feeds into hashability (only immutable objects can be dict keys or set members). Read PEP 591 online to understand the rationale for `typing.Final`. |

### Session 2 — Shared XOR Mutable (Borrow Checker)

**Goal:** understand Rust's unique "shared XOR mutable" rule — the principle that data can be either shared across multiple references or mutated through one exclusive reference, but never both simultaneously — and how Java and Python handle (or fail to handle) the same aliasing-plus-mutation problem.

| Step | Time | Activity |
|------|------|----------|
| 2.1 | 35 min | **Rust: `&T` vs `&mut T` and the borrow checker.** Read Klabnik Ch.4 pp. 67–83 (references and borrowing, mutable references, the rules: at any given time you can have either one `&mut T` or any number of `&T`, but not both; dangling references). Read Blandy Ch.5 pp. 100–121 (full "Sharing Versus Mutation" section — why the rule exists, what bugs it prevents, how NLL makes it ergonomic). Read Gjengset Ch.1 pp. 5–8 (shared references, mutable references, the fundamental invariant that `&T` guarantees the referent will not change and `&mut T` guarantees exclusive access). Read Bos Ch.1 pp. 7–12 ("Borrowing and Data Races" — how the shared-XOR-mutable rule directly prevents data races at compile time; the formal connection between aliasing restrictions and thread safety). Key insight: Rust's borrow checker enforces a compile-time invariant: if any shared reference (`&T`) to data exists, no mutable reference (`&mut T`) to the same data can exist, and vice versa. This is the "shared XOR mutable" rule. The rule eliminates three categories of bugs at compile time: (1) data races — concurrent reads and writes to the same memory are impossible because `&mut T` is exclusive; (2) iterator invalidation — you cannot modify a collection while iterating over it because iteration holds `&T` while modification requires `&mut T`; (3) aliased mutation — you cannot accidentally modify data that another part of the code assumes is stable. Bos frames this from the concurrency angle: the borrow checker's aliasing rules are the same rules that prevent data races — "shared XOR mutable" is not two separate guarantees but one unified principle. Non-lexical lifetimes (NLL, RFC 2094, stable since Rust 2018) refined this: a borrow now ends at its last use point, not at the end of the enclosing scope, making the rule significantly less restrictive in practice. |
| 2.2 | 30 min | **Java: unrestricted aliasing, no mutation control.** Read Horstmann Ch.4 pp. 55–67 (reference types, pass-by-value of references). Read Bloch Item 50 pp. 231–235 ("Make defensive copies when needed" — the pattern Java must use because it has no aliasing control). Read Bloch Item 78 pp. 311–316 ("Synchronize access to shared mutable data" — the JMM requirements when multiple threads access mutable state). Read Valeev Ch.8 pp. 213–230 (modifying unmodifiable collections, mutable objects in HashMap keys — bugs caused by uncontrolled aliased mutation). Read Goetz Ch.3 pp. 33–45 (§3.1 Visibility — stale data from unsynchronized access; §3.2 Publication and escape — how mutable internal state can leak through aliased references). Key insight: Java has no aliasing restrictions whatsoever. Any number of variables can reference the same object, and any of those references can mutate it at any time. This is the root cause of an entire class of Java bugs: (1) a method receives an object reference, stores it, and the caller later mutates the object, corrupting the stored state (solved by defensive copying, Bloch Item 50); (2) a mutable object used as a HashMap key is mutated after insertion, making it unfindable (Valeev Ch.8); (3) multiple threads read and write the same mutable object without synchronization, causing data races (Bloch Item 78). Goetz adds the visibility dimension: even simple reads of unsynchronized mutable fields can return stale values, because the JMM permits caching and reordering. Java's `final` does not help here — `final` prevents reassignment of the *reference* but does not prevent mutation of the *referent*. Java's `Collections.unmodifiableList()` wraps a list to prevent mutation through the wrapper, but the underlying list can still be mutated through the original reference (it is a view, not a copy). Java compensates for unrestricted aliasing with runtime patterns (defensive copies, synchronization, immutable objects) rather than compile-time rules. |
| 2.3 | 30 min | **Python: shared references everywhere, mutation by convention.** Read Ramalho Ch.6 pp. 212–220 (aliasing: `a = [1,2,3]; b = a; b.append(4)` mutates both, identity with `is`, equality with `==`). Read Ramalho Ch.6 pp. 220–228 (shallow copies with `copy.copy()`, deep copies with `copy.deepcopy()`, the `copy` module). Read Martelli Ch.8 pp. 247–265 (the `copy` module in detail — `__copy__` and `__deepcopy__` protocols for custom objects). Read Slatkin Item 22 pp. 92–97 ("Never Modify Containers While Iterating over Them" — Python's equivalent of iterator invalidation: modifying a list/dict/set during iteration can silently produce wrong results or raise `RuntimeError`; the solution is to iterate over a copy or build a separate list of changes). Key insight: Python's reference model is similar to Java's but even more uniform — every variable is a reference, including to integers and strings. `b = a` makes `b` an alias for the same object; `b is a` returns `True`. Python has no mechanism analogous to Rust's borrow checker to prevent aliased mutation. The defense against aliased mutation bugs is the same as in Java but with an additional tool: Python's immutable types (`tuple`, `str`, `frozenset`, `int`, `float`) make aliasing safe because the shared object cannot be changed. When you need a mutable copy, you must explicitly create one: `b = list(a)` (shallow copy), `b = a[:]` (shallow copy via slice), or `b = copy.deepcopy(a)` (deep copy). Slatkin's Item 22 demonstrates Python's iterator invalidation: unlike Rust (which prevents it at compile time) or Java (which throws `ConcurrentModificationException` at runtime), Python's behavior varies by collection — `list` may silently skip or repeat elements, `dict` and `set` raise `RuntimeError` since Python 3, making the bug class inconsistent and harder to catch. The convention `for item in list(original):` iterates over a snapshot copy. |
| 2.4 | 15 min | **Comparative synthesis.** Map the three approaches: Rust prevents aliased mutation at compile time (zero runtime cost, but restricts valid programs); Java allows aliased mutation and compensates with runtime patterns (defensive copies, synchronization — programmer must get it right); Python allows aliased mutation and compensates with immutable types and conventions (no formal enforcement). ### Session 3 — Shallow vs Deep Immutability

**Goal:** understand the critical distinction between shallow immutability (the container cannot be reassigned, but its contents can be mutated) and deep immutability (the container and all of its transitive contents are immutable) — how Java's `final` and `Collections.unmodifiable*` provide only shallow immutability, how Python's `tuple` and `frozenset` provide structural but not deep immutability, and how Rust's ownership model provides deep immutability by default.

| Step | Time | Activity |
|------|------|----------|
| 3.1 | 35 min | **Rust: deep immutability by default through ownership.** Read Klabnik Ch.4 pp. 59–67 (ownership rules, how immutability of a variable propagates to all data it owns). Read Blandy Ch.5 pp. 93–100 (shared references `&T` — immutability is transitive: if you have `&Vec<String>`, you cannot mutate the `Vec` *or* any of its `String` elements). Read McNamara Ch.4 pp. 107–120 (ownership and borrowing — the practical consequence that owning a data structure immutably means the entire tree is immutable). Key insight: in Rust, immutability is deep (transitive) by default. When you have `let v: Vec<String> = ...;`, the `Vec` is immutable, its elements are immutable, and the characters within those strings are immutable — all from a single `let` (no `mut`). Similarly, `&Vec<String>` is a shared reference that transitively prevents mutation of the entire data structure and its contents. There is no way to "reach inside" a `&Vec<String>` and modify an element without interior mutability types. This is fundamentally different from Java and Python, where a "constant reference" only prevents reassigning the outermost variable. The ownership system enforces this: since the `Vec` owns its `String`s, and the binding owns the `Vec`, immutability of the binding transitively locks the entire ownership tree. This deep immutability is what makes shared references (`&T`) safe to use across multiple parts of the code without defensive copies — if you have `&T`, you know the entire `T` and everything it owns is frozen for the duration of the borrow. The only escape from this is interior mutability (`Cell`, `RefCell`, etc.), which is always explicit in the type. |
| 3.2 | 35 min | **Java: `final` is shallow, `Collections.unmodifiable*` is shallow.** Read Bloch Item 17 pp. 80–86 ("Minimize mutability" — the five rules, with emphasis on rule 5: "Ensure exclusive access to any mutable components"). Read Bloch Item 50 pp. 231–235 ("Make defensive copies when needed" — the pattern required because `final` is shallow). Read Valeev Ch.8 pp. 213–248 (modifying "unmodifiable" collections, mutable objects as HashMap keys). Read Evans Ch.5 pp. 119–135 (immutability as a thread-safety strategy). Read Goetz Ch.16 pp. 337–352 (§16.3 Initialization safety — the JMM guarantee that all threads see correctly constructed values of `final` fields without synchronization; this is Java's deepest formal guarantee about immutability, but it only applies to the reference in the `final` field, not to the object that reference points to). Key insight: Java's `final` keyword is strictly shallow. `final List<StringBuilder> list = List.of(new StringBuilder("hello"));` prevents reassigning `list`, and `List.of` prevents adding/removing elements, but `list.get(0).append(" world")` still mutates the `StringBuilder` inside. Java has no language-level mechanism for deep immutability. `Collections.unmodifiableList(originalList)` returns a wrapper that throws `UnsupportedOperationException` on add/remove/set, but: (1) the elements themselves can still be mutated if they are mutable, and (2) the original list is still mutable, so changes to `originalList` are visible through the wrapper (it is a view, not a copy). Java 9+ `List.of()`, `Set.of()`, `Map.of()` create truly immutable collections (not just wrappers), but their elements can still be mutable. Goetz's Ch.16 on initialization safety is the formal underpinning: the JMM guarantees that properly constructed `final` fields are visible to all threads — but this guarantee is *shallow*: if a `final` field points to a mutable `ArrayList`, other threads see the reference correctly but can still observe (or cause) mutations to the list's contents. Bloch's five rules for immutable classes address this gap: rule 5 ("ensure exclusive access to mutable components") means you must defensively copy any mutable component on the way in and on the way out. Records help — `record Point(int x, int y)` is deeply immutable because `int` is a primitive — but `record Group(List<String> members)` is only shallowly immutable. |
| 3.3 | 30 min | **Python: structural immutability with mutable elements.** Read Ramalho Ch.6 pp. 212–220 ("Relative Immutability of Tuples" — a tuple's reference structure is immutable, but if it contains mutable objects, those objects can still be mutated: `t = ([1,2], [3,4]); t[0].append(5)` works). Read Ramalho Ch.2 pp. 21–40 (tuples as immutable sequences, tuples as records vs tuples as immutable lists). Read Ramalho Ch.3 pp. 77–95 (hashable requirement: only immutable objects can be dict keys or set members, but "immutable" means the hash must not change, which requires deep immutability of the hash-contributing state). Read Ramalho Ch.5 pp. 180–200 (`@dataclass(frozen=True)` — creates a frozen dataclass where attribute assignment raises `FrozenInstanceError`, but frozen is shallow: if an attribute is a mutable object, that object can still be mutated). Read Slatkin Item 56 pp. 250–259 ("Prefer dataclasses for Creating Immutable Objects" — practical patterns for using `@dataclass(frozen=True)` with `__slots__`, discussion of when frozen dataclasses suffice and when you need deeper immutability). Key insight: Python's immutable types — `tuple`, `frozenset`, `str`, `bytes`, `int`, `float` — are structurally immutable: their internal structure (which slots hold which references) cannot change. But this is shallow: a tuple containing a list can have its list element mutated. `t = ([1, 2],); t[0].append(3)` is legal and `t` is now `([1, 2, 3],)` — the tuple's identity and structure have not changed (it still holds one reference), but the object at that reference has mutated. This "relative immutability" has a practical consequence for hashability: `hash(t)` will raise `TypeError` if `t` contains any unhashable (mutable) elements, because the hash could change if those elements are mutated. `frozenset` requires all elements to be hashable (hence immutable). `@dataclass(frozen=True)` prevents attribute reassignment (`obj.x = 5` raises `FrozenInstanceError`) but does not prevent mutation of mutable attributes (`obj.items.append(5)` works if `items` is a list). Slatkin's Item 56 provides practical guidance: use `@dataclass(frozen=True)` with `__slots__` for lightweight immutable value objects, and ensure all fields hold immutable types if true deep immutability is needed. True deep immutability in Python requires all objects in the transitive closure to be immutable types. |
| 3.4 | 15 min | **Cross-language synthesis: the depth-of-immutability spectrum.** Compare: Rust provides deep immutability by default through the ownership system (a single `let` freezes the entire ownership tree); Java provides shallow immutability through `final` and requires defensive copying for depth (Bloch's rules), with JMM guarantees for safe publication of `final` fields (Goetz Ch.16); Python provides structural immutability through type choice (`tuple`, `frozenset`) but cannot prevent mutation of mutable elements. Create a concrete example: a "team roster" data structure containing a team name and a list of member names. In Rust, `let roster = Roster { name: String::from("A"), members: vec!["Alice".into()] };` is deeply immutable. In Java, `final var roster = new Roster("A", new ArrayList<>(List.of("Alice")));` is shallowly immutable — `roster.members().add("Bob")` mutates it. In Python, `roster = Roster("A", ["Alice"])` with `@dataclass(frozen=True)` prevents `roster.members = [...]` but not `roster.members.append("Bob")`. Note how Rust's approach eliminates an entire category of bugs that Java and Python must handle with discipline and patterns. |

### Session 4 — Interior Mutability Patterns

**Goal:** understand interior mutability — the controlled escape hatch that allows mutation through an ostensibly immutable reference — as it manifests in Rust (`Cell`, `RefCell`, `Mutex`, `RwLock`, `OnceCell`), Java (`volatile`, `AtomicInteger`, `synchronized`, "effectively immutable" objects with mutable internals), and Python (mutable objects inside "immutable" containers, `__dict__` mutation on frozen dataclasses).

| Step | Time | Activity |
|------|------|----------|
| 4.1 | 40 min | **Rust: Cell, RefCell, Mutex, RwLock as API design tools.** Read Blandy Ch.9 pp. 200–209 (interior mutability: `Cell<T>` for `Copy` types, `RefCell<T>` for runtime borrow checking). Read Klabnik Ch.15 pp. 335–345 (`RefCell<T>` and the interior mutability pattern, combining `Rc<T>` and `RefCell<T>`). Read Klabnik Ch.16 pp. 353–370 (`Mutex<T>` as the concurrent analogue of `RefCell<T>`, `Arc<T>` as the concurrent analogue of `Rc<T>`). Read Bos Ch.1 pp. 12–29 (the complete interior mutability taxonomy: `Cell<T>` — no borrowing, get/set only, zero overhead; `RefCell<T>` — runtime borrow checking, panics on violation; `Mutex<T>` and `RwLock<T>` — OS-level locking for multi-threaded access; `UnsafeCell<T>` — the primitive underpinning all of them; Atomics — lock-free interior mutability; `Send` and `Sync` traits determining which types can cross thread boundaries). Read Gjengset Ch.1 pp. 6–8 (interior mutability overview — when and why to use it). Read Bos Ch.2 pp. 41–47 (lazy initialization with atomics — `compare_exchange` for one-time init, the pattern that `OnceCell`/`OnceLock` formalize). Key insight: interior mutability is Rust's mechanism for mutation through a shared reference (`&T`). It is not a workaround or a hack — it is a deliberate API design pattern for cases where the shared-XOR-mutable rule is too restrictive. The key types form a hierarchy of guarantees: `Cell<T>` (single-threaded, `Copy` types only, zero overhead — get/set without borrowing); `RefCell<T>` (single-threaded, any type, runtime borrow checking — panics on violation); `Mutex<T>` (multi-threaded, any type, blocks on contention — OS-level locking); `RwLock<T>` (multi-threaded, multiple readers or one writer — more granular than Mutex). Bos provides the deepest treatment: all of these are built on `UnsafeCell<T>`, the only type in Rust that permits aliased mutation. The `Send` and `Sync` marker traits encode the thread-safety boundary: `Cell<T>` is `!Sync` (cannot be shared between threads), while `Mutex<T>` is `Sync` (can be). `OnceCell`/`OnceLock` (stabilized in Rust 1.70/1.80) add lazy-initialization interior mutability — write once, then read immutably. When designing an API, you choose the interior mutability type based on: single-threaded vs multi-threaded, `Copy` vs non-`Copy` data, and whether contention is expected. |
| 4.2 | 35 min | **Java: volatile, atomics, synchronized — mutation control through concurrency primitives.** Read Bloch Item 78 pp. 311–316 ("Synchronize access to shared mutable data" — both atomicity and visibility require synchronization). Read Goetz Ch.3 pp. 33–40 (§3.1 Visibility — stale data, `volatile` guarantees; §3.2 Publication and escape — how mutable state can leak). Read Goetz Ch.15 pp. 319–336 (Atomic Variables and Nonblocking Synchronization — CAS as the hardware foundation, `AtomicInteger`/`AtomicReference` operations, nonblocking algorithms like Treiber stack). Read Evans Ch.5 pp. 140–166 (JMM, happens-before, `volatile` semantics, `synchronized` blocks). Read Evans Ch.6 pp. 169–195 (`ConcurrentHashMap` — fine-grained locking for shared mutable state). Key insight: Java does not have a concept called "interior mutability" because Java has no shared-XOR-mutable rule to break — all object references allow mutation. However, Java has an analogous design problem: how to provide thread-safe mutation of shared state. The Java tools for this are: (1) `volatile` — guarantees visibility across threads (reads/writes go directly to main memory, no caching in CPU registers) and prevents reordering, but does not provide atomicity for compound operations; (2) `AtomicInteger`/`AtomicReference`/etc. — provide lock-free atomic read-modify-write operations (compare-and-swap); (3) `synchronized` — provides both mutual exclusion and visibility (a `synchronized` block establishes a happens-before relationship). The API design parallel to Rust is real: `AtomicInteger` in Java serves the same role as `AtomicI32` in Rust (lock-free interior mutability); `synchronized` in Java serves the same role as `Mutex<T>` in Rust (lock-based interior mutability). Goetz's Ch.15 is the definitive treatment of Java atomics: CAS is the hardware primitive, and `AtomicInteger.compareAndSet()` is the Java exposure. The critical difference is that Rust's type system *forces* you to use these wrappers (you cannot get `&mut T` from `&T` without them), while Java relies on the programmer to use them correctly — there is nothing in the Java type system preventing unsynchronized access to a non-volatile field. |
| 4.3 | 30 min | **Python: mutable objects inside immutable containers, and the illusion of immutability.** Read Ramalho Ch.6 pp. 212–220 (relative immutability of tuples — mutable elements inside immutable containers). Key insight: Python has no formal interior mutability concept because it has no formal immutability enforcement to begin with. However, the same pattern arises in practice: a "frozen" dataclass (`@dataclass(frozen=True)`) prevents attribute reassignment, but its mutable attributes (lists, dicts, sets) can still be mutated freely. `t = ([1, 2],); t[0].append(3)` is the canonical example — the tuple is "immutable" but its list element is mutated. This is Python's equivalent of interior mutability: the outer container's immutability contract is violated by the inner object's mutability. The practical consequence is that `hash()` on a tuple containing a list raises `TypeError` — Python detects that shallow immutability is insufficient for safe hashing. For concurrent Python code, Slatkin's Item 69 addresses the common misconception that the GIL makes locking unnecessary: while the GIL prevents true parallel execution of Python bytecode, compound operations (check-then-act, read-modify-write) are still susceptible to race conditions because the GIL can release between bytecodes. `threading.Lock` provides mutual exclusion, analogous to Java's `synchronized` and Rust's `Mutex`. Python has no equivalent of lock-free atomics (the GIL makes them unnecessary for CPython, but this is an implementation detail, not a language guarantee). |
| 4.4 | 20 min | **Cross-language comparison: the interior mutability spectrum.** Map the patterns: Rust's `Cell<T>` has no Java/Python equivalent (zero-cost, single-threaded, `Copy`-only). Rust's `RefCell<T>` is analogous to Java's non-synchronized mutable field (runtime checking, single-threaded) but Rust panics on violation while Java silently produces data races. Rust's `Mutex<T>` maps to Java's `synchronized` and Python's `threading.Lock`. Rust's `RwLock<T>` maps to Java's `ReadWriteLock`. Rust's `AtomicI32` maps to Java's `AtomicInteger` (both built on hardware CAS). Python has no equivalent of lock-free atomics. Create a table: rows = {single-threaded cell, runtime borrow check, mutual exclusion lock, reader-writer lock, atomic operations, lazy initialization}, columns = {Rust type, Java type, Python type}. The key takeaway: Rust makes the choice of mutability control *visible in the type signature* (`Mutex<T>` vs `RwLock<T>` vs `RefCell<T>`), while Java and Python leave it to runtime conventions and documentation. Optionally read Bos Ch.4 pp. 75–83 (building a spin lock from scratch) and Ch.9 pp. 181–211 (building Mutex/RwLock from scratch) to understand the implementation layer beneath these abstractions. ### Session 5 — Copy-on-Write

**Goal:** understand the copy-on-write (CoW) optimization strategy — deferring the cost of copying data until mutation is actually needed — as implemented by Rust's `Cow<T>` enum, Java's `CopyOnWriteArrayList`, and Python's implicit CoW via `os.fork()` and the `copy` module patterns.

| Step | Time | Activity |
|------|------|----------|
| 5.1 | 35 min | **Rust: `Cow<T>` — a zero-cost borrowed-or-owned enum.** Read Blandy Ch.13 pp. 293–301 (`Cow` — Clone-on-Write, the `Borrow` and `ToOwned` traits that `Cow` is built on). Read Matthews Ch.5 pp. 112–118 (`Cow` in practice — avoiding unnecessary allocations by deferring cloning). Read the Rust std docs for `Cow` online (`https://doc.rust-lang.org/std/borrow/enum.Cow.html`). Key insight: `Cow<'a, B>` (Clone-on-Write) is an enum with two variants: `Borrowed(&'a B)` and `Owned(<B as ToOwned>::Owned)`. It transparently holds either a reference to existing data or an owned copy. When you call `.to_mut()`, if the data is borrowed, it clones (allocates and copies) and switches to the `Owned` variant; if already owned, it returns a mutable reference directly. This is a type-level optimization pattern: functions can accept `Cow<str>` to work with both `&str` (no allocation) and `String` (owned) uniformly, only allocating when mutation is needed. The classic use case is a string-processing function that usually returns the input unchanged but sometimes needs to modify it: `fn process(input: &str) -> Cow<str>` returns `Borrowed(input)` for the common no-change case and `Owned(modified_string)` for the rare mutation case, avoiding allocation in the common path. `Cow` implements `Deref`, so you can use it transparently as `&B` for reading. The `Borrow` trait and `ToOwned` trait provide the abstraction: `Borrow` unifies owned and borrowed forms (`String` borrows as `&str`), while `ToOwned` provides the cloning operation (`&str` becomes `String`). |
| 5.2 | 35 min | **Java: `CopyOnWriteArrayList` and defensive copying patterns.** Read Goetz Ch.5 pp. 79–86 (§5.1 Synchronized collections — problems with `Collections.synchronizedList`, client-side locking, iterator issues; §5.2 Concurrent collections — `CopyOnWriteArrayList` as a thread-safe alternative for read-heavy workloads, its design rationale and performance trade-offs). Read Evans Ch.6 pp. 169–195 (`CopyOnWriteArrayList` — implementation details, when to use it). Read Bloch Item 50 pp. 231–235 ("Make defensive copies when needed" — manual CoW pattern for API safety). Read Bloch Item 13 pp. 58–65 ("Override clone judiciously" — the mechanics of shallow and deep copying in Java). Key insight: Java's `CopyOnWriteArrayList` is a concurrent collection optimized for read-heavy, write-rare workloads. Goetz explains the design rationale: its internal implementation holds a volatile reference to an immutable array snapshot. Every read operates on the current snapshot (no locking needed, fast). Every write (add, set, remove) creates a complete copy of the internal array, modifies the copy, and atomically swaps the reference (using a `ReentrantLock` for mutual exclusion among writers). Iterators always operate on the snapshot at iterator-creation time and never throw `ConcurrentModificationException`. The trade-off is explicit: writes are O(n) (full array copy) while reads are O(1) with no contention. Goetz's treatment is the most authoritative — he explains that `CopyOnWriteArrayList` is suitable when iteration frequency vastly exceeds modification frequency (e.g., event-listener lists). This is a specific concurrent data structure, not a general-purpose CoW abstraction like Rust's `Cow<T>`. Java has no type-level CoW equivalent to `Cow<T>`. Defensive copying (Bloch Item 50) is Java's manual CoW pattern: `this.items = new ArrayList<>(items);` in a constructor copies on the way in, and `return new ArrayList<>(this.items);` copies on the way out, to prevent aliased mutation. |
| 5.3 | 30 min | **Python: copy module, os.fork(), and implicit CoW.** Read Martelli Ch.8 pp. 247–265 (the `copy` module — `copy.copy()` for shallow copies, `copy.deepcopy()` for deep copies, `__copy__` and `__deepcopy__` protocols for custom objects). Read Ramalho Ch.6 pp. 220–228 (copies of objects — shallow vs deep, practical implications). Key insight: Python has no explicit CoW type analogous to Rust's `Cow<T>`. However, the CoW concept manifests in several ways. (1) At the OS level: `os.fork()` creates a child process that shares the parent's memory pages marked as copy-on-write by the operating system — pages are only physically copied when either process writes to them. CPython's reference counting partially defeats this optimization because incrementing reference counts writes to every visited object's header, triggering page copies even for read-only traversals. (2) At the library level: the `copy` module provides explicit shallow and deep copying, but there is no "lazy copy" that defers allocation. The pattern `def process(items): if needs_modification: items = list(items); ...modify items...; return items` is Python's manual CoW — copy only when modification is needed. (3) At the string level: CPython interns small strings and integers, which is conceptually related to CoW (multiple names share the same object; a new object is created only if a "different" value is needed, though strings are immutable so the "write" never happens). (4) NumPy's `np.array` views provide a form of CoW: slicing an array returns a view (shared memory) that becomes a copy only if you explicitly call `.copy()` or if advanced indexing triggers it. |
| 5.4 | 15 min | **Comparative synthesis: when and why CoW.** Create a comparison: Rust's `Cow<T>` is a generic, type-level, zero-cost CoW abstraction usable in any API; Java's `CopyOnWriteArrayList` is a specific concurrent collection with CoW semantics (Goetz: suitable when "iteration frequency vastly exceeds modification frequency"), and defensive copying is the manual general-purpose pattern; Python has no language-level CoW, relying on OS-level CoW (fork), library conventions (copy-before-mutate), and NumPy view semantics. The key trade-off: CoW defers the cost of copying to the mutation point, which is optimal when reads vastly outnumber writes. Rust's `Cow<T>` makes this explicit in the type system (the caller sees `Cow<str>` and understands the performance contract). Java's `CopyOnWriteArrayList` documents this in the class name and Javadoc. Python's manual patterns require documentation and convention. Note the connection to immutability: CoW is most useful in contexts where data is *usually* read-only but *occasionally* needs modification — exactly the scenario where Rust's immutable-by-default philosophy shines (the common case is free, and the rare mutation pays the cost). Read the Rust By Example Cow page online for additional examples. |
